# AI 编码工具实战培训方案

- **客户**：杭州世平信息科技有限公司
- **时间**：2025年3月中旬 · 2天（每天6小时）
- **地点**：西安研发中心
- **学员**：开发工程师（前端/后端/全栈）、测试、产品经理，约20人

---

## 培训背景

根据贵公司提供的培训需求信息，团队大约20人，以中级开发工程师为主，涵盖前端、后端、全栈方向，同时有测试人员和部分产品经理。团队已有约半年自主学习 AI 编码工具的经验，具备一定实践基础，希望通过这次培训进一步提效。

结合我个人在分布式系统架构和 AI 编码工具方面的实战经验，以及在企业级项目中推动 AI 研发落地的实践，我对这次培训有几个基本判断：

1. **是这次不该讲入门。** 团队半年的自学基础摆在那里，再讲安装配置、基础问答这些内容是浪费时间。培训的价值应该在于：把大家散点式的个人经验，拉升到系统化的方法论层面。为什么有人用 AI 提效明显，有人用了反而更慢——差距往往不在工具本身，而在使用方法和工程化配套上。

2. **是不能只讲编码。** 团队有测试和产品经理，而且从行业实践来看，AI 对研发流程的改变不只发生在写代码这个环节。需求分析、技术方案、测试用例设计、Code Review、自动化测试——这些环节的 AI 融合同样重要，甚至对团队整体提效的影响更大。编码效率的提升如果卡在上下游的协作节点上，整体交付速度还是快不起来。

3. **是要有实战产出。** 培训最怕"听的时候觉得有道理，回去还是老样子"。两天的时间，需要让学员真正动手完成至少一个 AI 辅助开发的完整任务，形成肌肉记忆，而不只是知道概念。

基于这三点，整份方案的设计方向是：**帮助团队从个人零散使用 AI 工具，过渡到系统化地用 AI 改造研发流程。**

## 培训目标

两天下来，希望学员带走三样东西：

**认知对齐。** 团队对 AI 到底能做到什么程度有一致的理解。不是停留在"帮我补全代码"这个层面，而是清楚从辅助编码到 AI 协同开发再到 AI 自主完成任务，这条路走到哪了，哪些今天就能落地，哪些还需要等待。

**方法和手感。** 掌握一套经过实际项目验证的 AI 编码方法论，并且通过实战真正动手跑通。不是听完觉得"有道理"，是回去第二天就能用在手头的工作里。

**团队可复用的资产。** 项目规范模板、工具配置标准、Prompt 最佳实践——这些不是个人经验，是团队层面可以直接沉淀下来、持续发挥作用的工程资产。

## 我准备了什么

**方法论层面**，我整理了一套 AI 产研全链路的提效体系，覆盖从需求到上线的各个阶段。同时基于 Claude Code 和 Cursor 的深度使用经验，总结了一套进阶实践指南——不是官方文档的复述，是实际项目里踩出来的东西。

**案例层面**，我准备了一个完整的实战项目——基于 ETCD 的分布式 Proxy 系统，纯 AI Coding 实现。这个项目涵盖集群管理、Controller 选举、时间轮、GRPC 服务、节点故障切换，是一个架构完整的分布式系统。除此之外还有代码重构、Code Review、单测生成、自动化测试等多个小案例。

**实战层面**，培训的实战项目支持两种模式：可以用我准备的案例，也可以由贵团队提供真实的业务需求，现场用 AI 完成。我本身有分布式系统的架构背景，能 cover 各类技术方向的实战指导。

## 方案选择

我准备了三套课程方案，侧重不同：

| | 方案 A：认知赋能型 | 方案 B：理论+实战型 | 方案 C：实战营 |
|---|---|---|---|
| 定位 | 系统认知 + 经验分享 | 认知升级 + 动手实战 | 全程动手，以战代训 |
| 理论与实战比例 | 70 : 30 | 40 : 60 | 10 : 90 |
| 实战项目 | 讲师案例演示为主 | 客户出题或讲师案例均可 | 客户出题，现场AI实现 |
| 适合场景 | 侧重方法论和战略视角 | 多角色混合团队 | 纯开发团队冲刺 |

**建议选方案 B。** 贵团队角色多元，已有实践基础，B 方案在认知拉齐和动手实战之间平衡得最好。方案 A 偏理论，对开发者来说不够过瘾；方案 C 偏硬核编码，测试和产品经理参与感会弱。当然，三套方案也可以根据贵团队的实际偏好做微调。

以下是三套方案的详细大纲。

### 各模块详细说明

#### Day 1 上午

**AI 提效的核心矛盾（9:00 - 9:50）**

从一个核心不等式开始：用 AI 工具 ≠ 个人提效 ≠ 组织提效。大部分团队卡在第二个不等号上——个人编码快了，但需求吞吐量和交付周期没变。问题出在哪，怎么解。这个环节把培训的视角从"学工具"拉高到"改变工作方式"，为两天定调。

**AI 研发成熟度演进（9:50 - 10:40）**

L1 辅助（Copilot 补全）、L2 协同（人和 AI 深度分工）、L3 自主（AI 端到端完成小颗粒度任务）。每个级别对应什么样的工作方式、什么样的工具配套、什么样的组织形态。不画饼，帮大家判断自己团队当前在哪、下一步该往哪走。

**案例讲解：分布式 Proxy 纯 AI Coding 实现（10:50 - 12:00）**

用一个真实项目展示 AI 编码的能力边界。这个项目包含 ETCD 集群管理、Controller 选举、时间轮调度、GRPC 服务通信、Redis 消息持久化，是一个完整的分布式系统。重点不是讲技术细节，是展示从方案设计到代码落地，纯 AI Coding 的工作流程是什么样的——哪些地方 AI 能直接搞定，哪些地方人必须介入，整个过程的决策链条是什么。

#### Day 1 下午

**产研全链路 AI 介入（14:00 - 15:00）**

逐环节拆解：需求阶段 AI 能做什么、设计阶段能做什么、开发阶段能做什么、测试阶段能做什么、发布运维能做什么。每个环节给出当前可落地的实践和工具选型建议，不讲远期概念。产品经理和测试同学在这个环节能看到 AI 对自己岗位的实际价值，不只是开发的事。

**测试侧深入（15:00 - 15:50）**

三块内容：AI 辅助测试用例设计的实际效果（结合知识库 vs 不结合知识库的差异）、Cursor 辅助自动化用例开发的效率数据（哪些环节提升 60-75%，哪些只有 10-15%）、自动化失败分析的思路和落地方式。这块对测试同学直接可用，对开发同学理解质量保障体系也有帮助。

**团队落地路径（16:00 - 16:40）**

不讲技术，讲管理。三个阶段怎么走（试点期 → 推广期 → 常态化）、度量指标怎么设（不是看 AI 生成了多少行代码）、常见的坑和反模式（比如"全员装了 Copilot = AI 落地"这类误区）。

**分组讨论（16:40 - 17:00）**

每组根据自己的业务场景，拟一个三个月的 AI 提效行动计划。不是形式主义，是让大家把第一天的内容转化成可执行的动作。

#### Day 2 上午

**规范驱动开发 SDD（9:00 - 9:40）**

核心思路：先写 spec 再让 AI 编码，把需求偏差的修复成本从改代码左移到改 Markdown。讲三层产物的组织方式——需求 spec、技术 spec、实现 spec，AI 在每层的角色不同。配合真实项目的 spec 文档做讲解。

**验证驱动开发（9:40 - 10:20）**

给 AI 反馈循环，让它能自我纠正。四种验证策略：测试验证、构建验证、lint 验证、视觉验证。重点讲 TDD 在 AI 编码场景下的变化——先写测试不是为了"规范"，是为了防幻觉、给 AI 一个可自动验证的目标。

**上下文工程（10:30 - 11:10）**

AI 编码效果好不好，80% 取决于你给它的上下文。CLAUDE.md 怎么写（项目背景、技术栈、编码规范、常见陷阱）、怎么分层加载（全局 → 项目 → 模块）、Cursor Rules 的组织策略。什么该放、什么不该放、放多了会怎样。

**案例演示（11:10 - 12:00）**

三个实操演示穿插讲解。用 Cursor 做 Code Review 的正确方式 vs 错误方式（精确提问 vs 泛泛要求的巨大差异）、用 Claude Code 的 Plan Mode 做代码分析和方案设计、用采访模式做需求澄清（让 AI 反过来问你问题，逼出隐含需求）。

#### Day 2 下午

**Claude Code 进阶（14:00 - 14:50）**

不是功能清单，是结合场景讲什么时候用什么。Hooks 做事件驱动自动化（比如每次 commit 前自动跑检查）、Skills 做可复用工作流（团队沉淀的最佳实践变成 AI 可执行的指令）、子代理机制做任务分解（大任务拆成独立上下文的小任务并行处理）。三者怎么配合起来，形成一个完整的 AI 辅助开发体系。

**智能体设计模式精选（15:00 - 16:00）**

挑和日常开发最相关的几个模式讲透：提示链（把复杂任务拆成步骤串联）、反思模式（让 AI 自我评估和迭代）、多智能体协作（不同角色的 AI 分工协作）。重点是让大家理解这些不是论文里的概念——Claude Code 的子代理就是多智能体模式，Hooks 就是事件驱动模式，手头的工具已经在用这些模式，只是你可能没意识到。

**Prompt 结构化写法（16:00 - 16:40）**

四段式模板：角色设定 + 上下文 + 任务描述 + 输出约束。配真实的 before/after 对比——同一个需求，不同写法的 Prompt 产出质量差距有多大。给出可直接复用的模板和写法习惯。

**两天总结与答疑（16:40 - 17:00）**

回顾两天核心要点，回答学员问题，收集反馈。

**上下文工程（10:30 - 11:10）**

#### Day 1 下午

**案例讲解：分布式 Proxy（14:00 - 14:40）**

压缩版的案例分析。不展开技术细节，聚焦在工作流：需求怎么拆、spec 怎么写、代码怎么生成、出问题怎么纠偏。40 分钟讲完整个决策链条，给下午的实操做铺垫。

**实操一：端到端 AI 开发（14:40 - 15:20）**

学员动手。每人选一个模块，用 AI 从 spec 到代码跑通。讲师巡场指导，重点关注 Prompt 写法和上下文管理。这是大部分人第一次系统地按照方法论来用 AI，而不是凭感觉。

**实操二：AI 辅助 Code Review + 重构（15:30 - 16:30）**

换一个维度动手。用 AI 对已有代码做 Review 和重构，练习精确提问 vs 泛泛提问的差别。用真实案例体会"怎么问"比"用什么工具"重要得多。

**复盘（16:30 - 17:00）**

收集实操中的典型问题，现场讲解 Prompt 纠偏技巧。不是讲理论，是针对刚才遇到的真实问题给解法。

#### Day 2 上午

**实战项目说明 + 分组（9:00 - 9:30）**

**分组实战（9:30 - 12:00）**

测试组：用 AI 辅助设计测试用例（结合业务知识库），然后用 Cursor 生成自动化测试脚本。对比纯手写和 AI 辅助的效率差异。

讲师全程巡场，遇到共性问题随时叫停做集中讲解。

#### Day 2 下午

**实战收尾 + 联调（14:00 - 15:00）**

各组完成手头任务，开发组和测试组做联调。遇到问题用 AI 辅助排查和修复。

**成果展示 + 点评（15:00 - 15:40）**

每组展示成果，讲师重点点评 AI 使用方式——不是评代码质量，是评 AI 协作方式。哪些 Prompt 写得好、哪些地方上下文管理有优化空间、哪些环节人工介入的时机选得对。

**进阶串讲（15:50 - 16:30）**

两天实操之后，带着实际体感回头看进阶工具。Claude Code 的 Hooks、Skills、子代理——这些机制能怎么帮你把刚才做的事情做得更高效。智能体设计模式和手头工具的对应关系：子代理 = 多智能体，Hooks = 事件驱动，Skills = 可复用工作流。有了实操经验再听这些，理解深度完全不同。

**总结 + 行动计划（16:30 - 17:00）**

回顾两天核心收获，每组讨论并产出一份团队 AI 提效行动计划。

### 各模块详细说明

#### Day 1 上午

**项目发布 + 分组（9:00 - 9:30）**

公布实战项目。优先使用客户团队提供的真实业务需求——这样培训产出直接就是可用的代码，不是练习作业。如果客户暂无合适需求，用讲师准备的分布式 Proxy 项目。按 3-4 人分组，每组覆盖开发和测试角色。检查环境：Claude Code / Cursor 账号、开发环境、权限。

**快速串讲（9:30 - 10:00）**

整个两天唯一的集中理论环节，30 分钟讲完三件事：SDD 流程（先 spec 后代码）、上下文工程（CLAUDE.md / Rules 怎么写）、验证驱动（测试先行防幻觉）。给出模板，后面直接用。

**实战阶段一：需求拆解 + spec 编写（10:00 - 10:40）**

各组拿到需求后，用 AI 做需求澄清（采访模式）和任务拆解，产出 spec 文档。讲师巡场检查 spec 质量——这个阶段如果 spec 写得不好，后面生成的代码会跑偏，正好让大家体会"左移"的价值。

**实战阶段二：从 spec 到代码（10:50 - 12:00）**

按 spec 让 AI 生成核心模块代码。每组至少跑通一个模块的完整流程：spec → 代码生成 → 构建通过 → 基础测试通过。讲师重点关注 Prompt 质量和上下文管理方式。

#### Day 1 下午

**实战阶段三：功能推进 + 联调排错（14:00 - 15:20）**

继续推进开发。这个阶段会遇到真实的问题——AI 生成的代码有 bug、模块间接口不一致、上下文丢失导致 AI "忘记"之前的约定。讲师不直接给答案，引导学员用 AI 自身的能力排查和修复（验证驱动的实际应用）。

**实战阶段四：测试（15:30 - 16:30）**

开发到一定程度后，切换到测试环节。用 AI 设计测试用例，再用 Cursor 生成自动化测试脚本。开发同学也参与这个环节——写测试是 AI 编码中防止幻觉的核心手段，不是只有测试同学需要会。

**Day 1 复盘（16:30 - 17:00）**

各组同步进度，讲师收集共性问题做集中讲解。典型问题包括：上下文太长导致 AI 质量下降怎么办、多文件修改如何保持一致性、Prompt 越改越乱怎么重新组织。每个问题给出可操作的解法。

#### Day 2 上午

**Day 2 启动（9:00 - 9:20）**

快速回顾昨天各组进度和遗留问题，明确今天的目标：完成项目、做 Review 和优化、最终交付。

**实战阶段五：Code Review + 重构（9:20 - 10:20）**

组间交叉 Review。用 AI 辅助 Review 对方组的代码，找出问题并提出重构建议。然后用 AI 执行重构。这个环节练两件事：怎么用 AI 做有效的 Review（精确提问 vs 泛泛提问）、怎么用 AI 做可控的重构（小步走、每步验证）。

**实战阶段六：功能收尾 + 联调（10:30 - 12:00）**

根据 Review 结果修复问题，完成功能开发，跑通所有测试用例。讲师协助解决卡点问题。

#### Day 2 下午

**实战阶段七：最终联调 + 成果打包（14:00 - 15:00）**

最终联调，确保交付物完整可运行。各组整理代码、spec 文档、测试用例，形成可交接的项目包。

**成果展示 + 深度点评（15:00 - 15:40）**

每组展示两天的成果：做了什么、AI 在哪些环节发挥了作用、遇到了什么问题、怎么解决的。讲师做深度点评，不只是点评代码，重点点评每组的 AI 协作模式——Prompt 策略、上下文管理、人机分工方式、验证策略的选择。

**进阶工具串讲（15:50 - 16:30）**

两天高强度实操之后，学员对 AI 编码有了真实体感。这时候讲进阶工具效果最好：Claude Code 的 Hooks 能帮你自动化哪些刚才手动做的事、Skills 能帮你把团队最佳实践固化下来、子代理能帮你并行处理刚才串行做的任务。智能体设计模式和两天实战的对应关系——你已经在用这些模式了，只是可以用得更好。

**总结 + 落地讨论（16:30 - 17:00）**

两天总结。讨论如何把培训中的实践方式带回日常工作：团队 CLAUDE.md 怎么维护、Rules 库怎么共建、AI 辅助 Review 怎么纳入流程、实战中踩过的坑怎么沉淀成团队知识。
