# $mq9.AI.* Protocol Design Document

## Overview

`$mq9.AI.*` is a subject namespace convention designed by RobustMQ for Agent asynchronous communication. It is built on the NATS protocol and runs on top of RobustMQ's unified storage layer.

The core problem it solves is: **asynchronous communication between Agents, where the sender and receiver do not need to be online at the same time.**

This is not a new protocol, but a set of semantic conventions defined on top of NATS subjects. An Agent can understand the intent of a message simply by seeing the subject, without consulting documentation or performing additional configuration.

---

## Scenario Overview

Agent communication has three essential characteristics: Agents are asynchronous with unpredictable online status; Agents are unaware of each other with mesh-like relationships; Agents are temporary with a lifecycle that may be only a few seconds.

Around these three characteristics, `$mq9.AI.*` covers the following eight core scenarios:

**Scenario 1: Sub-Agent completes a task, asynchronously notifies the primary Agent.**
The primary Agent dispatches tasks and does not block waiting. When a sub-Agent completes, it sends a message to the primary Agent's mailbox; the primary Agent picks it up when available.

**Scenario 2: Primary Agent monitors the status of all sub-Agents.**
Each sub-Agent periodically reports its own status. The primary Agent uses a wildcard subscription to automatically detect all Agents coming online, running, and ceasing to exist.

**Scenario 3: Task broadcast — whoever has the capability comes to claim it.**
The primary Agent broadcasts tasks; multiple Workers compete to consume them; the Worker that claims a task sends a confirmation and result to the primary Agent's mailbox.

**Scenario 4: Agent detects an anomaly and broadcasts an alert.**
An Agent detects an anomaly and broadcasts an event; Agents subscribed to that domain respond automatically; the detector does not need to know who is handling it.

**Scenario 5: Edge Agent offline message backlog.**
A cloud Agent sends instructions to an edge Agent; while the edge Agent is offline, messages are persisted and waiting; after reconnecting, messages are retrieved by priority.

**Scenario 6: Human-machine hybrid workflow.**
An Agent reaches a node requiring human judgment and sends an approval request to the human approver's mailbox; the approver handles it when available; the result is sent back to the Agent's mailbox to continue execution.

**Scenario 7: Agent requests another Agent and waits for a response.**
Agent A sends a request to Agent B's mailbox; if B is offline the request waits; B processes it when online and sends the response to A's mailbox; A matches via correlation_id.

**Scenario 8: Agent registers capabilities; other Agents discover it.**
An Agent writes its capability declaration to its status on startup; other Agents discover Agents with a given capability by subscribing to status or broadcasting a capability query.

## Scenario Details

---

### Scenario 1: Sub-Agent Completes a Task, Asynchronously Notifies the Primary Agent

**Problem**

The primary Agent breaks down a task and dispatches it to sub-Agents for parallel execution. When a sub-Agent completes, it needs to notify the primary Agent. The primary Agent cannot block waiting and needs to handle other things asynchronously, picking up results when the sub-Agent is done.

**Flow**

```
Primary Agent creates a mailbox and obtains an mail_id
Primary Agent dispatches tasks to sub-Agents, passing reply_to
Sub-Agent processes the task
Sub-Agent completes and sends a message to the primary Agent's mailbox
Primary Agent subscribes to the mailbox and picks up the result to continue execution
```

**Core requirements**: Point-to-point, asynchronous, temporary persistence; the primary Agent does not need to be online waiting.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| REQ | `$mq9.AI.MAILBOX.CREATE` | Primary Agent requests a mailbox and obtains an mail_id |
| SUB | `$mq9.AI.INBOX.{primary_mail_id}.*` | Primary Agent subscribes to its own mailbox |
| PUB | `$mq9.AI.INBOX.{primary_mail_id}.normal` | Sub-Agent completes and sends result |

Three operations, two subjects, covering the complete scenario. The primary Agent does not need to be online waiting; the sub-Agent does not need to know which machine the primary Agent is on.

---

### Scenario 2: Primary Agent Monitors the Status of All Sub-Agents

**Problem**

An Orchestrator manages multiple Worker Agents and needs real-time awareness of each Worker's status. Polling cannot be used; a static list cannot be maintained; new Workers joining must be detected automatically; Worker deaths must also be detected automatically.

**Flow**

```
Worker Agent starts and periodically reports status
Primary Agent uses a wildcard subscription to all Agent status
New Worker joins; primary Agent automatically receives its status
Worker exceeds TTL without reporting; status expires; primary Agent detects death
```

**Core requirements**: One-to-many, active reporting, wildcard subscription, decentralized detection, retain=latest.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.STATUS.{sub_mail_id}` | Sub-Agent reports status periodically, retain=latest, ttl=60 |
| SUB | `$mq9.AI.STATUS.*` | Primary Agent subscribes to all sub-Agent status |
| REQ | `$mq9.AI.STATUS.{sub_mail_id}` | Primary Agent queries a specific sub-Agent's latest status on demand |

Two subjects; the primary Agent uses one subscription to have global awareness. New Agents are detected automatically on join; TTL expiry detects death automatically — no registration or deregistration required.

---

### Scenario 3: Task Broadcast — Whoever Has the Capability Comes to Claim It

**Problem**

The primary Agent has a batch of tasks to dispatch but doesn't know which Workers are available and doesn't want to maintain a Worker list. It broadcasts tasks; capable and idle Workers claim them; each task is guaranteed to be processed by only one Worker.

**Flow**

```
Primary Agent broadcasts tasks to $mq9.AI.BROADCAST.task.available
Workers subscribe to this subject; queue group ensures competitive consumption
The Worker that claims the task sends confirmation to the primary Agent's mailbox
Worker completes; result is sent to the primary Agent's mailbox
```

**Core requirements**: Broadcast distribution, competitive consumption, queue group mutual exclusion, result callback to mailbox.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.BROADCAST.task.available` | Primary Agent broadcasts task |
| SUB | `$mq9.AI.BROADCAST.task.available` queue_group | Workers compete to consume; only one claims the task |
| PUB | `$mq9.AI.INBOX.{primary_mail_id}.normal` | Worker sends confirmation and result to primary Agent's mailbox |

Three operations, two subjects. `queue_group` is the key — it ensures a task is processed by only one Worker; NATS supports this natively without requiring a distributed lock.

---

### Scenario 4: Agent Detects an Anomaly and Broadcasts an Alert

**Problem**

An Agent detects a system anomaly and needs to notify all Agents that care about this event. It doesn't know who needs to be notified and doesn't need to wait for a response. Send it out; capable Agents respond themselves.

**Flow**

```
Agent detects anomaly
Broadcasts to $mq9.AI.BROADCAST.{domain}.{event}
Agents subscribed to that domain receive it automatically
Each responds according to its own logic
The detector does not need to know who is handling it
```

**Core requirements**: Event broadcast, subscriber self-determination, publisher awareness-free, wildcard multi-level subscription.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.BROADCAST.{domain}.{event}` | Agent broadcasts anomaly event |
| SUB | `$mq9.AI.BROADCAST.system.*` | Subscribe to all events in a domain |
| SUB | `$mq9.AI.BROADCAST.*.anomaly` | Subscribe to anomaly events across all domains |
| SUB | `$mq9.AI.BROADCAST.#` | Subscribe to all broadcasts |

One PUB, multiple SUB patterns. The publisher just sends; subscribers decide what granularity they care about. Wildcards at three levels cover single-domain, cross-domain, and global subscription needs.

---

### Scenario 5: Edge Agent Offline Message Backlog

**Problem**

Edge devices have unstable networks — intermittently connected. Cloud Agents need to send instructions to edge Agents that may be offline. Instructions must not be lost; when the edge Agent reconnects it must be able to retrieve all offline messages and process them by priority.

**Flow**

```
Cloud Agent sends instruction to edge Agent mailbox with persistent storage
Edge Agent is offline; message waits in the mailbox
Edge Agent reconnects and subscribes to its own mailbox
Retrieves messages by priority: urgent before normal before notify
Processes and sends response to cloud Agent mailbox
```

**Core requirements**: Offline messages, priority queue, seamless reconnection after network disruption, different TTL values corresponding to different levels of importance.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.INBOX.{edge_mail_id}.urgent` | Cloud sends urgent instruction, persist=true, ttl=86400 |
| PUB | `$mq9.AI.INBOX.{edge_mail_id}.normal` | Cloud sends ordinary data, persist=true, ttl=3600 |
| SUB | `$mq9.AI.INBOX.{edge_mail_id}.*` | Edge Agent subscribes after reconnecting; retrieves messages by priority |
| PUB | `$mq9.AI.INBOX.{cloud_mail_id}.normal` | Edge Agent reports telemetry data |

Same INBOX subject, different priority suffixes, different TTL values. Edge Agent gets urgent before normal before notify after reconnecting; messages are not lost during network outages.

---

### Scenario 6: Human-Machine Hybrid Workflow

**Problem**

An Agent reaches a node in an automated process that requires human judgment. The Agent cannot wait indefinitely; human approvers don't need to be online in real time. Both parties collaborate asynchronously via mailboxes, each at their own pace.

**Flow**

```
Agent reaches the human-in-the-loop node
Sends approval request to human approver's mailbox with reply_to and deadline
Agent continues processing other tasks without blocking
Approver handles it when available
Approval result is sent back to Agent mailbox
Agent receives result and continues executing the process
```

**Core requirements**: Asynchronous human-machine collaboration, bidirectional mailboxes, correlation_id pairing, deadline for timeout control.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.INBOX.{approver_id}.urgent` | Agent sends approval request with reply_to and correlation_id |
| SUB | `$mq9.AI.INBOX.{approver_id}.*` | Approver subscribes to their own mailbox |
| PUB | `$mq9.AI.INBOX.{mail_id}.urgent` | Approver sends result to reply_to address |
| SUB | `$mq9.AI.INBOX.{mail_id}.*` | Agent subscribes to its own mailbox; receives result and continues execution |

Human approvers and Agents use exactly the same protocol — mailboxes are identical; no separate interface design for humans. `correlation_id` ensures the Agent can match which approval corresponds to which request.

---

### Scenario 7: Agent Requests Another Agent and Waits for a Response

**Problem**

Agent A needs a capability provided by Agent B. This is a request-reply pattern, but B is not required to be online. If B is offline, the request waits in the mailbox; B processes it on coming online and replies. Asynchronous request-response.

**Flow**

```
Agent A sends request to Agent B's mailbox with correlation_id and reply_to
Agent B may be offline; message is persisted and waiting
Agent B comes online and picks up the request to process
Agent B sends response to Agent A's reply_to address
Agent A matches request and response via correlation_id
```

**Core requirements**: Asynchronous request-reply, correlation_id pairing, request waits if B is offline, timeout fallback.

**Protocol combination**

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.INBOX.{B_mail_id}.normal` | A sends request to B's mailbox with correlation_id and reply_to |
| SUB | `$mq9.AI.INBOX.{B_mail_id}.*` | B subscribes to its own mailbox and picks up the request to process |
| PUB | `$mq9.AI.INBOX.{A_mail_id}.normal` | B sends response to A's reply_to address |
| SUB | `$mq9.AI.INBOX.{A_mail_id}.*` | A subscribes to its own mailbox and matches the response via correlation_id |

Two mailboxes, four operations, complete asynchronous request-reply. The essential difference from HTTP: if B is offline the request does not fail — the message waits in the mailbox. A also does not block; it continues processing other things and matches the response when it arrives.

---

### Scenario 8: Agent Registers Capabilities; Other Agents Discover It

**Problem**

In a complex Agent system, various specialized Agents provide different capabilities. When Agent A needs a certain capability, how does it know who can provide it? It cannot rely on hard-coding or a centralized registration service — a decentralized capability discovery mechanism is needed.

**Flow**

```
Agent starts and writes its capability declaration to $mq9.AI.STATUS.{mail_id}
When another Agent needs a certain capability:
  Option 1: Subscribe to $mq9.AI.STATUS.* and filter locally for Agents with that capability
  Option 2: Broadcast a capability query; Agents with the capability respond to the discoverer's mailbox
After finding the Agent with the capability, connect to Scenario 7 to send the request
```

**Core requirements**: Decentralized capability discovery, reuse of the STATUS subject, two discovery modes chosen by scenario.

**Protocol combination**

Option 1: Passive discovery (for Agents already in a listening state)

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.STATUS.{mail_id}` | Agent reports status; capabilities field contains capability declarations |
| SUB | `$mq9.AI.STATUS.*` | Discoverer subscribes to all status and filters locally for Agents with the target capability |

Option 2: Active discovery (when a capability is needed temporarily)

| Operation | Subject | Description |
|-----------|---------|-------------|
| PUB | `$mq9.AI.BROADCAST.capability.query` | Discoverer broadcasts a capability query with reply_to |
| SUB | `$mq9.AI.BROADCAST.capability.query` | Agents with the capability subscribe and respond when idle |
| PUB | `$mq9.AI.INBOX.{discoverer_id}.normal` | Agent with the capability responds to the discoverer's mailbox |

After discovery, connect to Scenario 7 to send a request to the target Agent's mailbox. Capability declarations reuse the STATUS subject, introducing no new subjects and keeping the protocol namespace minimal.

## Protocol Design

---

### Subject Space

```
$mq9.AI.
  ├── MAILBOX.
  │     └── CREATE                    # Request mailbox, returns mail_id
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # Urgent messages
  │           ├── normal              # Regular messages
  │           └── notify              # Notifications
  │
  ├── STATUS.
  │     └── {mail_id}               # Status reporting + capability declaration
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # Event broadcast
```

---

### Core Concepts

**mail_id**: The unique identity of each Agent, also its mailbox address. Requested via `$mq9.AI.MAILBOX.CREATE`; the system returns a globally unique ID. Identity is acquired on call — no registration required.

**TTL**: Every mailbox and message has a TTL bound to the Agent's lifecycle. While the Agent is alive, heartbeats renew it; when the Agent ceases to exist, TTL expiry triggers automatic cleanup.

**persist**: The persistence strategy for messages. `$mq9.AI.INBOX.*` defaults to persistent; `$mq9.AI.BROADCAST.*` defaults to non-persistent; can be explicitly overridden.

**priority**: Mailboxes support three priority levels: urgent, normal, notify. Same-priority delivery is FIFO; higher priority is processed first.

**token**: Lightweight identity verification. Returned when a mailbox is created; carried when sending messages to verify mailbox ownership without complex access control.

**retain=latest**: Only the most recent message is retained for each subject; new messages overwrite old ones. Used in status reporting scenarios.

---

### $mq9.AI.MAILBOX.CREATE — Request a Mailbox

```bash
# Request
REQ $mq9.AI.MAILBOX.CREATE
{
  "ttl": 3600                          # Mailbox time-to-live (seconds)
}

# Response
{
  "mail_id": "agt-uuid-001",          # Globally unique ID
  "token": "tok-xxx",                  # Identity token
  "inbox": "$mq9.AI.INBOX.agt-uuid-001"    # Mailbox address
}
```

---

### $mq9.AI.INBOX.{mail_id}.{priority} — Point-to-Point Mailbox

```bash
# Send message to mailbox
PUB $mq9.AI.INBOX.{mail_id}.{priority}
{
  "from": "agt-uuid-002",              # Sender, required
  "type": "task_result",              # Message type, required
  "correlation_id": "req-uuid-001",   # Request-response pairing, fill as needed
  "reply_to": "$mq9.AI.INBOX.agt-uuid-A.normal",  # Reply address, fill as needed
  "payload": { ... },                 # Business data
  "ts": 1234567890                    # Send timestamp (milliseconds), required
}
persist=true   ttl=3600

# Subscribe to mailbox (all priorities)
SUB $mq9.AI.INBOX.{mail_id}.*

# Subscribe to mailbox (urgent only)
SUB $mq9.AI.INBOX.{mail_id}.urgent
```

**priority values**

| Value | Description | Default persistent | Suggested TTL |
|-------|-------------|-------------------|---------------|
| urgent | Urgent messages, processed first | true | 86400s |
| normal | Regular messages, processed in order | true | 3600s |
| notify | Notifications, background processing | false | — |

---

### $mq9.AI.STATUS.{mail_id} — Status Reporting and Capability Declaration

```bash
# Report status (including capability declaration)
PUB $mq9.AI.STATUS.{mail_id}
{
  "mail_id": "agt-uuid-002",
  "status": "running",                 # online | running | idle | offline
  "task": "analysis-001",             # Current task, null if none
  "load": 0.6,                        # Load 0-1
  "capabilities": [
    {
      "name": "data.analysis",
      "version": "1.0",
      "description": "Data analysis capability",
      "input_schema": { ... },
      "output_schema": { ... }
    }
  ],
  "ts": 1234567890
}
retain=latest   ttl=60

# Subscribe to all Agent status
SUB $mq9.AI.STATUS.*

# Query a specific Agent's latest status
REQ $mq9.AI.STATUS.{mail_id}
→ RESP { latest status }
```

---

### $mq9.AI.BROADCAST.{domain}.{event} — Event Broadcast

```bash
# Publish broadcast event
PUB $mq9.AI.BROADCAST.{domain}.{event}
{
  "from": "agt-uuid-004",
  "type": "event_type",
  "severity": "high",                  # low | medium | high | critical, fill as needed
  "payload": { ... },
  "ts": 1234567890
}
persist=false   # Non-persistent by default; explicitly set persist=true for important broadcasts

# Subscribe to all events in a single domain
SUB $mq9.AI.BROADCAST.system.*

# Subscribe to a specific event across all domains
SUB $mq9.AI.BROADCAST.*.anomaly

# Subscribe to all broadcasts
SUB $mq9.AI.BROADCAST.#

# Competitive consumption (only one subscriber receives each message)
SUB $mq9.AI.BROADCAST.task.available   queue_group=task.workers
```

**Suggested domain naming**

| domain | Description |
|--------|-------------|
| system | System-level events, e.g., anomalies, restarts, scaling |
| task | Task-related, e.g., available, completed, failed |
| capability | Capability discovery, e.g., query, response |
| data | Data change events |
| {custom} | Business domain custom |

---

### Common Message Field Specification

All `$mq9.AI.*` messages are recommended to include the following common fields:

```json
{
  "from": "mail_id",           # Sender, required
  "type": "message_type",       # Message type, required
  "ts": 1234567890,             # Send timestamp (milliseconds), required
  "correlation_id": "uuid",     # Request-response pairing, required for request/response
  "reply_to": "$mq9.AI.INBOX...",   # Reply address, fill when a response is expected
  "deadline": 1234567890,       # Expected processing deadline, fill as needed
  "payload": { ... }            # Business data, fill as needed
}
```

**Suggested type values**

| type | Description | Applicable scenarios |
|------|-------------|---------------------|
| task_result | Task result | Scenarios 1, 3 |
| request | Request | Scenario 7 |
| response | Response | Scenario 7 |
| approval_request | Approval request | Scenario 6 |
| approval_result | Approval result | Scenario 6 |
| task_accepted | Task confirmation | Scenario 3 |
| capability_response | Capability response | Scenario 8 |
| command | Instruction | Scenario 5 |
| telemetry | Telemetry data | Scenario 5 |

## Design Principles

**Semantics encoded in the address**: seeing the subject immediately conveys the intent of the message without consulting documentation. `$mq9.AI.INBOX.*` is a mailbox, `$mq9.AI.STATUS.*` is status, `$mq9.AI.BROADCAST.*` is broadcast — the hierarchy is clear, with no mixing.

**Minimal subject space**: four top-level subjects cover eight scenarios with no redundancy and no overlap. When adding new scenarios, prefer reusing existing subjects rather than extending the namespace.

**Persistence on demand**: persistence is not forced for all messages; the sender explicitly declares it based on message semantics and business importance.

**Decentralized**: no centralized registration service, no centralized routing table. Agents address each other directly via subjects and achieve decentralized service discovery through broadcasts and status reporting.

**Backward compatible**: the semantic meaning of published `$mq9.AI.*` subjects is not changed lightly. New capabilities are added by introducing new subjects, not by modifying the semantics of existing ones.
