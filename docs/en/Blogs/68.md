# $mq9.AI.* Protocol Design

## Overview

`$mq9.AI.*` is mq9's protocol for Agent asynchronous communication. Three commands cover all Agent communication scenarios: create mailbox/channel, point-to-point delivery, and public broadcast.

Core problem solved: **Asynchronous communication between Agents — sender and receiver do not need to be online simultaneously.**

mq9 only solves the communication problem — how to reliably deliver messages. What the message contains, how it is used, and what the business semantics are — mq9 does not care. Message content is a byte array: not parsed, not validated, not restricted.

---

## Core Concepts

**mail_id**: A globally unique communication address returned when creating an INBOX via `MAILBOX.CREATE`. Not bound to Agent identity — one Agent can apply for different mail_ids for different tasks. Leave it alone when done; TTL handles cleanup automatically.

**TTL**: Declared at INBOX or BROADCAST creation time; the mailbox is automatically destroyed on expiry and all messages are cleaned up. No explicit delete operation. TTL is fixed by the first CREATE call; subsequent CREATEs do not override it.

**priority**: INBOX supports three priority levels: `urgent`, `normal`, `notify`. Same priority is FIFO; across priorities, higher priority is processed first.

**token**: Returned when creating an INBOX. Anyone who knows the mail_id can send messages; the token is only used for mailbox management operations. BROADCAST has no token.

**msg_id**: A unique identifier per message; used by clients for deduplication.

**Subscription semantics**: INBOX and BROADCAST behave identically — each subscription delivers all non-expired messages in full. No read/unread distinction, no consume offset tracking. Zero consumer state on the server.

**Message flow**: Message arrives → written to storage → pushed to online subscribers in real time; if offline, waits → delivered in full on next subscription.

**CREATE idempotency**: Creating the same INBOX or BROADCAST again does not error; it silently succeeds. TTL is fixed by the first creation.

**No state queries**: mq9 does not expose "does this mailbox exist" or "does this channel exist" query interfaces. CREATE always returns success.

## Commands

| Command | Subject | Direction | Description |
|---------|---------|-----------|-------------|
| `MAILBOX.CREATE` | `$mq9.AI.MAILBOX.CREATE` | REQ | Create a mailbox or broadcast channel. INBOX returns mail_id + token; BROADCAST returns success |
| `INBOX` | `$mq9.AI.INBOX.{mail_id}.{priority}` | PUB | Point-to-point message delivery, persisted, supports offline receipt |
| `INBOX` | `$mq9.AI.INBOX.{mail_id}.*` | SUB | Subscribe to a mailbox; receive all non-expired messages in full |
| `BROADCAST` | `$mq9.AI.BROADCAST.{domain}.{event}` | PUB | Public broadcast, persisted, anyone can publish |
| `BROADCAST` | `$mq9.AI.BROADCAST.{domain}.{event}` | SUB | Subscribe to broadcast channel; full delivery, supports wildcards and queue groups |
| `BROADCAST` | `$mq9.AI.BROADCAST.system.*` | SUB | Subscribe to the system bulletin board (built-in, TTL never expires) |

---

## Relationship Between INBOX and BROADCAST

Both share the same underlying behavior: persistence, TTL auto-cleanup, full delivery on subscription, CREATE idempotency. The difference is the permission model.

| | INBOX (private mailbox) | BROADCAST (public channel) |
|---|---|---|
| CREATE returns | mail_id + token | success (no mail_id, no token) |
| Who can send | Anyone (with mail_id) | Anyone |
| Who can receive | Only the mailbox owner | Anyone |
| Authentication | token for mailbox management | None |
| Priority | urgent / normal / notify | None |
| Use case | Private communication | Public forum / bulletin board |

## $mq9.AI.MAILBOX.CREATE

A unified creation entry point. The subject prefix in the request body distinguishes whether an INBOX or BROADCAST is being created.

### Create a Private Mailbox (INBOX)

```bash
REQ $mq9.AI.MAILBOX.CREATE
{
  "ttl": 3600,              # mailbox lifetime (seconds)
  "subject": "$mq9.AI.INBOX"
}

# Response
{
  "mail_id": "m-uuid-001",
  "token": "tok-xxx",
  "inbox": "$mq9.AI.INBOX.m-uuid-001"
}
```

### Create a Broadcast Channel (BROADCAST)

```bash
REQ $mq9.AI.MAILBOX.CREATE
{
  "ttl": 3600,              # channel lifetime (seconds)
  "subject": "$mq9.AI.BROADCAST.pipeline.complete"
}

# Response
{
  "subject": "$mq9.AI.BROADCAST.pipeline.complete"
}
```

The server identifies the `$mq9.AI.BROADCAST.` prefix, does not generate a mail_id or token. After creation, anyone can publish to or subscribe from the channel.

CREATE is idempotent. If the channel already exists, it silently returns success; TTL is fixed by the first creation and is not overridden.

## $mq9.AI.INBOX.{mail_id}.{priority}

Point-to-point message delivery. Anyone who knows the mail_id can send; no token required.

```bash
PUB $mq9.AI.INBOX.{mail_id}.{priority}
{
  "msg_id": "msg-uuid-001",
  "from": "m-uuid-002",
  "type": "task_result",
  "correlation_id": "req-uuid-001",
  "reply_to": "$mq9.AI.INBOX.m-uuid-002.normal",
  "deadline": 1234567890,
  "payload": { ... },
  "ts": 1234567890
}
```

Subscribing:

```bash
SUB $mq9.AI.INBOX.{mail_id}.*        # all priorities
SUB $mq9.AI.INBOX.{mail_id}.urgent   # urgent only
```

### priority

| Value | Description | Persisted | Suggested TTL |
|-------|-------------|-----------|---------------|
| urgent | Urgent, processed first | Yes | 86400s |
| normal | Normal, processed in order | Yes | 3600s |
| notify | Notification, background processing | No | short |

Messages expire together with their mailbox. When the mailbox TTL expires, all messages inside are destroyed.

---

## $mq9.AI.BROADCAST.{domain}.{event}

Public broadcast. The channel must be CREATEd first; after that, anyone can publish to it and anyone can subscribe. Messages are persisted; offline subscribers receive all non-expired messages in full upon reconnection.

```bash
PUB $mq9.AI.BROADCAST.{domain}.{event}
{
  "msg_id": "msg-uuid-002",
  "from": "m-uuid-004",
  "type": "event_type",
  "severity": "high",
  "reply_to": "$mq9.AI.INBOX.m-uuid-004.normal",
  "payload": { ... },
  "ts": 1234567890
}
```

Subscribing:

```bash
SUB $mq9.AI.BROADCAST.system.*              # all events in one domain
SUB $mq9.AI.BROADCAST.*.anomaly             # a specific event across domains
SUB $mq9.AI.BROADCAST.#                     # all broadcasts
SUB $mq9.AI.BROADCAST.task.available  queue_group=task.workers  # competing consumers
```

### System Bulletin Board

`$mq9.AI.BROADCAST.system.*` is automatically created at system startup, TTL never expires, and cannot be deleted. Behavior is identical to user-created broadcast channels.

| Subject | Purpose |
|---------|---------|
| `$mq9.AI.BROADCAST.system.capability` | Capability announcement |
| `$mq9.AI.BROADCAST.system.status` | Status announcement |
| `$mq9.AI.BROADCAST.system.channel` | Channel announcement |

### Recommended domain naming

| domain | Description |
|--------|-------------|
| system | System bulletin board, built-in |
| task | Task-related |
| data | Data change events |
| {custom} | User-defined business domains |

## Protocol Summary

| Subject | Direction | Persisted | Description |
|---------|-----------|-----------|-------------|
| `$mq9.AI.MAILBOX.CREATE` | REQ/RESP | — | Create mailbox or broadcast channel |
| `$mq9.AI.INBOX.{id}.urgent` | PUB/SUB | Yes | Urgent messages |
| `$mq9.AI.INBOX.{id}.normal` | PUB/SUB | Yes | Normal messages |
| `$mq9.AI.INBOX.{id}.notify` | PUB/SUB | No | Notifications |
| `$mq9.AI.BROADCAST.system.capability` | PUB/SUB | Yes | System bulletin board: capability |
| `$mq9.AI.BROADCAST.system.status` | PUB/SUB | Yes | System bulletin board: status |
| `$mq9.AI.BROADCAST.system.channel` | PUB/SUB | Yes | System bulletin board: channel |
| `$mq9.AI.BROADCAST.{domain}.{event}` | PUB/SUB | Yes | User-defined broadcasts |
