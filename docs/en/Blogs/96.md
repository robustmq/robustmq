# mq9 as A2A Transport: Design Thinking

## Preliminary Thinking

Before starting the design, I worked through a few questions first.

**Question one: What exactly is mq9.**

My earliest framing was "the protocol standard for async Agent communication" — too broad. I narrowed it down to a more concrete position: **a private communication pipeline between Agents inside an enterprise**. One sentence: deploy mq9 and every Agent in your company can freely register and communicate, both real-time and async.

This positioning has several advantages. The market is clear — target customers are mid-to-large enterprises deploying Agents. The pain point is real — these companies have 5+ Agent applications, multiple teams using different frameworks (LangGraph, LangChain, AutoGen, CrewAI), and no unified communication infrastructure. The business model works — open-source core + enterprise edition advanced features, similar to Kafka and Confluent.

**Question two: What is the relationship between mq9 and A2A.**

After several rounds of discussion, it became clear — they solve problems at different layers and don't conflict. A2A is the protocol content layer, defining what information Agents exchange (Task, Message, Artifact, state machine). mq9 is the transport layer, defining how that information gets from one Agent to another. The two are orthogonal.

Concretely, users have three usage modes:

1. Fully use mq9 — registration, discovery, and communication all inside mq9
2. mq9 for discovery + mq9 for A2A async transport
3. External service discovery + mq9 for A2A async transport

The mq9 protocol layer doesn't need to branch for these three scenarios. As long as the mailbox abstraction is right, all three are supported.

**Question three: How big should mq9 be.**

Controlling scope is the most important discipline for an infrastructure project. What mq9 does and doesn't do must be defined early.

mq9 does: mailbox abstraction, async communication, simple Agent registration and discovery.

mq9 does not do: protocol content abstraction (not an A2A replacement), capability description schema (let A2A, MCP, and the application layer define that), complex service discovery (not competing with etcd/Consul), Agent orchestration (not competing with LangGraph/AutoGen), automatic routing and intent resolution (that's an upper-layer concern).

Clear boundaries bring antifragility — whichever upper-layer solution wins, mq9 can serve it; whichever discovery solution wins, mq9 can work with it. mq9 only cares about "how a message gets from one mailbox to another." That problem doesn't go out of date.

Now for the concrete design.

## Core Abstraction: mailbox

mq9 has exactly one core abstraction — the mailbox.

A mailbox is a lightweight, disposable communication resource. An Agent that wants to communicate creates a mailbox, uses it, and discards it. No session lifecycle to manage. This is the mq9 version of Unix's "everything is a file" philosophy — everything is a mailbox.

**Mailbox characteristics:**

- **Lightweight**: Near-zero creation cost. A single broker should support 1 million+ concurrently active mailboxes.
- **Disposable**: Every mailbox has a TTL and is automatically cleaned up. Agents don't manage lifecycle.
- **Persistent**: Messages are persisted by default. Subscribers automatically receive stored messages when they come online.
- **Opaque**: mq9 does not parse message content. The message body is a byte array.
- **Unidirectional**: Each mailbox is a one-way channel, pub in and sub out. Two mailboxes for bidirectional communication.

**Security model: an unguessable mail_address is the security boundary.**

Knowing the mail_address means you can publish and subscribe. Not knowing it means you can't do anything. No tokens, no ACLs. System-generated private mail_addresses use UUIDs; public mail_addresses use user-defined meaningful names.

This security model is simple but sufficient. Private scenarios rely on unguessability; public scenarios rely on application-layer authentication (signatures, tokens). mq9 does not impose security mechanisms at the protocol layer.

## mail_address Format Specification

A mail_address is the unique identifier for a mailbox. The naming rules are strict.

**Character set:** lowercase letters (a-z), digits (0-9), dot (.)

**Length:** 1 to 128 characters

**Case:** strictly lowercase; a mail_address containing uppercase characters will be rejected by the broker

**Positional rules:** `.` may only appear in the middle; the first and last characters must be a lowercase letter or digit; consecutive `.` are not allowed

**Semantics:** opaque string; `.` does not participate in protocol routing or matching

**Encoding:** URL percent-encoding is not allowed

Valid examples: `task.001`, `agent.inbox`, `analytics.result`, `acme.org.task.queue`

Invalid examples: `task-001` (hyphen), `Task.001` (uppercase), `task..001` (consecutive dots), `.task.001` (leading dot)

## URI Scheme: mq9:// and mq9s://

mq9 expresses mailbox addresses in standard URI form:

```
mq9://host[:port]/{mail_address}    # plaintext
mq9s://host[:port]/{mail_address}   # TLS encrypted
```

Full examples:

```
mq9://127.0.0.1:4332/task.001
mq9://broker.example.com/agent.inbox
mq9s://broker.example.com:4332/analytics.result
mq9s://mq9.acme.org/acme.task.queue
mq9://127.0.0.1:4332/session.20260502
mq9s://broker.example.com/order.processing.urgent
mq9s://broker-east-1.acme.org:9332/agent.001.inbox
```

The key value of this design: **A2A protocol needs zero changes**. A2A's `pushNotification.url` field is just a URL. HTTP webhook is `https://...`, mq9 mailbox is `mq9://...`. From the protocol layer it's the same field; from the implementation layer it's two different transports. The SDK dispatches based on URL scheme.

The A2A working group doesn't need to do anything — their protocol already allows any URL scheme. This exploits the extension space the protocol left open, not a protocol change.

## Protocol Overview

```
Mailbox Management:
  $mq9.AI.MAILBOX.CREATE                                 Create a mailbox

Mailbox Communication:
  $mq9.AI.MAILBOX.MSG.{mail_address}                     Send/subscribe messages (normal priority)
  $mq9.AI.MAILBOX.MSG.{mail_address}.urgent              Urgent messages
  $mq9.AI.MAILBOX.MSG.{mail_address}.critical            Highest priority messages

Mailbox Query:
  $mq9.AI.MAILBOX.QUERY.{mail_address}                   Query messages in a mailbox

Mailbox Cleanup:
  $mq9.AI.MAILBOX.MSG.DELETE.{mail_address}.{msg_id}     Delete a specific message
  (mailbox does not support explicit destruction; TTL handles cleanup)

Agent Management:
  $mq9.AI.AGENT.REGISTER                                 Register an Agent
  $mq9.AI.AGENT.UNREGISTER                               Unregister an Agent
  $mq9.AI.AGENT.HEARTBEAT                                Heartbeat renewal
  $mq9.AI.AGENT.DISCOVER                                 Discover Agents by tag
```

**A few intentional design decisions:**

**No explicit mailbox destruction.** Mailboxes are cleaned up entirely by TTL. Agents don't manage mailbox lifecycle — crashes, forgotten calls, and bad timing don't affect the system. Simplicity is a design virtue, not a missing feature.

**LIST and GET merged into QUERY.** Both query messages inside a mailbox, so merging them gives consistent semantics. With a key parameter it returns the latest message for that key; without one it returns a metadata list.

**AGENT.DISCOVER stays separate.** It queries the registry, a different concept from mailbox queries. Merging them would destroy interface clarity.

**Priority expressed via subject suffix.** Subscribers can precisely subscribe to a specific priority level (NATS native filtering), which is more efficient than putting it in a header.

## Key Capability One: Key-based Compaction

This is the core capability that distinguishes mq9 from an ordinary message queue.

**Background:** A2A Task state pushes have two types of events — TaskStatusUpdateEvent (state change) where only the latest matters, and TaskArtifactUpdateEvent (artifact) where every one matters. Without distinction, a three-day task accumulates thousands of status updates, and a reconnecting client gets buried.

**Solution:** Messages can carry a key (in the header). Messages with the same key keep only the latest in the storage layer.

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.3 }

PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.7 }
# previous message overwritten

PUB $mq9.AI.MAILBOX.MSG.task.001.out
Body: { "type": "artifact", "data": {...} }
# no key, treated as append stream
```

**Key design points:**

- **Key is a message property, not a mailbox property.** Different keys in the same mailbox independently keep their latest version; keyless messages are append-only.
- **Compaction strategy declared at mailbox creation.** It affects storage engine indexing and must be set upfront.
- **Real-time delivery is not compacted; storage layer compacts periodically.** Online subscribers receive all state changes; clients reconnecting offline see only the latest version. This semantic is exactly right for A2A — live rendering of the flow while online, fast state recovery when reconnecting.

**mq9's dual identity:** This capability gives mq9 both stream (message flow) and KV store (state storage) semantics. Kafka provides similar capability through Streams + Compacted Topics; NATS through KV Store + JetStream — but both are two separate concepts composed together. mq9 unifies them under the single mailbox abstraction.

## Key Capability Two: DeliverPolicy

Subscribers can choose a delivery policy at subscribe time:

```
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=all     # full history (default)
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=new     # new messages only
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=from --since=1234567890   # from timestamp
```

**Why it's needed:** In long-running task scenarios, "subscribe and get full history" breaks down. Client Agents may have short lifecycles, going offline and coming back. If every reconnect pushes thousands of historical messages, performance explodes.

DeliverPolicy lets clients choose what to fetch and what to skip on reconnect. Combined with key-based compaction, a reconnecting client gets only the latest state and unconsumed artifacts — not buried in history.

## Key Capability Three: Direct State Access via QUERY

Not just subscriptions — you can also directly query the current state of a mailbox:

```
# Query all message metadata
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{}'

# Query latest message for key=status
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"key": "status"}'

# Most recent 10 messages
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"limit": 10}'

# Messages after a timestamp
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"since": 1234567890}'
```

**Why it's needed:** Not every client needs a subscription. Multiple observers checking Task state — each querying independently is lighter than each maintaining a subscription. State after Task completion is still queryable (within TTL), no extra storage needed.

QUERY elevates mq9 from a "message queue" to a "message queue + queryable state store." This capability is particularly valuable for state-driven protocols like A2A.

## Agent Registration and Discovery

**The fundamental difference between mq9 registry and traditional registries:**

Traditional registries (etcd, Consul, Nacos) answer the question "where is the service" — IP:Port, health checks, load balancing. The content of a service is transparent to them; they don't care.

mq9 registry answers the question "who can do this" — what's registered is an A2A AgentCard containing capability descriptions, skills, and tags. Discovery is capability matching, not address lookup. This is purpose-built for the Agent ecosystem — traditional registries have no concept of this.

**Registration: each Agent registers with an A2A AgentCard structure; mailbox is the unique identifier.**

```
# Register
PUB $mq9.AI.AGENT.REGISTER
Body: {
  "mailbox": "mq9://broker/agent.translator.inbox",
  "card": {
    "name": "Translator Agent",
    "description": "Translates text between languages",
    "skills": [
      { "name": "translate", "tags": ["en-zh", "zh-en", "domain:legal"] }
    ]
  },
  "ttl": 300
}

# Discover (current: structured search by tag)
nats request '$mq9.AI.AGENT.DISCOVER' '{"tag": "translate"}'
# Returns: [{"mailbox": "mq9://broker/agent.translator.inbox", "card": {...}}]

# Heartbeat renewal
PUB $mq9.AI.AGENT.HEARTBEAT
Body: { "mailbox": "mq9://broker/agent.translator.inbox" }

# Unregister
PUB $mq9.AI.AGENT.UNREGISTER
Body: { "mailbox": "mq9://broker/agent.translator.inbox" }
```

**QUERY evolution roadmap:**

At registration time, the broker automatically indexes AgentCard content, enabling progressively richer QUERY capabilities:

- **Current (structured search):** Filter by `skills.name`, `skills.tags`, `name`, and other fields — broader coverage than plain tag string matching, zero AI dependency.
- **Long-term (semantic search):** Vector embeddings on `description` and `skills[].description`, with a `semantic` field in QUERY. Client Agents describe their intent; mq9 returns the best-matching Agent list.

This roadmap doesn't break existing interfaces — QUERY body fields are additive and backward-compatible. Semantic search doesn't violate the "opaque message body" principle, because AgentCard is machine-readable registration metadata, not a business message.

**A few deliberate restraints:**

- **Eventual consistency is enough; no Raft or strong consistency.** Registration data changes infrequently. Eventual consistency suffices.
- **No namespace or tenant isolation.** Keep the open-source version simple; enterprise edition can add it.
- **No Agent orchestration.** Find the Agent, get the mailbox address — what happens next is the application layer's concern.

## A2A Full Flow Over mq9

Combining all the capabilities above, a complete A2A Task carried over mq9 looks like this:

**1. Server creates inbox and registers**

Server Agent on startup first creates its inbox mailbox, then registers with that mailbox address:

```
# Step 1: create inbox mailbox
PUB $mq9.AI.MAILBOX.CREATE
Body: { "name": "agent.translator.inbox", "ttl": 0 }
# Returns: { "mail_address": "agent.translator.inbox@mq9" }

# Step 2: register AgentCard with the mailbox address
PUB $mq9.AI.AGENT.REGISTER
Body: {
  "mailbox": "mq9://broker/agent.translator.inbox",
  "card": {
    "name": "Translator Agent",
    "description": "Translates text between languages",
    "skills": [
      { "name": "translate", "tags": ["en-zh", "a2a-compatible"] }
    ]
  },
  "ttl": 300
}
```

The mailbox exists before the registration. The registration entry is just "what the owner of this mailbox can do" — the mailbox itself is an independent resource.

**2. Client discovers Server, creates Task bidirectional channels**

Client Agent discovers the Server by tag or semantic query, getting the AgentCard and mailbox address:

```
nats request '$mq9.AI.AGENT.DISCOVER' '{"tag": "en-zh"}'
# Returns: [{"mailbox": "mq9://broker/agent.translator.inbox", "card": {...}}]
```

With the address in hand, Client creates dedicated mailboxes for this Task:

```
# Task in mailbox (client → server)
PUB $mq9.AI.MAILBOX.CREATE
Body: { "name": "task.001.in", "ttl": 86400 }

# Task out mailbox (server → client, with key compaction)
PUB $mq9.AI.MAILBOX.CREATE
Body: {
  "name": "task.001.out",
  "ttl": 86400,
  "compaction": { "enabled": true, "strategy": "key-latest" }
}
```

in and out have different compaction strategies — in is a continuous stream of client input, out needs key compaction for server state updates.

**3. Client notifies Server of channel addresses**

```
PUB $mq9.AI.MAILBOX.MSG.agent.translator.inbox
Body: {
  "task_id": "task-001",
  "in_mailbox": "mq9://broker/task.001.in",
  "out_mailbox": "mq9://broker/task.001.out"
}
```

**4. Client delivers initial task; Server subscribes to in and writes to out**

```
# Client continuously writes to in
PUB $mq9.AI.MAILBOX.MSG.task.001.in
Body: { /* full A2A tasks/send payload */ }

# Server subscribes to in
SUB $mq9.AI.MAILBOX.MSG.task.001.in --deliver=all

# Server writes status (key compaction, only latest kept)
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.3 }

# Server writes artifact (no key, all kept)
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Body: { "type": "artifact", "data": {...} }
```

**5. Multi-turn scenario**

Server enters input-required:

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "input-required", "prompt": "Need..." }
```

Client adds input directly to in, no new mailbox needed:

```
PUB $mq9.AI.MAILBOX.MSG.task.001.in
Body: { "type": "input", "content": "..." }
```

Multi-turn conversation is natively supported with no state switching.

**6. Client goes offline and reconnects**

```
SUB $mq9.AI.MAILBOX.MSG.task.001.out --deliver=all
# Gets: latest status (compacted to 1 message) + all artifacts since going offline
# Not buried under thousands of historical status messages
```

Or query the current state directly:

```
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"key": "status"}'
```

**7. Task completes**

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "completed", "result": "..." }
```

No active cleanup needed. Wait for TTL expiry.

## Deeper Advantages of This Design

**A2A working group evolution does not affect mq9.** A2A adds fields, changes the Task state machine, adjusts message format — mq9 protocol layer changes zero lines of code. All changes are inside the message body, which mq9 doesn't parse. Kafka went 13 years without changing its core protocol because it doesn't parse messages. mq9 should have the same stability.

**Multi-Agent collaboration is natively supported.** If a Task requires N Agents to collaborate, each pair communicates over one mailbox pair. N Agents use O(N) mailboxes (hub-spoke) or O(N²) mailboxes (mesh). mq9 protocol layer is unchanged.

**Debugging and auditing come for free.** All messages for each Task are in the mailbox, queryable within TTL. When ops needs to debug "why did the Task fail," they directly inspect the message history of the in and out mailboxes. Enterprise-grade capability at zero cost.

**Zero-change integration with existing A2A ecosystem.** A2A SDKs only need to change `pushNotification.url` from `https://...` to `mq9://...` and add one mq9 transport adapter. Nothing else moves.

**Multiple integration paths.** Full-stack mq9 is the deepest integration (mailbox + register). mq9 for async transport only also works (keep A2A well-known discovery). mq9 for discovery only also works (other communication over HTTP). All three entry points are valid.

## Design Principles from This Exercise

**Protocol layer orthogonality.** mq9 does not parse A2A; A2A does not depend on mq9. The two combine at the application layer through the `mq9://` URL scheme.

**Simplicity first.** Use TTL cleanup instead of adding DESTROY. Merge when possible (LIST/GET → QUERY). Avoid introducing concepts unless necessary (no schema, no namespace, no strong consistency).

**Restrained registry.** Only register/discover/heartbeat — scope strictly controlled, not competing with general-purpose solutions like etcd.

**Antifragility by design.** Not dependent on any single standard succeeding. A2A takes off and mq9 benefits; A2A doesn't and mq9 still works. MCP extends to Agent-Agent and mq9 can carry it. Any upper-layer protocol needs a transport layer. That doesn't go out of date.

mq9's value proposition is simple: **a private communication pipeline between Agents inside an enterprise**. With mq9 registration, enterprises run one fewer component — Agent communication in a single system. For users with existing discovery, mq9 supports that too.

With this positioning, mq9 has a small protocol surface area but a powerful capability combination — mailbox abstraction + priority + key compaction + DeliverPolicy + QUERY + lightweight registry. Sufficient, stable, small, long-lived.

That's enough.
