# mq9: RobustMQ's Communication Layer Designed for AI Agents

## Starting From a Real Problem

People have Lark, DingTalk, and email. I send something, you read it when you have time — no need to be online at the same time. We take this for granted, because the asynchronous communication infrastructure for humans is already very mature.

What about Agent-to-Agent communication?

Agents are not services — they are temporary, ceasing to exist when a task is done, possibly existing for only a few seconds. Today, when Agent A sends a message to Agent B and B is offline, the message is simply lost. There is no standard mechanism to guarantee that "the message I sent will be received when the other party comes online." Every team building multi-Agent systems is working around this problem with their own temporary solutions — Redis pub/sub, polling databases, custom task queues. They work, but they are all workarounds.

This is the problem mq9 wants to solve.

---

## What mq9 Is

mq9 is a protocol layer designed specifically for AI Agent communication within RobustMQ, sitting alongside MQTT, Kafka, NATS, and AMQP, built on top of RobustMQ's unified storage architecture.

RobustMQ is a unified messaging engine built in Rust — a single binary, zero external dependencies, natively supporting four protocols: MQTT, Kafka, NATS, and AMQP, with unified storage underneath so a message is written once and consumable by any protocol. This is the foundation RobustMQ has refined over three years.

mq9 is the fifth protocol layer that naturally grew from this foundation. It is not a plugin, not an external integration — it is a native protocol capability of RobustMQ. Just as MQTT was designed specifically for IoT and Kafka was designed specifically for data stream pipelines, mq9 is designed specifically for Agent communication.

**Deploy one RobustMQ instance and all of mq9's capabilities are ready. IoT devices send data via MQTT, analytics systems consume via Kafka, Agents collaborate via mq9 — same broker, same storage, zero bridging, zero replication.**

---

## Why Existing Solutions Are Not Good Enough

Before mq9, there was no good standard answer for Agent communication:

**HTTP** is synchronous. The caller must wait; the other party must be online. Agents go online and offline frequently, and HTTP's assumptions simply do not hold.

**Redis pub/sub** has no persistence. If the other party is offline, the message is lost — delivery cannot be guaranteed.

**Kafka** is too heavy. Designed for large-batch persistent data streams, it has no concept of Agent identity, no temporary mailbox, no priority queue, and topic creation requires administrator operations. Agents are disposable; Kafka's design assumes resources are long-lived and need careful maintenance — a fundamental mismatch.

**Custom task queues** — every team reinvents the wheel, each differently, and they cannot interoperate.

These solutions share a common flaw: **they all assume the other party is online.** The fact that Agents go online and offline frequently is a problem that all existing solutions need to work around, not one that is directly solved.

## The Essential Characteristics of Agent Communication

Understanding mq9's design requires first understanding what fundamentally distinguishes Agent communication from traditional service communication.

**Messages are small, but connections are numerous and relationships are mesh-like.** An Agent message is typically only tens to hundreds of bytes — a task instruction, a status update, a result notification. The challenge of Agent communication is not the size of individual messages, but the number of connections and the complexity of relationships. An Orchestrator communicates simultaneously with dozens of Workers, each of which may in turn be communicating with other Agents, forming a complex mesh topology.

**Point-to-point cannot solve mesh problems.** If A needs to communicate with B, C, D, and E, and B also needs to communicate with C and D — point-to-point requires maintaining n² connections, and any node joining or leaving requires updating all related connections. Broadcast plus wildcard subscription is the correct model for mesh communication.

**Agents are temporary with unpredictable lifecycles.** Tasks complete and Agents cease to exist; they are killed when resources are insufficient; network jitter causes disconnections; version updates cause restarts. An Agent's online time may be only a few seconds. The communication system must be designed specifically for this characteristic — it cannot assume the other party is always online.

These three characteristics determine mq9's three core primitives.

## Three Core Primitives

### Mailbox (Inbox)

The mailbox solves the problem of "I know who to reach, but I don't know if they're online."

Each Agent has its own mailbox address. The sender sends directly to the recipient's mailbox; the message is persisted and waiting. The sender doesn't need to care when the recipient comes online to pick it up.

```bash
# Request a mailbox — identity acquired on call
nats req '$mq9.AI.MAILBOX.CREATE' '{}'
# → {"agent_id": "agt-uuid-001", "token": "tok-xxx"}

# Send a message to another Agent's mailbox
nats pub '$mq9.AI.INBOX.agt-uuid-002.NORMAL' \
  '{"from":"agt-uuid-001","type":"task_result","payload":"..."}'

# Subscribe to your own mailbox
nats sub '$mq9.AI.INBOX.agt-uuid-001.*'
```

Mailboxes support three priority levels: `URGENT`, `NORMAL`, `NOTIFY`. Same-priority delivery is FIFO; urgent messages are processed first and are not buried under ordinary tasks.

Mailboxes are temporary. Agents declare a TTL when registering; heartbeats renew it; expiry triggers automatic destruction. Messages are cleaned up along with the mailbox. Agents don't need to think about resource management at all.

### Broadcast

Broadcast solves the problem of "I don't know who needs this, but those who need it should know."

Publishers don't care about consumers at all — they send and they're done. Subscribers decide what they care about and express interest with wildcards. New Agents joining simply subscribe to the relevant subject without notifying anyone.

```bash
# Broadcast task availability
nats pub '$mq9.AI.BROADCAST.task.available' \
  '{"task_id":"t-001","reply_to":"$mq9.AI.INBOX.agt-001.NORMAL"}'

# Subscribe to all task events
nats sub '$mq9.AI.BROADCAST.task.*'

# Subscribe to anomaly events across all domains
nats sub '$mq9.AI.BROADCAST.*.anomaly'

# Competitive consumption — only one Worker processes each message
nats sub '$mq9.AI.BROADCAST.task.available' --queue task.workers
```

Broadcasts are non-persistent by default — they are time-sensitive; missing them is fine. Enable persistence explicitly for important broadcasts:

```bash
nats pub '$mq9.AI.BROADCAST.system.shutdown' '{}' --persist --ttl 3600
```

### Status Reporting

Status reporting solves the problem of "how does the primary Agent become aware of all sub-Agents' existence and status."

Each Agent periodically writes its own status to `$mq9.AI.STATUS.{agent_id}` using retain=latest semantics — only the most recent message is retained. The primary Agent uses a wildcard subscription, one line of code to be aware of the global state.

```bash
# Agent reports status (including capability declaration)
nats pub '$mq9.AI.STATUS.agt-uuid-002' \
  '{"status":"running","load":0.6,"capabilities":["data.analysis"]}'

# Primary Agent monitors all sub-Agents
nats sub '$mq9.AI.STATUS.*'
```

New Agents joining don't need to register; dying Agents don't need to deregister — exceeding the TTL without reporting causes the status to expire automatically, and the primary Agent detects the death. This is the foundation of decentralized Agent orchestration without any centralized registration service.

## Call It and It Exists — No Configuration Needed

The adoption barrier for mq9 approaches zero.

No registration process, no configuration files, no administrator operations. One command, and the Agent has an identity and a mailbox. Any language's NATS client is directly a mq9 client — the NATS ecosystem is as large as the mq9 integration ecosystem, requiring no new SDKs to be built.

This is not a lazy design choice, but a carefully considered decision. Software for Agents must align with the ancient mental models already baked into the models. NATS's pub/sub, subject addressing, and queue group — these mental models Agents already understand. mq9 doesn't invent new concepts; it stands on what Agents already know and defines semantic conventions, with an adoption cost of zero.

## Eight Real-World Scenarios

mq9's three primitives combine to cover the core communication scenarios for Agent systems.

**Scenario 1: Sub-Agent completes a task, asynchronously notifies the primary Agent.** The sub-Agent sends the result to `$mq9.AI.INBOX.{primary_agent}.NORMAL` when done; the primary Agent doesn't need to block waiting and picks it up when available.

**Scenario 2: Primary Agent monitors all sub-Agent status.** One subscription to `$mq9.AI.STATUS.*`, new Workers detected automatically on join, Worker deaths detected automatically — no Agent list to maintain.

**Scenario 3: Task broadcast, competitive consumption.** The primary Agent broadcasts to `$mq9.AI.BROADCAST.task.available`; capable Workers claim tasks themselves; queue group ensures each task is processed by only one Worker without needing a distributed lock.

**Scenario 4: Anomaly alert broadcast.** An Agent detects an anomaly and broadcasts to `$mq9.AI.BROADCAST.{domain}.anomaly`; Agents with handling capability respond automatically; the publisher doesn't need to know who is handling it.

**Scenario 5: Edge Agent offline message backlog.** Cloud sends instructions to edge Agent at `$mq9.AI.INBOX.{edge}.URGENT`; while the edge is offline the messages are persisted and waiting; after reconnecting, they are processed by priority.

**Scenario 6: Human-machine hybrid workflow.** Agent sends approval request to human approver's `$mq9.AI.INBOX.{approver}.URGENT`; after approval the result is sent back to Agent mailbox; process continues. Human approvers and Agents use exactly the same protocol.

**Scenario 7: Agent requests another Agent and waits for a response.** Asynchronous request-reply — if the other party is offline the request waits in the mailbox; it's processed when online and sent back via reply_to. `correlation_id` ensures request-response pairing.

**Scenario 8: Capability registration and discovery.** Agent writes capability declaration to `$mq9.AI.STATUS.{id}`; other Agents discover Agents with that capability by subscribing to status or broadcasting a query to `$mq9.AI.BROADCAST.capability.query`.

## Why mq9 Is Suitable for AI

None of the individual primitives is novel: priority queue messaging systems have existed for a long time; broadcast pub/sub is everywhere; NATS already achieves lightweight.

But the three combined, designed specifically for Agent characteristics, become something nobody else has.

Agents are asynchronous — solved by the priority queue; messages wait, processed by priority when online. Agents are unaware of each other — solved by broadcast; send it out, those who care come themselves. Agents are temporary — solved by lightweight; acquired on call, gone on expiry, automatically reclaimed by the system.

Three primitives cover all modes of Agent communication, with no scenario outside the coverage.

The most critical aspect is not having many features, but being **light enough**. Three primitives, native NATS SDK, a few lines of code. Adoption cost approaches zero — that is the real barrier. Not a technical barrier, but an adoption barrier. The more Agents that join RobustMQ, the more collaboration opportunities each Agent has, and the greater the value of the entire system. This is a network effect.

---

## Single Machine Is Sufficient; Scale Up on Demand

Most Agent communication scenarios don't need high availability — if a message is lost, retry; if a task fails, re-dispatch. Agents' fault-tolerance model is completely different from traditional systems; Agents are naturally capable of retrying.

mq9 on a single machine can meet a large number of needs. An ordinary cloud VM, one `docker run` command, and the Agent mailbox is up. No cluster to set up, no operations experience required. When high availability is needed, switch to a cluster — the interface remains unchanged and Agents are unaware of the change.
