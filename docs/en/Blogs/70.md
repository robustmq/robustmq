# mq9: What It Is

## One Line

mq9 is RobustMQ's communication layer designed specifically for AI Agents. It gives Agents mailboxes and broadcast channels — send it out, the other party receives it when they come online.

---

## What Problem It Solves

Agents need to communicate. But Agents are not services — they are ephemeral, going online and offline at any time, with a lifecycle that may be only a few seconds.

Agent A sends a message to Agent B; B is offline, and the message is gone. Every team works around this problem with HTTP callbacks, Redis, database polling, or homegrown queues.

mq9 solves it directly: **messages are written to storage first; if the other party is online, they are pushed in real time; if offline, the messages wait and are delivered in full when they come online.**

mq9 only handles communication — how to reliably deliver messages. What the message contains and what the business semantics are — mq9 does not care.

---

## Two Core Concepts

### Mailbox (INBOX)

I know the other party's address; I compose a message and drop it in. Whether they are home or not doesn't matter — the message waits in their mailbox. Messages are split into urgent, normal, and notify. When they get home, they open the mailbox and read urgent messages first.

A mailbox is private — anyone can send to it, but only the owner can receive. Declare TTL at creation; it is automatically destroyed on expiry, no manual deletion needed.

### Broadcast Channel (BROADCAST)

A community bulletin board. Anyone can create one; once created, anyone can post to it, and anyone can read from it. No need to know who will be reading — whoever is interested comes to look. A task posted to the board can only be picked up by one person.

Broadcast channels are also persisted and have TTL. People who were offline can still see non-expired posts after they come back online.

### What They Have in Common

Both require creation, using the same CREATE command. Both are persisted; both have TTL auto-cleanup. Every subscription delivers all non-expired messages in full; no read/unread tracking. CREATE is idempotent — creating again does not error; TTL is fixed by the first creation.

The only difference: a mailbox is private (has mail_id, has token, only the owner can receive); a broadcast channel is public (no mail_id, no token, anyone can publish or subscribe).

### System Bulletin Board

mq9 has three built-in permanent broadcast channels, automatically created by the system, TTL never expires:

- `BROADCAST.system.capability` — Capability announcement: what I can do
- `BROADCAST.system.status` — Status announcement: I'm still alive
- `BROADCAST.system.channel` — Channel announcement: I created a new channel

Once an Agent comes online and subscribes to `BROADCAST.system.*`, it can discover the entire network. No registry service needed.

---

## Three Commands

| Command | Description |
|---------|-------------|
| `MAILBOX.CREATE` | Create a mailbox or broadcast channel |
| `INBOX` | Point-to-point delivery, persisted, three priority levels (urgent / normal / notify) |
| `BROADCAST` | Public broadcast, persisted, anyone can publish or subscribe |

No QUERY — subscribing is querying. No DELETE — TTL handles cleanup automatically.

For detailed protocol design, see the [$mq9.AI.* Protocol Design Document].

---

## Key Design Decisions

- **mail_id is not bound to Agent identity.** One Agent, multiple mailboxes; leave them alone when done; TTL handles cleanup.
- **Two mailbox types.** `standard`: messages accumulate. `latest`: only the latest message is kept, for status reporting.
- **Store first, then push.** Persistence is the default behavior, not an option.
- **Client-side deduplication.** Every message has a msg_id; the server maintains zero consumer state.
- **CREATE is idempotent.** Creating again silently succeeds; TTL is fixed by the first creation. No state query interface.

## Relationship with RobustMQ

mq9 is RobustMQ's fifth native protocol layer, alongside MQTT, Kafka, NATS, and AMQP, sharing the unified storage layer. Deploy one RobustMQ; all mq9 capabilities are ready.

```bash
docker run -d -p 4222:4222 robustmq/robustmq:latest
```

IoT devices send data via MQTT, analytics systems consume via Kafka, Agents collaborate via mq9 — the same broker, the same storage.

---

## Relationship with NATS

mq9 is compatible with the NATS protocol; all language NATS clients work out of the box, with zero ecosystem cost.

mq9 is not NATS — it is an independent broker. It occupies a middle ground between Core NATS (too lightweight, no persistence) and JetStream (too heavy, stream/consumer/offset) — adding persistence, priority, and TTL, without adding streams and consumer groups. Designed only for Agent communication, not a general messaging system.
