# mq9: What It Is

## One Line

mq9 is a communication protocol layer designed specifically by RobustMQ for AI Agents, sitting alongside MQTT, Kafka, NATS, and AMQP, providing Agents with mailbox and broadcast capabilities.

![img](../../images/69-1.jpg)

---

## What Problem It Solves

Agents need to communicate. But Agents are not services — they are temporary, going online and offline at any time, with a lifecycle that may be only a few seconds.

Today, when Agent A sends a message to Agent B and B is offline, the message is simply lost. Every team is working around this problem with Redis, polling databases, or custom queues.

mq9 solves it directly: **send it out, and the other party receives it naturally when they come online.**

---

## Core Concept: Mailbox

mq9's core concept comes from the real-world mail system.

**Private Mailbox (INBOX)**: I know the other party's address, write the letter, and drop it in. It doesn't matter whether they're home — the letter waits in the mailbox. Messages are divided into urgent, normal, and notify. When they get home they open the mailbox, urgent messages first. If you're worried about missing something, you can go to the post office and check whether there are any uncollected letters.

**Public Broadcast (BROADCAST)**: The community bulletin board — post something and everyone can see it. No need to know who will read it; whoever cares comes themselves. A task posted on the board may have several people wanting to take it, but only one person can do the job.

**Mailbox Expiry**: If nobody uses it for a long time, the post office automatically cancels it and cleans up the letters along with it. No manual destruction needed.

## Four Commands

mq9's complete client protocol:

```text
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # Request mailbox, returns mail_id
  │     └── QUERY.{mail_id}          # Check unread messages (fallback pull)
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # Urgent messages
  │           ├── normal              # Regular messages
  │           └── notify              # Notifications
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # Event broadcast
```

```bash
# Request a mailbox
nats req '$mq9.AI.MAILBOX.CREATE' '{"type":"standard","ttl":3600}'
# → {"mail_id":"m-uuid-001","token":"tok-xxx"}

# Send a message to a mailbox
nats pub '$mq9.AI.INBOX.m-uuid-001.normal' '{"from":"m-uuid-002","payload":"task complete"}'

# Send an urgent message
nats pub '$mq9.AI.INBOX.m-uuid-001.urgent' '{"from":"m-uuid-003","payload":"anomaly alert"}'

# Check unread messages (fallback)
nats req '$mq9.AI.MAILBOX.QUERY.m-uuid-001' '{"token":"tok-xxx"}'

# Broadcast a task
nats pub '$mq9.AI.BROADCAST.task.available' '{"task_id":"t-001"}'

# Competitive consumption
nats sub '$mq9.AI.BROADCAST.task.available' --queue task.workers
```

### Key Design Decisions

**mail_id is not bound to Agent identity.** An Agent can request different mail_ids for different tasks, leave them alone when done, and TTL handles cleanup automatically. A mailbox is a channel-level concept, not an identity-level one.

**Two mailbox types.** `standard`: ordinary mailbox, messages accumulate. `latest`: status mailbox, only the most recent message is kept — used for status reporting and capability declarations.

**Store first, then push.** When a message arrives it is written to storage first, then pushed to online subscribers. If push fails, the message waits in storage and QUERY provides a fallback pull.

**No new concepts.** ACK uses NATS native reply-to, subscriptions use NATS native sub, competitive consumption uses NATS native queue group.

## Why No New SDK Is Needed

mq9 is based on the NATS protocol. NATS clients in any language — Go, Python, Rust, Java, JavaScript — are directly mq9 clients. Nothing new to install, zero ecosystem cost.

The `$mq9.AI.*` namespace design makes the protocol itself into documentation. `$mq9.AI.INBOX.m-001.URGENT` — mq9 protocol, AI scenario, inbox, specific channel, urgent priority. One string, five layers of information.

---

## Storage Tiering

mq9 runs on RobustMQ's unified storage layer. Each message chooses its storage level on demand:

**Memory** — Pure in-memory, the lightest option. Coordination signals; if lost, resend.

**RocksDB** — Temporary persistence. The default choice for mailboxes; auto-cleaned when TTL expires.

**File Segment** — Long-term persistence. Audit logs; retained permanently.

Not every message deserves three replicas, and not every message can afford to be dropped. The choice belongs to the user.

---

## Relationship with RobustMQ

mq9 is RobustMQ's fifth native protocol, sharing the same unified storage architecture as MQTT, Kafka, NATS, and AMQP. Deploy one RobustMQ instance and all of mq9's capabilities are ready. IoT devices send data via MQTT, analytics systems consume via Kafka, Agents collaborate via mq9 — same broker, same storage, zero bridging, zero replication.

---

## Positioning Within the NATS Ecosystem

mq9 sits between Core pub/sub and JetStream. Core NATS is too light — no persistence, messages are lost if the recipient is offline. JetStream is too heavy — streams, consumers, offsets, replay, a full suite of Kafka-equivalent semantics. mq9 adds optional persistence, priority, and TTL auto-management on top of pub/sub, without introducing the heavy concepts of streams, consumer groups, or offsets.

JetStream is a general-purpose persistence layer that tries to cover every scenario. mq9 is designed for Agent communication only. Doing less, but doing it precisely.
