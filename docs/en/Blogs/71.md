# Our Answer: mq9

In the previous post, "The Next Decade of Message Queues: Forget Kafka," I offered a few judgments: append-only log is not the whole of communication; the S3 and data lake narrative is where that framework runs out of road; the underlying layer must be atomic; and the evaluation framework needs to shift from "volume" to "variety."

That post only raised questions; it didn't propose solutions. This one is about our answer.

---

## Where We Started

RobustMQ has been in development for three years. It started by solving the problem of multi-protocol fragmentation — a unified storage layer, with protocols as read/write interfaces: a message is written once, and MQTT, Kafka, NATS, and AMQP each read it according to their own semantics.

As we built it, we noticed a side effect of this architecture: **adding a new protocol costs almost nothing.** The storage layer stays unchanged; a protocol is just an additional semantic parsing layer.

mq9 grew from this foundation as the fifth protocol — designed specifically for AI Agent communication.

---

## Why a Mailbox

In the previous post I said communication takes far richer forms than the append-only log — mailboxes, broadcasts, latest values, request-reply, temporary channels. mq9 chose the two most fundamental: **mailbox and broadcast channel.**

Why? Because Agents are not services. Services are long-running and always reachable; you send a request and they respond. Agents are ephemeral, may live for only a few seconds, and go online and offline at any time. Today Agent A sends a message to Agent B; B is offline, and the message is gone.

Every team building multi-Agent systems is using temporary workarounds — Redis pub/sub, database polling, homegrown task queues. Workable, but all workarounds.

mq9 solves it directly: I send it; you pick it up when you're free; no need to be online simultaneously. Not a message queue, not a stream — it's a mail system.

The mailbox metaphor is not rhetoric; it is the design itself:

- Register a mailbox, get an address → `MAILBOX.CREATE`
- Send mail, regardless of whether the other party is home → `INBOX.{mail_id}.{priority}`
- Receive mail, urgent items first → subscribing delivers all non-expired messages in full
- Reply, the letter contains the other party's address → reply-to
- Unused for a long time, the post office automatically closes it → TTL auto-destroy, no manual deletion
- Community bulletin board, anyone can create it, anyone can post, anyone can read → `BROADCAST.{domain}.{event}`
- A job posted to the board, only one person can take it → queue group
- Permanent building-management board, notifies the whole community who moved in and what skills they have → `BROADCAST.system.*`

## Three Commands

mq9's entire client protocol has just three commands:

| Command | Description |
|---------|-------------|
| `MAILBOX.CREATE` | Create a mailbox or broadcast channel |
| `INBOX` | Point-to-point delivery, persisted, three priority levels |
| `BROADCAST` | Public broadcast, persisted, anyone can publish or subscribe |

No QUERY — every subscription delivers all non-expired messages in full; subscribing is querying; no separate pull interface needed. No read/unread state, no consume offset. Zero consumer state on the server.

No DELETE — mailboxes and broadcast channels are managed entirely by TTL, cleaned up automatically on expiry. Leave it alone when done; create a new one when you need a clean context.

mq9 only handles communication — how to reliably deliver messages. What the message contains and what the business semantics are — mq9 does not care. Message content is a byte array: not parsed, not validated, not restricted.

### Mailboxes and Broadcast Channels

Both share the same underlying behavior — both require CREATE, both are persisted, both have TTL auto-cleanup, both deliver all non-expired messages on subscription. CREATE is idempotent; creating again silently succeeds; TTL is fixed by the first creation.

The only difference is the permission model.

**Mailbox (INBOX)** is private — has mail_id, has token, anyone can send in, only the owner can receive. Use a mailbox when you know who you're addressing.

**Broadcast channel (BROADCAST)** is public — no mail_id, no token, anyone can create it, anyone can publish to it, anyone can subscribe. Use a broadcast when you don't know who you're addressing. A broadcast channel is a public forum, not a private mailbox.

mq9 also has three built-in permanent broadcast channels — system bulletin boards that never expire: `BROADCAST.system.capability` (capability announcements), `BROADCAST.system.status` (status announcements), `BROADCAST.system.channel` (channel announcements). An Agent that comes online and subscribes to `BROADCAST.system.*` can discover the entire network — no registry service needed.

### A Few Key Design Decisions

**mail_id is not bound to Agent identity.** mq9 only knows mail_id, not agent_id. One Agent can apply for different mail_ids for different tasks; leave them alone when done; TTL handles cleanup. Mailboxes are task-level, not identity-level.

**Store first, then push.** Messages are written to storage on arrival, then pushed to online subscribers. Online subscribers take the real-time path; offline messages wait in storage. Persistence is the default behavior, not an option.

**Client-side deduplication.** Every message has a unique msg_id; the server does not track consumer state; clients handle deduplication themselves.

**No state queries.** mq9 does not expose "does this mailbox exist" or "does this channel exist" query interfaces. CREATE always returns success. No directory maintained, no registry role.

## Eight Real Scenarios

Three commands combined — what can they solve?

**Async task result delivery.** Main Agent creates a mailbox; sub-Agent sends results to the main Agent's INBOX when done. No blocking required; pick it up when ready.

**Global status awareness.** Workers create broadcast channels and report status periodically. Main Agent uses wildcard subscriptions to monitor the full picture; new Workers joining are detected automatically; TTL expiry signals death automatically.

**Task broadcast and competing consumption.** Create a broadcast channel to publish tasks; Workers use a queue group to compete; the winner sends results back to the main Agent's INBOX. Offline Workers that come back online can still receive non-expired tasks.

**Anomaly alert broadcast.** Create a broadcast channel to publish anomaly events; subscribers respond on their own. Offline handlers that come back online can still receive non-expired alerts. Publishers don't need to know who is handling it.

**Edge offline message queuing.** Send to an edge Agent's INBOX (urgent/normal priority levels); if the edge is offline, messages are persisted and wait; upon reconnection, they are received by priority — urgent before normal.

**Async request-reply.** A sends a request to B's INBOX with reply_to and correlation_id; B processes it upon coming online and sends the response to A's INBOX. Nothing lost if offline; no blocking.

**Capability registration and discovery.** Agents publish their capabilities to the system bulletin board `BROADCAST.system.capability` at startup; other Agents subscribe and can sense the capability distribution across the entire network. Decentralized, zero extra services.

**Human-Agent hybrid workflows.** Agent sends an approval request to a human approver's INBOX; the approver sends the result back to the Agent's INBOX. Humans and Agents use exactly the same protocol; the workflow is uninterrupted.

## Not Append-Only, It's Pub/Sub

In the previous post I said everything in Kafka flows from the append-only log design decision. mq9 chose a different path: a pub/sub model, based on the NATS protocol.

This choice means mq9 is a completely different thing from the Kafka family. Under mailbox semantics, every assumption of the append-only log becomes unnecessary: messages don't need ordered replay — they need priority-based retrieval; no offset management needed — TTL auto-cleanup instead; no partitions needed — dynamic subjects instead; not every message needs three-replica persistence — choose storage level by scenario.

mq9's position in the NATS ecosystem is the middle ground between Core pub/sub and JetStream. Core NATS is too lightweight — no persistence, offline means lost. JetStream is too heavy — streams, consumers, offsets, replay, a full Kafka-equivalent semantic stack. mq9 adds persistence, priority, and TTL auto-management on top of pub/sub, without introducing streams, consumer groups, or offsets. Just enough, no more.

To be clear: **mq9 is not NATS — it is an independent broker that is compatible with the NATS protocol.** All NATS SDKs in all languages work out of the box, with zero ecosystem cost. The `$mq9.AI.*` namespace design makes the protocol itself documentation — `$mq9.AI.INBOX.{id}.urgent`, one string, five layers of information; see the subject and understand the semantics.

## Atomic Underneath, Composable Above

In the previous post I spent considerable space on one judgment: the underlying layer must be atomic; the upper layer composes and wraps by scenario. mq9 is that judgment in practice.

RobustMQ's storage layer provides three atomic capabilities:

**Memory** — pure in-memory, lightest, no persistence. Coordination signals between Agents; gone once sent; retransmit if lost.

**RocksDB** — temporary persistence. Messages written to disk, waiting for the other party to come online and fetch them; TTL auto-cleanup on expiry. The default choice for mailboxes and broadcast channels.

**File Segment** — long-term persistence. Audit logs, critical events — retained permanently.

A coordination signal takes the memory path; a task instruction takes RocksDB; an audit record takes File Segment. Same system, same API, different costs. Every message chooses on demand — not every message deserves three-replica persistence, and not every message can be dropped.

The "choice matters more than absolute guarantees" I mentioned in the previous post — this is what that looks like in practice.

## Summary

mq9 looks like a message queue, like pub/sub, like JetStream — it resembles something from each. But it is its own thing: each individual capability is not new, but the combination is purpose-built for Agent scenarios, expressed uniformly through mailbox and broadcast channel semantics, and no one has done this before.

Like email is not a database, even though email systems use databases to store mail. mq9 is not a message queue, even though mq9 uses message queue capabilities to implement mailboxes.

**As long as you need to run many Agents, you need mq9.** One command to start, three commands to use, all patterns covered.

In the previous post I said: the more atomic the underlying layer, the freer the upper layer. This post is the practice: three years of RobustMQ's polished unified storage is the atom; mq9 is the communication layer composed from atomic capabilities, custom-built for Agent scenarios.

The questions have been asked. This is our answer. Still on the road, taking it one step at a time.
