# Our Answer: mq9

In the previous post, "The Next Decade of Message Queues: Forget Kafka," I offered a few judgments: append-only log is not the whole of communication; the S3 and data lake narrative is where that framework runs out of road; the underlying layer must be atomic; and the evaluation framework needs to shift from "volume" to "variety."

That post only raised questions; it didn't propose solutions. This one is about our answer.

---

## Where We Started

RobustMQ has been in development for three years. The original problem it set out to solve was the fragmentation caused by multiple protocols — a unified storage layer where protocols are simply read/write interfaces, a message is written once, and MQTT, Kafka, NATS, and AMQP each consume it according to their own semantics.

Along the way, we discovered that this architecture has a side effect: **adding a new protocol costs almost nothing.** The storage layer stays untouched; a protocol is just another layer of semantic parsing on top.

mq9 is the fifth protocol that grew from this foundation — designed specifically for AI Agent communication.

---

## Why a Mailbox

In the previous post I said that communication takes far more forms than append-only log — mailboxes, broadcast, latest-value, request-response, ephemeral channels. mq9 chose the most fundamental of these: **the mailbox.**

Why? Because Agents are not services. Services are always online — you send a request and they respond. Agents are temporary; they may live for only a few seconds, coming and going at any time. Today, when Agent A sends a message to Agent B and B is offline, the message is simply lost.

Every team building multi-Agent systems is working around this problem with temporary solutions — Redis pub/sub, polling databases, custom task queues. They work, but they are all detours.

mq9 solves it directly: I send it out, you pick it up when you're free, no need for both parties to be online at the same time. Not a message queue, not a stream — it's a mail system.

The mailbox metaphor isn't just rhetoric; it is the design itself:

- Register a mailbox, get an address → `MAILBOX.CREATE`
- Mail a letter; it doesn't matter if the other party is home — the letter waits in the mailbox → `INBOX.{mail_id}.{priority}`
- Collect mail; urgent letters first, ordinary ones later → push from server when online
- Worried about missing something, go to the post office and check → `MAILBOX.QUERY`
- Reply; the letter carries the sender's return address → NATS native reply-to
- Nobody uses it for a long time; the post office auto-cancels it → TTL auto-destruction
- Community bulletin board; post something and everyone can see it → `BROADCAST`
- A job posted on the board, but only one person can take it → queue group

## Four Commands

mq9's complete client protocol:

```text
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # Request mailbox, returns mail_id
  │     └── QUERY.{mail_id}          # Check unread messages (fallback pull)
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # Urgent messages
  │           ├── normal              # Regular messages
  │           └── notify              # Notifications
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # Event broadcast
```

Four commands covering eight scenarios. ACK uses NATS native reply-to, subscriptions use NATS native sub, competitive consumption uses NATS native queue group. No new concepts introduced.

A few key design decisions:

**mail_id is not bound to Agent identity.** mq9 recognizes only mail_id, not agent_id. A mail_id is a communication channel — who created it, who uses it, what it's for: mq9 doesn't care. An Agent can request different mail_ids for different tasks, ignore them when done, and TTL handles cleanup automatically.

**Two mailbox types.** Declared at CREATE time: `standard` is an ordinary mailbox where messages accumulate; `latest` is a status mailbox that retains only the most recent message, used for status reporting and capability declarations.

**Mailboxes do not need to be explicitly deleted.** Pure TTL lifecycle management. The simpler the client semantics, the better.

**INBOX is a private mailbox; BROADCAST is public.** If you know who you're reaching, use INBOX — requires CREATE to obtain a mail_id. If you don't know who you're reaching, use BROADCAST — no prerequisites, just publish and subscribe directly.

**Store first, then push.** When a message arrives it is written to the storage layer first, then pushed to online subscribers. Push is the fast path; storage is the safety net. QUERY is the last resort.

## Eight Real-World Scenarios

What can four commands actually solve?

**Async task return.** The primary Agent CREATEs a mailbox; the sub-Agent PUBs the result to the primary Agent's INBOX when done. No blocking required — pick it up when free.

**Global status awareness.** Workers request a type=latest status mailbox and periodically PUB their status. The primary Agent subscribes to get a global view; new Workers joining are detected automatically; TTL expiry signals a Worker's death.

**Task broadcast with competitive consumption.** The primary Agent BROADCASTs tasks; Workers use queue group to compete for them; the Worker that claims a task sends the result to the primary Agent's INBOX.

**Anomaly alert broadcast.** BROADCAST an anomaly event; Agents subscribed to that domain respond automatically. The publisher doesn't need to know who is handling it.

**Edge offline message backlog.** PUB to an edge Agent's INBOX (urgent/normal by priority); messages are persisted while the edge is offline and retrieved by priority after reconnection. QUERY fallback ensures nothing is missed.

**Async request-response.** A PUBs a request to B's INBOX with reply_to and correlation_id; B processes it on coming online and PUBs the response to A's INBOX. Offline messages are not lost.

**Capability registration and discovery.** An Agent PUBs its capability declaration to its own latest mailbox; other Agents subscribe to detect it, or actively query via `BROADCAST.capability.query`. Decentralized, zero additional services.

**Human-machine hybrid workflow.** An Agent PUBs an approval request to a human approver's INBOX; the approver handles it and PUBs the result back to the Agent's INBOX. Humans and Agents use exactly the same protocol; the workflow is uninterrupted.

## Not Append-Only — It's Pub/Sub

In the previous post I said that everything about Kafka flows from the design decision to use an append-only log. mq9 took a different path: a pub/sub model, based on the NATS protocol.

This choice makes mq9 a fundamentally different thing from the Kafka family. Under mailbox semantics, every assumption of an append-only log becomes unnecessary: messages don't need ordered replay — they need priority-based retrieval; no offset management needed — TTL auto-cleanup instead; no partitions needed — dynamic subjects instead; not every message needs three-replica persistence — per-scenario storage levels instead.

mq9's position in the NATS ecosystem is the middle ground between Core pub/sub and JetStream. Core NATS is too light — no persistence, messages are lost when the recipient is offline. JetStream is too heavy — streams, consumers, offsets, replay, a full suite of Kafka-equivalent semantics. mq9 adds optional persistence, priority, and TTL auto-management on top of pub/sub, without introducing the heavy concepts of streams, consumer groups, or offsets. Just enough, nothing more.

Building on NATS also brings an important benefit: **zero ecosystem cost.** All NATS SDKs in all languages work out of the box — no new SDK to build. The `$mq9.AI.*` namespace design makes the protocol itself into documentation — `$mq9.AI.INBOX.{id}.URGENT`, one string, five layers of information; the semantics are clear from the subject alone.

## Atomic Underneath, Composable on Top

In the previous post I spent considerable space making the case that the underlying layer must be atomic, with the upper layer composing and packaging capabilities by scenario. mq9 is the practice of that judgment.

RobustMQ's storage layer provides three atomic capabilities:

**Memory** — Pure in-memory, the lightest option, no persistence. Coordination signals between Agents — sent and gone; if lost, resend.

**RocksDB** — Temporary persistence. Messages are written to disk, waiting for the recipient to come online and retrieve them; TTL expiry triggers automatic cleanup. The default choice for mailboxes.

**File Segment** — Long-term persistence. Audit logs, critical events — retained permanently.

A coordination signal goes through Memory; a task instruction goes through RocksDB; an audit record goes through File Segment. Same system, same API, different costs. Each message chooses its storage on demand — not every message deserves three replicas, and not every message can afford to be dropped.

This is the concrete form of "choice matters more than absolute guarantees" from the previous post.

## Summary

mq9 looks like a message queue, like pub/sub, like JetStream — a bit like all of them. But it is its own thing. Take it apart and none of the individual capabilities are new; put them together, specifically designed for Agent scenarios, unified under mailbox semantics, and nobody has done this before.

Just as email is not a database, even though mail systems use databases to store messages. mq9 is not a message queue, even though mq9 uses message queue capabilities to implement mailboxes.

**If you need to run a lot of Agents, you need mq9.** One command to start, simple to operate, four commands, every pattern covered.

In the previous post I said: the more atomic the underlying layer, the freer the upper layer. This post is the practice: three years of refined unified storage in RobustMQ is the atomic layer; mq9 is the communication layer composed from those atomic capabilities, purpose-built for Agent scenarios.

The question was asked. This is our answer. Still on the road, walking steadily.
