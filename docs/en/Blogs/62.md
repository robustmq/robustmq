# NATS JetStream Protocol Reference

JetStream is NATS's built-in persistence engine. It adds message storage, replay, flow control, and precise delivery semantics on top of Core NATS Pub/Sub. It **introduces no new wire protocol commands** — all operations are completed via standard `PUB` (with reply-to) and `MSG`. The server internally exposes a set of Subjects starting with `$JS.API.`, clients send JSON requests to these Subjects, and the server sends results back to the reply-to address.

## Why JetStream

Core NATS is purely real-time: if there are no online subscribers when a message is sent, the message is lost. JetStream solves the following problems through **Stream (persistent storage) + Consumer (consumption cursor)**:

- Resume consumption from the last checkpoint after a service restart
- Multiple consumers independently replay the same batch of messages
- Guarantee messages are processed at least once (or exactly once)
- Apply backpressure when message volume exceeds consumption speed

## Core Concepts

### Stream

A Stream is a persistent container for messages, bound to a set of Subjects (wildcards supported). Messages published to these Subjects are automatically captured and stored according to the retention policy.

**Storage types:**

| Type | Description |
| ---- | ---- |
| `file` | Persisted to disk; messages survive server restarts |
| `memory` | Stored in memory; higher performance but messages are lost on restart |

**Retention policies:**

| Policy | Description |
| ---- | ---- |
| `limits` | Retain based on count, size, or age limits; supports message replay; default policy |
| `workqueue` | Messages are deleted immediately after acknowledgement; each message is processed by only one consumer |
| `interest` | Messages are only retained while a Consumer is associated; deleted after all Consumers acknowledge |

### Consumer

A Consumer is a consumption cursor on a Stream that records how far consumption has progressed and supports resumption after disconnection. The same Stream can have multiple independent Consumers, each maintaining its own progress.

**Durable vs Ephemeral:**

| Type | Description |
| ---- | ---- |
| Durable | Has a name; cursor is retained after disconnection, resumes from where it left off |
| Ephemeral | No name; server automatically cleans it up when the connection closes |

**Push vs Pull:**

| Type | Description | Use Case |
| ---- | ---- | -------- |
| Push Consumer | Server proactively pushes messages to a specified Subject | Single consumer, ordered replay |
| Pull Consumer | Client actively requests messages, controlling consumption pace | Multi-consumer horizontal scaling, backpressure control |

**Deliver policies:**

| Policy | Description |
| ---- | ---- |
| `all` | Start from the first message in the Stream |
| `new` | Only receive messages published after subscribing |
| `last` | Start from the last message |
| `last_per_subject` | Take the last message per Subject |
| `by_start_sequence` | Start from a specified sequence number |
| `by_start_time` | Start from a specified timestamp |

**ACK policies:**

| Policy | Description |
| ---- | ---- |
| `explicit` | Each message must be explicitly ACKed; unACKed messages are redelivered after timeout |
| `all` | ACKing one message confirms all previous messages as well |
| `none` | No ACK required; messages are immediately considered processed after delivery |

**Replay policies:**

| Policy | Description |
| ---- | ---- |
| `instant` | Replay messages as fast as possible, ignoring original publish timing |
| `original` | Replay at the same intervals as originally published, simulating real traffic |

## Protocol Basics

All JetStream management operations use the standard NATS **Request-Reply** pattern:

```text
Client → PUB $JS.API.<operation> <reply-to> <len>
         <json request body>

Server → MSG <reply-to> <sid> <len>
         <json response body>
```

The client first `SUB _INBOX.<random>` to subscribe to a random address, uses it as the reply-to in the request, then waits for the server to send the response to that address.

## Stream Management

![Stream management interaction diagram](../../images/62-1.jpg)

### Create a Stream

```text
PUB $JS.API.STREAM.CREATE.mystream _INBOX.1 148\r\n
{
  "name": "mystream",
  "subjects": ["orders.>"],
  "storage": "file",
  "retention": "limits",
  "max_msgs": -1,
  "max_bytes": -1,
  "max_age": 0,
  "num_replicas": 1
}\r\n

→ MSG _INBOX.1 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "config": { ... },
  "state": { "messages": 0, "bytes": 0, "first_seq": 0, "last_seq": 0 }
}\r\n
```

### Query / List / Delete / Purge

```text
# Query info
PUB $JS.API.STREAM.INFO.mystream _INBOX.2 0\r\n
\r\n

# List all Streams
PUB $JS.API.STREAM.LIST _INBOX.3 2\r\n
{}\r\n

# Delete a Stream
PUB $JS.API.STREAM.DELETE.mystream _INBOX.4 0\r\n
\r\n
→ MSG _INBOX.4 1 28\r\n
{"success":true}\r\n

# Purge messages (optional filter by subject or sequence)
PUB $JS.API.STREAM.PURGE.mystream _INBOX.5 32\r\n
{"filter": "orders.new", "seq": 100}\r\n
→ MSG _INBOX.5 1 32\r\n
{"success":true,"purged":42}\r\n
```

### Direct Message Fetch (Without Consumer)

`STREAM.MSG.GET` reads a message by sequence number or Subject without updating any consumption progress:

```text
# Fetch by sequence number
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 12\r\n
{"seq": 42}\r\n

→ MSG _INBOX.11 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_msg_get_response",
  "message": {
    "subject": "orders.new",
    "seq": 42,
    "data": "<base64 encoded payload>",
    "time": "2024-01-01T00:00:00Z"
  }
}\r\n

# Fetch the last message for a Subject
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 32\r\n
{"last_by_subj": "orders.new"}\r\n
```

`Direct Get` is a lighter variant — the response returns the raw message content directly (metadata in headers), without JSON wrapping:

```text
# Get the latest message for a Subject
PUB $JS.API.DIRECT.GET.mystream.orders.new _INBOX.12 0\r\n
\r\n

→ HMSG _INBOX.12 1 120 137\r\n
NATS/1.0\r\n
Nats-Stream: mystream\r\n
Nats-Sequence: 42\r\n
Nats-Subject: orders.new\r\n
Nats-Time-Stamp: 2024-01-01T00:00:00Z\r\n
\r\n
<raw message payload>\r\n
```

### Snapshot and Restore

```text
# Create snapshot (server streams data to deliver_subject)
PUB $JS.API.STREAM.SNAPSHOT.mystream _INBOX.13 64\r\n
{"deliver_subject": "snapshot.mystream", "check_msgs": false}\r\n

# Restore snapshot
PUB $JS.API.STREAM.RESTORE.mystream _INBOX.14 0\r\n
{}\r\n
```

## Consumer Management

![Consumer management interaction diagram](../../images/62-2.jpg)

### Create a Consumer

**Push Consumer (Durable):** server proactively pushes messages to `deliver_subject`

```text
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.mycons _INBOX.6 192\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "mycons",
    "deliver_subject": "mycons.delivery",
    "deliver_policy": "all",
    "ack_policy": "explicit",
    "ack_wait": 30000000000,
    "replay_policy": "instant"
  }
}\r\n
```

The client then `SUB mycons.delivery` to receive messages pushed by the server. Each message's reply-to field is the ACK address.

**Pull Consumer:** client fetches on demand, no `deliver_subject` needed

```text
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.pullcons _INBOX.7 96\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "pullcons",
    "deliver_policy": "all",
    "ack_policy": "explicit"
  }
}\r\n
```

### Query / List / Delete

```text
# Query Consumer info
PUB $JS.API.CONSUMER.INFO.mystream.mycons _INBOX.8 0\r\n
\r\n

# List all Consumers
PUB $JS.API.CONSUMER.LIST.mystream _INBOX.9 2\r\n
{}\r\n

# Delete a Consumer
PUB $JS.API.CONSUMER.DELETE.mystream.mycons _INBOX.10 0\r\n
\r\n
→ MSG _INBOX.10 1 16\r\n
{"success":true}\r\n
```

### Pause a Consumer

Pausing stops message delivery until a specified time:

```text
PUB $JS.API.CONSUMER.PAUSE.mystream.mycons _INBOX.11 48\r\n
{"pause_until": "2024-06-01T00:00:00Z"}\r\n

→ MSG _INBOX.11 1 64\r\n
{"paused":true,"pause_until":"2024-06-01T00:00:00Z"}\r\n
```

## Message Consumption

### Pull Consumer Fetch

```text
PUB $JS.API.CONSUMER.MSG.NEXT.mystream.pullcons _INBOX.20 48\r\n
{"batch": 10, "expires": 5000000000}\r\n

→ MSG _INBOX.20 1 $JS.ACK.mystream.pullcons.1.1.1.1700000000.0 64\r\n
<message payload>\r\n

→ MSG _INBOX.20 1 $JS.ACK.mystream.pullcons.1.2.2.1700000001.0 64\r\n
<message payload>\r\n
```

- `batch`: maximum number of messages to pull at once
- `expires`: wait timeout in nanoseconds; when no messages are available the server returns 408

```text
# No messages available, request timed out
→ HMSG _INBOX.20 1 16 16\r\n
NATS/1.0 408\r\n
\r\n
\r\n

# Consumer has expired or been deleted
→ HMSG _INBOX.20 1 16 16\r\n
NATS/1.0 409\r\n
\r\n
\r\n
```

### Publish with ACK (Publish ACK)

A normal `PUB` has no persistence confirmation. Publishing with a reply-to causes the server to reply after the message is written to the Stream:

```text
PUB orders.new _INBOX.21 24\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.21 1 56\r\n
{"stream":"mystream","seq":42,"duplicate":false}\r\n
```

**Deduplication (Exactly-Once Publish):** include `Nats-Msg-Id` in the Header; the server detects duplicates within the configured deduplication window:

```text
HPUB orders.new _INBOX.22 44 68\r\n
NATS/1.0\r\n
Nats-Msg-Id: order-123\r\n
\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.22 1 56\r\n
{"stream":"mystream","seq":42,"duplicate":true}\r\n
```

`duplicate: true` means the message already exists and was not written again.

## ACK Mechanism

![ACK interaction diagram](../../images/62-3.jpg)

When the server delivers a message, the reply-to field is the ACK address in the format:

```text
$JS.ACK.<stream>.<consumer>.<delivered_count>.<stream_seq>.<consumer_seq>.<timestamp>.<pending_msgs>
```

| Field | Description |
| ---- | ---- |
| `delivered_count` | Number of times this message has been delivered (increments on redelivery) |
| `stream_seq` | Global sequence number of the message in the Stream |
| `consumer_seq` | Sequence number of the message within the Consumer |
| `timestamp` | Delivery timestamp (nanoseconds) |
| `pending_msgs` | Number of messages currently pending for this Consumer |

Clients express different ACK semantics by PUBing different payloads to the ACK address:

| Payload | Type | Description |
| ------- | ---- | ---- |
| Empty or `+ACK` | Ack | Message processed successfully |
| `-NAK` | Nak | Reject; redeliver immediately |
| `-NAK {"delay": 5000000000}` | Nak with delay | Reject; redeliver after N nanoseconds |
| `-WPI` | Progress | Processing in progress; reset ack_wait timer to prevent redelivery |
| `-NXT` | Next | Ack and immediately request the next message (Pull Consumer only) |
| `-TERM` | Term | Terminate; no redelivery |

**Examples:**

```text
# Received message (reply-to is the ACK address)
MSG mycons.delivery 1 $JS.ACK.mystream.mycons.1.5.5.1700000000.0 13\r\n
Hello JetStream\r\n

# Acknowledge
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 0\r\n
\r\n

# Reject and redeliver after 5 seconds
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 26\r\n
-NAK {"delay":5000000000}\r\n

# Processing in progress, extend timeout
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 4\r\n
-WPI\r\n

# Terminate, no redelivery
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 5\r\n
-TERM\r\n
```

## Advisory Events

The server publishes advisory events to `$JS.EVENT.ADVISORY.*` when Stream and Consumer state changes occur (fire-and-forget, no reply-to). Clients subscribe to these Subjects to monitor cluster state.

**Event payload format (Stream created example):**

```text
MSG $JS.EVENT.ADVISORY.STREAM.CREATED.mystream 1 192\r\n
{
  "type": "io.nats.jetstream.advisory.v1.stream_action",
  "id": "uuid",
  "timestamp": "2024-01-01T00:00:00Z",
  "stream": "mystream",
  "action": "create",
  "client": { "acc": "default", "user": "admin" }
}\r\n
```

**Complete event list:**

| Event | Subject | Description |
| ----- | ------- | ----------- |
| Stream created | `$JS.EVENT.ADVISORY.STREAM.CREATED.<stream>` | Triggered when a Stream is created |
| Stream deleted | `$JS.EVENT.ADVISORY.STREAM.DELETED.<stream>` | Triggered when a Stream is deleted |
| Stream updated | `$JS.EVENT.ADVISORY.STREAM.UPDATED.<stream>` | Triggered when Stream config changes |
| Stream leader elected | `$JS.EVENT.ADVISORY.STREAM.LEADER_ELECTED.<stream>` | Triggered on leader change in cluster mode |
| Consumer created | `$JS.EVENT.ADVISORY.CONSUMER.CREATED.<stream>.<consumer>` | Triggered when a Consumer is created |
| Consumer deleted | `$JS.EVENT.ADVISORY.CONSUMER.DELETED.<stream>.<consumer>` | Triggered when a Consumer is deleted |
| Consumer leader elected | `$JS.EVENT.ADVISORY.CONSUMER.LEADER_ELECTED.<stream>.<consumer>` | Triggered on Consumer leader change in cluster mode |
| API audit | `$JS.EVENT.ADVISORY.API` | Triggered after every API call; records operation log |
| Snapshot started | `$JS.EVENT.ADVISORY.STREAM.SNAPSHOT.CREATE.<stream>` | Triggered when a snapshot task starts |
| Snapshot completed | `$JS.EVENT.ADVISORY.STREAM.SNAPSHOT.COMPLETE.<stream>` | Triggered when a snapshot task completes |
| Restore started | `$JS.EVENT.ADVISORY.STREAM.RESTORE.CREATE.<stream>` | Triggered when a restore task starts |
| Restore completed | `$JS.EVENT.ADVISORY.STREAM.RESTORE.COMPLETE.<stream>` | Triggered when a restore task completes |

## KV Store

KV Store is a key-value store implemented on JetStream Streams. The underlying Stream is named `KV_<bucket>`. Each write to a key is a message; keys are distinguished by Subject and versions are managed by sequence number.

### KV Bucket Management

Creating and deleting buckets are Stream operations on the underlying `KV_<bucket>` Stream:

```text
# Create a bucket (creates a Stream named KV_config)
PUB $JS.API.STREAM.CREATE.KV_config _INBOX.30 128\r\n
{
  "name": "KV_config",
  "subjects": ["$KV.config.>"],
  "storage": "file",
  "num_replicas": 1,
  "allow_rollup_hdrs": true,
  "deny_delete": true,
  "deny_purge": false,
  "max_msgs_per_subject": 1
}\r\n

# Delete a bucket
PUB $JS.API.STREAM.DELETE.KV_config _INBOX.31 0\r\n
\r\n
```

### Key-Value Operations

**Put:** publish a message to `$KV.<bucket>.<key>`; include a reply-to to receive a write confirmation

```text
PUB $KV.config.db.host _INBOX.32 13\r\n
192.168.1.100\r\n

→ MSG _INBOX.32 1 64\r\n
{"stream":"KV_config","seq":1,"duplicate":false}\r\n
```

**Get:** read the latest value for a key using Direct Get

```text
PUB $JS.API.DIRECT.GET.LAST.KV_config.$KV.config.db.host _INBOX.33 0\r\n
\r\n

→ HMSG _INBOX.33 1 128 141\r\n
NATS/1.0\r\n
Nats-Stream: KV_config\r\n
Nats-Subject: $KV.config.db.host\r\n
Nats-Sequence: 1\r\n
Nats-Time-Stamp: 2024-01-01T00:00:00Z\r\n
KV-Operation: \r\n
\r\n
192.168.1.100\r\n
```

**Delete:** publish a marker message with `KV-Operation: DEL` header

```text
HPUB $KV.config.db.host _INBOX.34 26 26\r\n
NATS/1.0\r\n
KV-Operation: DEL\r\n
\r\n
\r\n
```

**Purge:** publish a marker message with `KV-Operation: PURGE` header to remove all historical revisions of the key

```text
HPUB $KV.config.db.host _INBOX.35 28 28\r\n
NATS/1.0\r\n
KV-Operation: PURGE\r\n
\r\n
\r\n
```

**Watch:** create a Consumer to subscribe to changes on a key or an entire bucket

```text
# Watch a single key
SUB $KV.config.db.host 1\r\n

# Watch all keys in a bucket
SUB $KV.config.> 2\r\n
```

Change notifications are delivered as standard `MSG`. The `KV-Operation` header indicates the operation type (empty = PUT, `DEL` = delete, `PURGE` = purge).

## Object Store

Object Store is an object storage implementation built on JetStream Streams. The underlying Stream is named `OBJ_<bucket>`. It supports storing arbitrarily large binary objects via chunked upload. Each object consists of two parts:

- **Metadata**: stored at `$OBJ.<bucket>.info.<object>` Subject
- **Data chunks**: stored at `$OBJ.<bucket>.chunks.<nonce>` Subject

### Object Bucket Management

```text
# Create a bucket (creates a Stream named OBJ_files)
PUB $JS.API.STREAM.CREATE.OBJ_files _INBOX.40 96\r\n
{
  "name": "OBJ_files",
  "subjects": ["$OBJ.files.>"],
  "storage": "file",
  "num_replicas": 1,
  "max_age": 0
}\r\n

# Delete a bucket
PUB $JS.API.STREAM.DELETE.OBJ_files _INBOX.41 0\r\n
\r\n
```

### Upload an Object (Put)

Upload is a two-step process: send metadata first, then send data in chunks.

Step 1 — send metadata:

```text
HPUB $OBJ.files.info.myfile.txt _INBOX.42 0 192\r\n
NATS/1.0\r\n
\r\n
{
  "name": "myfile.txt",
  "description": "",
  "headers": null,
  "options": null,
  "nonce": "abc123",
  "bucket": "files",
  "chunks": 3,
  "size": 2097152
}\r\n
```

Step 2 — send data chunks:

```text
# Chunk 1
PUB $OBJ.files.chunks.abc123 _INBOX.43 262144\r\n
<chunk data 1>\r\n

# Chunk 2
PUB $OBJ.files.chunks.abc123 _INBOX.43 262144\r\n
<chunk data 2>\r\n

# Last chunk (triggers assembly)
PUB $OBJ.files.chunks.abc123 _INBOX.43 131072\r\n
<chunk data 3>\r\n
```

### Download an Object (Get)

The client sends a Get request; the server streams chunks to the deliver subject:

```text
PUB $OBJ.files _INBOX.44 96\r\n
{
  "name": "myfile.txt",
  "deliver_subject": "obj.deliver.myfile"
}\r\n

# Subscribe to receive chunks
SUB obj.deliver.myfile 5\r\n

# Server pushes chunks
→ MSG obj.deliver.myfile 5 262144\r\n
<chunk data 1>\r\n

→ MSG obj.deliver.myfile 5 262144\r\n
<chunk data 2>\r\n

→ MSG obj.deliver.myfile 5 131072\r\n
<chunk data 3>\r\n
```

### Get Object Metadata (Info)

```text
PUB $OBJ.files _INBOX.45 32\r\n
{"name": "myfile.txt"}\r\n

→ MSG _INBOX.45 1 192\r\n
{
  "name": "myfile.txt",
  "nonce": "abc123",
  "bucket": "files",
  "chunks": 3,
  "size": 2097152,
  "digest": "sha-256=...",
  "deleted": false
}\r\n
```

### Delete an Object

```text
PUB $OBJ.files _INBOX.46 32\r\n
{"name": "myfile.txt"}\r\n

→ MSG _INBOX.46 1 16\r\n
{"success":true}\r\n
```

### Watch for Changes

```text
# Subscribe to all object changes in a bucket (uploads and deletions)
SUB $OBJ.files.info.> 6\r\n
```

When an object is uploaded or deleted, the server publishes updated metadata to `$OBJ.<bucket>.info.<object>`.

## JetStream Domain

When the server is configured with a JetStream Domain, the API Subject prefix changes from `$JS.API` to `$JS.<domain>.API`:

```text
# Default
PUB $JS.API.STREAM.INFO.mystream _INBOX.1 0

# With specified domain
PUB $JS.hub.API.STREAM.INFO.mystream _INBOX.1 0
```

## Error Response Format

All JetStream API operations return a unified JSON error format on failure:

```json
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "error": {
    "code": 400,
    "err_code": 10058,
    "description": "stream name already in use"
  }
}
```

Common error codes:

| err_code | Description |
| -------- | ---- |
| 10039 | Stream does not exist |
| 10014 | Consumer does not exist |
| 10058 | Stream name already in use |
| 10059 | Subject is already bound to another Stream |
| 10071 | Consumer name already exists |

Clients should check whether the `error` field exists in the response JSON to determine whether an operation succeeded.

## Relationship with Core NATS

JetStream introduces no new wire protocol commands; protocol parsers require no modifications:

- Messages published via ordinary `PUB`, if they match a Stream's Subject, are automatically captured and persisted — publishers do not need to be aware of JetStream
- To receive a Publish ACK, simply include a reply-to in the `PUB`
- Messages received by Consumers are standard `MSG` or `HMSG`; the reply-to field is the ACK address
- All management operations are JSON requests sent to `$JS.API.*` Subjects

All the work of implementing JetStream compatibility is at the application layer: recognizing specific Subject prefixes, parsing JSON request bodies, executing the corresponding operations, and serializing results into JSON to reply via `MSG`.
