# NATS JetStream Protocol Reference

JetStream is NATS's built-in persistence engine. It adds message storage, replay, flow control, and precise delivery semantics on top of Core NATS's Pub/Sub. It does not introduce any new wire protocol commands — all operations are completed via standard `PUB` (with reply-to) and `MSG`. The server internally exposes a set of Subjects starting with `$JS.API.`, clients send JSON requests to these Subjects, and the server sends results back to the reply-to address.

## What Problems JetStream Solves

Core NATS is purely real-time: if there are no online subscribers when a message is sent, the message is lost. This is insufficient in many scenarios:

- After a service restart, consumption needs to resume from the last checkpoint
- Multiple consumers need to independently replay the same batch of messages
- Messages need to be guaranteed to be processed at least once (or exactly once)
- Backpressure is needed when message volume exceeds consumption speed

JetStream solves these problems through Stream (persistent storage) + Consumer (consumption cursor), while keeping the NATS protocol layer unchanged.

## Core Concepts

### Stream

A Stream is a persistent container for messages, bound to a set of Subjects (wildcards supported). Messages published to these Subjects are captured and stored by the server, retained for as long as configured by the retention policy.

**Storage types:**

| Type | Description |
| ---- | ---- |
| `file` | Persisted to disk; messages survive server restarts |
| `memory` | Stored in memory; higher performance but messages are lost on restart |

**Retention policies:**

| Policy | Description |
| ---- | ---- |
| `limits` | Retain based on count, size, or age limits; supports message replay; default policy |
| `workqueue` | Messages are deleted immediately after being acknowledged; implements work queue semantics where each message is processed by only one consumer |
| `interest` | Messages are only retained while a Consumer is associated; deleted after all Consumers have acknowledged |

### Consumer

A Consumer is a consumption cursor on a Stream that records how far consumption has progressed and supports resumption after disconnection. The same Stream can have multiple independent Consumers, each maintaining its own progress.

**Durable vs Ephemeral:**

- **Durable**: has a name; the cursor is retained after disconnection and reconnection, resuming from where it left off
- **Ephemeral**: has no name; the server automatically cleans it up when the connection closes

**Push vs Pull:**

| Type | Description | Use Case |
| ---- | ---- | -------- |
| Push Consumer | Server proactively pushes messages to a specified Subject; client SUBs to receive | Single consumer, needs ordered replay |
| Pull Consumer | Client actively requests messages from the server, controlling consumption pace | Multi-consumer horizontal scaling, backpressure control |

**Deliver policies:**

| Policy | Description |
| ---- | ---- |
| `all` | Start from the first message in the Stream |
| `new` | Only receive messages published after subscribing |
| `last` | Start from the last message |
| `last_per_subject` | Take the last message per Subject |
| `by_start_sequence` | Start from a specified sequence number |
| `by_start_time` | Start from a specified timestamp |

**ACK policies:**

| Policy | Description |
| ---- | ---- |
| `explicit` | Each message must be explicitly ACKed; unACKed messages will be redelivered after timeout |
| `all` | ACKing one message confirms all previous messages as well |
| `none` | No ACK required; messages are immediately considered processed after delivery |

**Replay policies:**

| Policy | Description |
| ---- | ---- |
| `instant` | Replay messages as fast as possible, ignoring the original publish timing |
| `original` | Replay messages at the same intervals as they were originally published, simulating real traffic |

## Protocol Interaction Mechanism

All JetStream operations use the standard NATS Request-Reply pattern:

```
Client → PUB $JS.API.<operation> <reply-to> <len>
         <json request body>

Server → MSG <reply-to> <sid> <len>
         <json response body>
```

The client first `SUB _INBOX.<random>`, uses this address as the reply-to in the request, then waits for the server to send the response to this Subject.

Below are complete interaction sequence diagrams for each JetStream operation.

### Stream Management Interaction

![Stream management interaction diagram](../../images/62-1.jpg)

**Create a Stream:**

```
PUB $JS.API.STREAM.CREATE.mystream _INBOX.1 148\r\n
{
  "name": "mystream",
  "subjects": ["orders.>"],
  "storage": "file",
  "retention": "limits",
  "max_msgs": -1,
  "max_bytes": -1,
  "max_age": 0,
  "num_replicas": 1
}\r\n

→ MSG _INBOX.1 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "config": { ... },
  "state": { "messages": 0, "bytes": 0, "first_seq": 0, "last_seq": 0 }
}\r\n
```

**Query Stream info:**

```
PUB $JS.API.STREAM.INFO.mystream _INBOX.2 0\r\n
\r\n
```

**List all Streams:**

```
PUB $JS.API.STREAM.LIST _INBOX.3 2\r\n
{}\r\n
```

**Delete a Stream:**

```
PUB $JS.API.STREAM.DELETE.mystream _INBOX.4 0\r\n
\r\n

→ MSG _INBOX.4 1 28\r\n
{"success":true}\r\n
```

**Purge Stream messages:**

```
PUB $JS.API.STREAM.PURGE.mystream _INBOX.5 0\r\n
\r\n
```

### Consumer Management and Consumption Interaction

![Stream management interaction diagram](../../images/62-2.jpg)

**Create a Push Consumer (Durable):**

```
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.mycons _INBOX.6 192\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "mycons",
    "deliver_subject": "mycons.delivery",
    "deliver_policy": "all",
    "ack_policy": "explicit",
    "ack_wait": 30000000000,
    "replay_policy": "instant"
  }
}\r\n
```

The client then SUBs `mycons.delivery` to receive messages pushed by the server.

**Create a Pull Consumer:**

```
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.pullcons _INBOX.7 128\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "pullcons",
    "deliver_policy": "all",
    "ack_policy": "explicit"
  }
}\r\n
```

**Query Consumer info:**

```
PUB $JS.API.CONSUMER.INFO.mystream.mycons _INBOX.8 0\r\n
\r\n
```

**Delete a Consumer:**

```
PUB $JS.API.CONSUMER.DELETE.mystream.mycons _INBOX.9 0\r\n
\r\n
```

### Message Consumption

**Pull Consumer fetch:**

```
PUB $JS.API.CONSUMER.MSG.NEXT.mystream.pullcons _INBOX.10 64\r\n
{"batch": 10, "expires": 5000000000}\r\n

→ MSG _INBOX.10 1 $JS.ACK.mystream.pullcons.1.1.1.1700000000.0 64\r\n
<message payload>\r\n

→ MSG _INBOX.10 1 $JS.ACK.mystream.pullcons.1.2.2.1700000001.0 64\r\n
<message payload>\r\n
```

`batch` specifies the maximum number of messages to pull at once; `expires` is the wait timeout (nanoseconds). The server sends however many messages are available; if fewer than `batch` are available, it returns when `expires` is reached.

When no messages are available and the timeout expires, the server returns an HMSG with `Status: 408` to indicate request timeout:

```
→ HMSG _INBOX.10 1 16 16\r\n
NATS/1.0 408\r\n
\r\n
\r\n
```

When the Pull Consumer has expired or been deleted, the server returns `Status: 409`.

**Direct message fetch by sequence number (without Consumer):**

```
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 12\r\n
{"seq": 42}\r\n

→ MSG _INBOX.11 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_msg_get_response",
  "message": {
    "subject": "orders.new",
    "seq": 42,
    "data": "<base64 encoded payload>",
    "time": "2024-01-01T00:00:00Z"
  }
}\r\n
```

You can also fetch the last message by Subject:

```
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 32\r\n
{"last_by_subj": "orders.new"}\r\n
```

### ACK Mechanism

![Stream management interaction diagram](../../images/62-3.jpg)

For messages the server pushes to consumers, the reply-to field is the ACK address, in the format:

```
$JS.ACK.<stream>.<consumer>.<delivered_count>.<stream_seq>.<consumer_seq>.<timestamp>.<pending_msgs>
```

Field meanings:

| Field | Description |
| ---- | ---- |
| `delivered_count` | Number of times this message has been delivered (increments on redelivery) |
| `stream_seq` | Global sequence number of the message in the Stream |
| `consumer_seq` | Sequence number of the message within the Consumer |
| `timestamp` | Delivery timestamp (nanoseconds) |
| `pending_msgs` | Number of messages currently pending for this Consumer |

Clients express different ACK semantics by PUBing different payloads to this address:

| Payload | ACK Type | Description |
| ------- | -------- | ---- |
| Empty (0 bytes) or `+ACK` | AckAck | Confirm this message has been processed |
| `-NAK` | AckNak | Reject; redeliver immediately |
| `-NAK {"delay": 5000000000}` | AckNak with delay | Reject; redeliver after the specified nanoseconds |
| `-WPI` | AckProgress | Processing in progress; extend the ack_wait timeout to prevent redelivery |
| `-NXT` | AckNext | Confirm and immediately request the next message (Pull Consumer only) |
| `-TERM` | AckTerm | Terminate; do not redeliver; message is considered handled but no longer needed |

**Examples:**

```
# Received message (reply-to is the ACK address)
MSG mycons.delivery 1 $JS.ACK.mystream.mycons.1.5.5.1700000000.0 13\r\n
Hello JetStream\r\n

# Acknowledge
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 0\r\n
\r\n

# Reject and redeliver with delay
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 26\r\n
-NAK {"delay":5000000000}\r\n

# Processing in progress, extend timeout
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 4\r\n
-WPI\r\n

# Terminate, do not redeliver
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 5\r\n
-TERM\r\n
```

### JetStream Publish (Publishing with ACK)

A normal `PUB` publishes a message without any confirmation. JetStream provides a publish-with-ACK method — publish with a reply-to included, and the server will reply with a Publish ACK confirming the message has been persisted:

```
PUB orders.new _INBOX.20 24\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.20 1 64\r\n
{"stream":"mystream","seq":42,"duplicate":false}\r\n
```

In the response, `seq` is the message's sequence number in the Stream, and `duplicate` indicates whether this is a duplicate message (used with deduplication).

**Deduplication (Exactly Once Publish):**

Include a `Nats-Msg-Id` Header when publishing; the server detects duplicates within the configured deduplication window:

```
HPUB orders.new _INBOX.21 44 68\r\n
NATS/1.0\r\n
Nats-Msg-Id: order-123\r\n
\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.21 1 56\r\n
{"stream":"mystream","seq":42,"duplicate":false}\r\n
```

If the same `Nats-Msg-Id` is published again within the deduplication window, the response will have `duplicate: true` and the message will not be stored again.

## JetStream Domain

When the server is configured with a JetStream Domain, the API Subject prefix changes from `$JS.API` to `$JS.<domain>.API`; all other formats remain the same. For example:

```
# Default
PUB $JS.API.STREAM.INFO.mystream _INBOX.1 0

# With specified domain
PUB $JS.hub.API.STREAM.INFO.mystream _INBOX.1 0
```

## Error Response Format

All JetStream API operations return a unified JSON error format on failure:

```json
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "error": {
    "code": 400,
    "err_code": 10058,
    "description": "stream name already in use"
  }
}
```

Common error codes:

| err_code | Description |
| -------- | ---- |
| 10039 | Stream does not exist |
| 10014 | Consumer does not exist |
| 10058 | Stream name already in use |
| 10059 | Subject is already bound to another Stream |
| 10071 | Consumer name already exists |

Clients should check whether the `error` field exists in the response JSON to determine whether an operation succeeded.

## Relationship with Core NATS

JetStream introduces no new wire protocol commands. Protocol parsers do not need any modifications to handle all JetStream communication — the server recognizes `$JS.API.*` requests, routes them to the internal JetStream processing logic, and sends results back to the client via standard `MSG`.

This means:

- Messages published via ordinary `PUB`, if they match a Stream's Subject, will be automatically captured and persisted — publishers do not need to be aware of JetStream's existence
- To receive a Publish ACK, simply include a reply-to in the `PUB`
- Messages received by Consumers are standard `MSG` or `HMSG`; the reply-to field is the ACK address
- All management operations are JSON requests sent to `$JS.API.*` Subjects

All the work of implementing JetStream compatibility is at the application layer: recognizing `$JS.API.*` Subjects, parsing JSON request bodies, executing the corresponding Stream/Consumer operations, and serializing results into JSON to reply via `MSG`.
