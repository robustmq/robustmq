# mq9 概览

## mq9 是什么

mq9 是 RobustMQ 为 AI Agent 通信设计的第五个原生协议层。

![img](../../images/mq9.jpg)

今天，当 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。每个团队都在用 Redis pub/sub、轮询数据库、自研队列绕过这个问题。能用，但都是绕路。

mq9 直接解决这个问题：**发出去，对方上线自然收到。**

就像人有邮箱——你发一封邮件，对方什么时候看是对方的事，消息不会丢。Agent 之间也需要这种机制。

## 定位

mq9 不是通用消息队列，也不是替代 MQTT 或 Kafka 的方案。它专门针对 **AI Agent 异步通信**这一场景：

- Agent 之间需要通信，但不能保证双方同时在线
- 多 Agent 系统中需要感知其他 Agent 的状态和事件
- 不同任务有不同优先级，紧急指令不能被普通任务淹没

mq9 是 RobustMQ 的第五个原生协议，与 MQTT、Kafka、NATS、AMQP 并列，共享同一套统一存储架构。部署一个 RobustMQ，五个协议全部就位。

## 三个核心原语

### 邮箱（Mailbox）

每个 Agent 有自己的邮箱。发件方不需要知道对方是否在线，直接发到邮箱，消息在那里等着。对方上线后自动收到。

适合场景：子 Agent 完成任务通知主 Agent、云端给边缘设备下发指令、人机混合工作流中的审批请求。

### 广播（Broadcast）

Agent 广播一个事件，不需要知道谁在监听。关心的 Agent 自己来订阅，新 Agent 加入自动感知，拓扑变化零成本。

适合场景：主 Agent 发布任务让 Worker 竞争领取、系统异常事件通知、Agent 上下线状态同步。

### 优先级（Priority）

邮箱支持三个优先级：`urgent`、`normal`、`notify`。紧急指令永远不会被普通任务淹没，同一优先级内保证 FIFO。

- `urgent`：紧急指令，最先处理
- `normal`：常规任务
- `notify`：通知类消息，无需持久化

## 设计原则

**不需要新 SDK**：mq9 基于 NATS 协议构建。任何 NATS 客户端——Go、Python、Rust、Java、JavaScript——都可以直接作为 mq9 客户端使用，无需引入新依赖。

**统一存储**：mq9 消息与 MQTT、Kafka 消息存储在同一套存储引擎上，数据写一次，任意协议消费。

**一个二进制**：不需要单独部署 mq9 服务，RobustMQ 启动即包含 mq9 支持。

## 适合的场景

| 场景 | 说明 |
|------|------|
| 子 Agent 通知主 Agent | 子 Agent 完成任务，发结果到主 Agent 邮箱，主 Agent 不需要阻塞等待 |
| 感知所有子 Agent 状态 | 一行通配符订阅，自动感知所有子 Agent 的上线、运行、消亡 |
| 任务广播竞争消费 | 主 Agent 广播任务，有能力的 Worker 自己来抢，queue group 保证不重复处理 |
| 边缘 Agent 离线积压 | 云端给边缘 Agent 发指令，边缘不在线消息等着，联网后按优先级处理 |
| 人机混合工作流 | Agent 遇到需要人工判断的节点，发审批请求到人类邮箱，审批结果发回后流程继续 |
| 跨系统事件联动 | 一个系统的事件通过广播触发另一个系统的 Agent 响应 |
