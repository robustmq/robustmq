# 集群模式下 QoS 2 的一个边界问题

最近在设计 RobustMQ 集群模式下 QoS 2 的处理逻辑，遇到一个值得认真想的问题，顺便记录一下思考过程。

## 问题从哪里来

MQTT QoS 2 是一个四步握手协议，保证消息恰好送达一次：

```
Client → Broker   PUBLISH
Client ← Broker   PUBREC
Client → Broker   PUBREL
Client ← Broker   PUBCOMP
```

![QoS 2 四步握手流程](../../images/53-01.png)

单节点的情况下这套流程没什么问题，Broker 本地记录会话状态，四步走完清掉就行。

但集群模式下出现了一个边界情况：Client 在收到 PUBREC 之后断连，重连时落到了另一个节点。新节点收到 PUBREL，本地没有任何这个 Client 的 QoS 2 状态记录。

![跨节点断连重连场景](../../images/53-2.png)

这不是一个极端的假设，网络抖动、负载均衡切换、节点重启，都会触发这个场景。

## 已有方案的问题

EMQX 的做法是维护一张全局路由表，记录每个 ClientId 对应的节点地址。任何节点收到请求，查表找到持有会话的节点，转发过去处理。

这个方案在小集群下工作得很好。问题是全局路由表需要实时同步到所有节点——100 万连接就是 100 万条记录，每次 Client 连接或断开都要触发全集群广播。节点数越多，同步开销越高。EMQX 官方建议单集群不超过 20 个节点，这个限制背后的原因之一就在这里。

另一个方向是把 QoS 2 的会话状态写入 Raft，任何节点都能查到，跨节点处理完全没有障碍。代价是每一步握手都要写 Raft，在高并发场景下这个开销不可忽视。

这两个方案都是在用全量的设计去解决一个小概率问题，有点过于沉重。

## 换个角度看这个问题

先想清楚概率分布。

绝大多数情况下，Client 的连接是稳定的，PUBLISH 和 PUBREL 会在同一个节点完成，这是高概率路径。跨节点的情况只在网络抖动或节点切换时出现，是低概率路径。

如果为低概率路径设计一套全量同步的方案，等于在正常路径上持续付出跨节点同步的代价，有点本末倒置。

更合理的思路是：高概率路径做到零开销，低概率路径按需处理，性能取决于实际的跨节点比例，而不是连接总数。

## 我们的方案

基于这个判断，我们选择了本地优先加按需广播发现的策略。

![本地优先加按需广播方案](../../images/53-3.png)

**正常路径**

Client 全程在同一节点，QoS 2 状态在本地完成流转。没有任何跨节点交互，这也是绝大多数请求的路径。

**跨节点路径**

新节点收到 PUBREL，本地查不到对应状态，向集群广播：谁持有这个 Client 的 QoS 2 状态？

持有状态的节点响应后，新节点临时缓存这条映射关系，由原节点完成状态机的剩余流转，新节点转发结果给 Client。

临时缓存只保留最近一条。如果 Client 频繁断连重连，历史映射的命中价值很低，保留下来意义不大。最坏情况是每次都走广播，多一次网络往返，性能可以接受。

**节点宕机**

广播发出去没有响应，说明持有状态的节点已经宕机。

这种情况下回错误码，要求 Client 重新发送 PUBLISH，从头走一遍 QoS 2 握手。

不会伪造 PUBCOMP。QoS 2 的语义承诺是 exactly-once，宁可让 Client 多发一次，也不能假装完成导致消息丢失。对 Client 来说，重传只是多了一次握手的成本，消息不会丢。

## 几个边界情况

**大量 Client 同时断连重连**

广播请求会短时间集中爆发。对广播做限流，异常情况下降速处理是合理的，不需要为极端场景过度设计。

**同一 Client 频繁断连重连**

临时缓存只保留最近一条映射，降低堆积风险。最坏情况是频繁走广播，性能有损耗，但不影响正确性。

## 为什么这样想

这个方案背后有一个基本判断：不追求所有路径都最优，追求整体期望值最优。

把资源集中在高概率路径，低概率路径接受合理的性能降级，但不牺牲语义正确性。正常情况下性能最优，异常情况有明确的处理策略，节点宕机时语义严格保证，没有灰色地带。

这不是一个复杂的方案，但把问题想清楚之后，它是一个不需要解释的方案。
