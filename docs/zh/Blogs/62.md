# NATS JetStream 协议详解

JetStream 是 NATS 内置的持久化引擎，在 Core NATS Pub/Sub 基础上增加了消息存储、重放、流量控制和精确投递语义。它**没有引入任何新的线上协议指令**——所有操作都通过标准的 `PUB`（带 reply-to）和 `MSG` 完成。服务端在内部暴露一组以 `$JS.API.` 开头的 Subject，客户端向这些 Subject 发 JSON 请求，服务端把结果发回 reply-to 地址。

## 为什么需要 JetStream

Core NATS 是纯实时的：消息发出时如果没有在线订阅者，消息就丢了。JetStream 通过 **Stream（持久化存储）+ Consumer（消费游标）** 解决以下问题：

- 服务重启后从上次断点继续消费
- 多个消费者各自独立重放同一批消息
- 保证消息至少被处理一次（或恰好一次）
- 消息量超过消费速度时做背压控制

## 核心概念

### Stream

Stream 是消息的持久化容器，绑定一组 Subject（支持通配符）。发布到这些 Subject 的消息会被自动捕获并按保留策略存储。

**存储类型：**

| 类型 | 说明 |
| ---- | ---- |
| `file` | 持久化到磁盘，服务重启后消息不丢失 |
| `memory` | 存储在内存，性能更高，重启后消息消失 |

**保留策略（Retention）：**

| 策略 | 说明 |
| ---- | ---- |
| `limits` | 按数量、大小、时间上限保留，支持消息重放，默认策略 |
| `workqueue` | 消息确认后立即删除，每条消息只被一个消费者处理 |
| `interest` | 只在有 Consumer 关联时保留，所有 Consumer 确认后删除 |

### Consumer

Consumer 是 Stream 上的消费游标，记录当前消费进度，支持断线续传。同一个 Stream 可以有多个独立 Consumer，各自维护自己的进度。

**Durable vs Ephemeral：**

| 类型 | 说明 |
| ---- | ---- |
| Durable | 有名字，断线重连后游标保留，从上次停止处继续 |
| Ephemeral | 无名字，连接断开后服务端自动清理 |

**Push vs Pull：**

| 类型 | 说明 | 适用场景 |
| ---- | ---- | -------- |
| Push Consumer | 服务端主动推消息到指定 Subject | 单消费者，有序重放 |
| Pull Consumer | 客户端主动请求消息，控制消费节奏 | 多消费者水平扩展，背压控制 |

**投递策略（Deliver Policy）：**

| 策略 | 说明 |
| ---- | ---- |
| `all` | 从 Stream 第一条消息开始 |
| `new` | 只接收订阅之后新发布的消息 |
| `last` | 从最后一条消息开始 |
| `last_per_subject` | 每个 Subject 各取最后一条 |
| `by_start_sequence` | 从指定序号开始 |
| `by_start_time` | 从指定时间点开始 |

**ACK 策略（Ack Policy）：**

| 策略 | 说明 |
| ---- | ---- |
| `explicit` | 每条消息都要显式 ACK，未 ACK 的消息超时重投 |
| `all` | ACK 一条等于确认此前所有消息 |
| `none` | 不需要 ACK，投递后立即视为已处理 |

**Replay 策略（Replay Policy）：**

| 策略 | 说明 |
| ---- | ---- |
| `instant` | 尽可能快地重放，不考虑原始发布的时间间隔 |
| `original` | 按消息原始发布的时间间隔重放，模拟真实流量 |

## 协议交互基础

所有 JetStream 管理操作都是标准的 NATS **Request-Reply** 模式：

```text
Client → PUB $JS.API.<operation> <reply-to> <len>
         <json request body>

Server → MSG <reply-to> <sid> <len>
         <json response body>
```

客户端先 `SUB _INBOX.<random>` 订阅一个随机地址，将其作为 reply-to 发出请求，然后等服务端把响应发到这个地址。

## Stream 管理

![Stream 管理交互图](../../images/62-1.jpg)

### 创建 Stream

```text
PUB $JS.API.STREAM.CREATE.mystream _INBOX.1 148\r\n
{
  "name": "mystream",
  "subjects": ["orders.>"],
  "storage": "file",
  "retention": "limits",
  "max_msgs": -1,
  "max_bytes": -1,
  "max_age": 0,
  "num_replicas": 1
}\r\n

→ MSG _INBOX.1 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "config": { ... },
  "state": { "messages": 0, "bytes": 0, "first_seq": 0, "last_seq": 0 }
}\r\n
```

### 查询 / 列表 / 删除 / 清空

```text
# 查询信息
PUB $JS.API.STREAM.INFO.mystream _INBOX.2 0\r\n
\r\n

# 列出所有 Stream
PUB $JS.API.STREAM.LIST _INBOX.3 2\r\n
{}\r\n

# 删除 Stream
PUB $JS.API.STREAM.DELETE.mystream _INBOX.4 0\r\n
\r\n
→ MSG _INBOX.4 1 28\r\n
{"success":true}\r\n

# 清空消息（Purge），可选按 subject 或 sequence 过滤
PUB $JS.API.STREAM.PURGE.mystream _INBOX.5 32\r\n
{"filter": "orders.new", "seq": 100}\r\n
→ MSG _INBOX.5 1 32\r\n
{"success":true,"purged":42}\r\n
```

### 直接获取消息（不走 Consumer）

`STREAM.MSG.GET` 按序号或 Subject 随机读取消息，不更新任何消费进度：

```text
# 按序号获取
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 12\r\n
{"seq": 42}\r\n

→ MSG _INBOX.11 1 256\r\n
{
  "type": "io.nats.jetstream.api.v1.stream_msg_get_response",
  "message": {
    "subject": "orders.new",
    "seq": 42,
    "data": "<base64 encoded payload>",
    "time": "2024-01-01T00:00:00Z"
  }
}\r\n

# 按 Subject 获取最后一条
PUB $JS.API.STREAM.MSG.GET.mystream _INBOX.11 32\r\n
{"last_by_subj": "orders.new"}\r\n
```

`Direct Get` 是更轻量的版本，响应直接返回消息原始内容（headers 中携带元数据），不包装 JSON：

```text
# 按 Subject 直接获取最新一条
PUB $JS.API.DIRECT.GET.mystream.orders.new _INBOX.12 0\r\n
\r\n

→ HMSG _INBOX.12 1 120 137\r\n
NATS/1.0\r\n
Nats-Stream: mystream\r\n
Nats-Sequence: 42\r\n
Nats-Subject: orders.new\r\n
Nats-Time-Stamp: 2024-01-01T00:00:00Z\r\n
\r\n
<raw message payload>\r\n
```

### 快照与恢复

```text
# 创建快照（服务端流式推送到 deliver_subject）
PUB $JS.API.STREAM.SNAPSHOT.mystream _INBOX.13 64\r\n
{"deliver_subject": "snapshot.mystream", "check_msgs": false}\r\n

# 恢复快照
PUB $JS.API.STREAM.RESTORE.mystream _INBOX.14 0\r\n
{}\r\n
```

## Consumer 管理

![Consumer 管理交互图](../../images/62-2.jpg)

### 创建 Consumer

**Push Consumer（Durable）：** 服务端主动推消息到 `deliver_subject`

```text
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.mycons _INBOX.6 192\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "mycons",
    "deliver_subject": "mycons.delivery",
    "deliver_policy": "all",
    "ack_policy": "explicit",
    "ack_wait": 30000000000,
    "replay_policy": "instant"
  }
}\r\n
```

客户端随后 `SUB mycons.delivery` 接收服务端推送的消息。每条消息的 reply-to 字段就是 ACK 地址。

**Pull Consumer：** 客户端按需拉取，不需要 `deliver_subject`

```text
PUB $JS.API.CONSUMER.DURABLE.CREATE.mystream.pullcons _INBOX.7 96\r\n
{
  "stream_name": "mystream",
  "config": {
    "durable_name": "pullcons",
    "deliver_policy": "all",
    "ack_policy": "explicit"
  }
}\r\n
```

### 查询 / 列表 / 删除

```text
# 查询 Consumer 信息
PUB $JS.API.CONSUMER.INFO.mystream.mycons _INBOX.8 0\r\n
\r\n

# 列出所有 Consumer
PUB $JS.API.CONSUMER.LIST.mystream _INBOX.9 2\r\n
{}\r\n

# 删除 Consumer
PUB $JS.API.CONSUMER.DELETE.mystream.mycons _INBOX.10 0\r\n
\r\n
→ MSG _INBOX.10 1 16\r\n
{"success":true}\r\n
```

### 暂停 Consumer

暂停后 Consumer 停止投递消息直到指定时间：

```text
PUB $JS.API.CONSUMER.PAUSE.mystream.mycons _INBOX.11 48\r\n
{"pause_until": "2024-06-01T00:00:00Z"}\r\n

→ MSG _INBOX.11 1 64\r\n
{"paused":true,"pause_until":"2024-06-01T00:00:00Z"}\r\n
```

## 消息消费

### Pull Consumer 拉取

```text
PUB $JS.API.CONSUMER.MSG.NEXT.mystream.pullcons _INBOX.20 48\r\n
{"batch": 10, "expires": 5000000000}\r\n

→ MSG _INBOX.20 1 $JS.ACK.mystream.pullcons.1.1.1.1700000000.0 64\r\n
<message payload>\r\n

→ MSG _INBOX.20 1 $JS.ACK.mystream.pullcons.1.2.2.1700000001.0 64\r\n
<message payload>\r\n
```

- `batch`：一次最多拉多少条
- `expires`：等待超时（纳秒），超时后无消息可用时服务端返回 408

```text
# 无消息可用，请求超时
→ HMSG _INBOX.20 1 16 16\r\n
NATS/1.0 408\r\n
\r\n
\r\n

# Consumer 已过期或被删除
→ HMSG _INBOX.20 1 16 16\r\n
NATS/1.0 409\r\n
\r\n
\r\n
```

### 带 ACK 的发布（Publish ACK）

普通 `PUB` 没有持久化确认。带 reply-to 发布时，服务端在消息写入 Stream 后回复确认：

```text
PUB orders.new _INBOX.21 24\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.21 1 56\r\n
{"stream":"mystream","seq":42,"duplicate":false}\r\n
```

**去重（Exactly-Once Publish）：** 在 Header 中携带 `Nats-Msg-Id`，服务端在去重窗口内检测重复：

```text
HPUB orders.new _INBOX.22 44 68\r\n
NATS/1.0\r\n
Nats-Msg-Id: order-123\r\n
\r\n
{"item":"widget","qty":5}\r\n

→ MSG _INBOX.22 1 56\r\n
{"stream":"mystream","seq":42,"duplicate":true}\r\n
```

`duplicate: true` 表示消息已存在，未重复写入。

## ACK 机制

![ACK 交互图](../../images/62-3.jpg)

服务端投递消息时，reply-to 字段就是 ACK 地址，格式为：

```text
$JS.ACK.<stream>.<consumer>.<delivered_count>.<stream_seq>.<consumer_seq>.<timestamp>.<pending_msgs>
```

| 字段 | 说明 |
| ---- | ---- |
| `delivered_count` | 该消息被投递的次数（重投时递增） |
| `stream_seq` | 消息在 Stream 中的全局序号 |
| `consumer_seq` | 消息在 Consumer 中的序号 |
| `timestamp` | 投递时间戳（纳秒） |
| `pending_msgs` | 当前 Consumer 待处理的消息数量 |

客户端向 ACK 地址 PUB 不同 payload 表达不同语义：

| Payload | 类型 | 说明 |
| ------- | ---- | ---- |
| 空或 `+ACK` | Ack | 确认消息已处理 |
| `-NAK` | Nak | 拒绝，立即重投 |
| `-NAK {"delay": 5000000000}` | Nak with delay | 拒绝，延迟 N 纳秒后重投 |
| `-WPI` | Progress | 处理中，重置 ack_wait 计时器，防止超时重投 |
| `-NXT` | Next | 确认并立即请求下一条（Pull Consumer 专用） |
| `-TERM` | Term | 终止，不再重投 |

**示例：**

```text
# 收到消息（reply-to 即 ACK 地址）
MSG mycons.delivery 1 $JS.ACK.mystream.mycons.1.5.5.1700000000.0 13\r\n
Hello JetStream\r\n

# 确认
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 0\r\n
\r\n

# 拒绝并延迟 5 秒后重投
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 26\r\n
-NAK {"delay":5000000000}\r\n

# 处理中，延长超时
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 4\r\n
-WPI\r\n

# 终止，不再重投
PUB $JS.ACK.mystream.mycons.1.5.5.1700000000.0 5\r\n
-TERM\r\n
```

## Advisory 事件

服务端在 Stream 和 Consumer 发生状态变化时，向 `$JS.EVENT.ADVISORY.*` 发布事件通知（fire-and-forget，无 reply-to）。客户端通过 `SUB` 订阅这些 Subject 来监控集群状态。

**事件 payload 格式（以 Stream 创建为例）：**

```text
MSG $JS.EVENT.ADVISORY.STREAM.CREATED.mystream 1 192\r\n
{
  "type": "io.nats.jetstream.advisory.v1.stream_action",
  "id": "uuid",
  "timestamp": "2024-01-01T00:00:00Z",
  "stream": "mystream",
  "action": "create",
  "client": { "acc": "default", "user": "admin" }
}\r\n
```

**完整事件列表：**

| 事件 | Subject | 说明 |
| ---- | ------- | ---- |
| Stream 创建 | `$JS.EVENT.ADVISORY.STREAM.CREATED.<stream>` | Stream 被创建时触发 |
| Stream 删除 | `$JS.EVENT.ADVISORY.STREAM.DELETED.<stream>` | Stream 被删除时触发 |
| Stream 更新 | `$JS.EVENT.ADVISORY.STREAM.UPDATED.<stream>` | Stream 配置变更时触发 |
| Stream Leader 选举 | `$JS.EVENT.ADVISORY.STREAM.LEADER_ELECTED.<stream>` | 集群模式下 Leader 变更时触发 |
| Consumer 创建 | `$JS.EVENT.ADVISORY.CONSUMER.CREATED.<stream>.<consumer>` | Consumer 被创建时触发 |
| Consumer 删除 | `$JS.EVENT.ADVISORY.CONSUMER.DELETED.<stream>.<consumer>` | Consumer 被删除时触发 |
| Consumer Leader 选举 | `$JS.EVENT.ADVISORY.CONSUMER.LEADER_ELECTED.<stream>.<consumer>` | 集群模式下 Consumer Leader 变更时触发 |
| API 审计 | `$JS.EVENT.ADVISORY.API` | 每次 API 调用后触发，记录操作日志 |
| 快照开始 | `$JS.EVENT.ADVISORY.STREAM.SNAPSHOT.CREATE.<stream>` | 快照任务开始时触发 |
| 快照完成 | `$JS.EVENT.ADVISORY.STREAM.SNAPSHOT.COMPLETE.<stream>` | 快照任务完成时触发 |
| 恢复开始 | `$JS.EVENT.ADVISORY.STREAM.RESTORE.CREATE.<stream>` | 恢复任务开始时触发 |
| 恢复完成 | `$JS.EVENT.ADVISORY.STREAM.RESTORE.COMPLETE.<stream>` | 恢复任务完成时触发 |

## KV Store

KV Store 是基于 JetStream Stream 实现的键值存储，底层是一个名为 `KV_<bucket>` 的 Stream，每个 key 的每次写入都是一条消息，通过 Subject 区分 key，通过序号实现版本控制。

### Bucket 管理

Bucket 的创建和删除本质上是对底层 Stream 的操作：

```text
# 创建 Bucket（创建名为 KV_config 的 Stream）
PUB $JS.API.STREAM.CREATE.KV_config _INBOX.30 128\r\n
{
  "name": "KV_config",
  "subjects": ["$KV.config.>"],
  "storage": "file",
  "num_replicas": 1,
  "allow_rollup_hdrs": true,
  "deny_delete": true,
  "deny_purge": false,
  "max_msgs_per_subject": 1
}\r\n

# 删除 Bucket
PUB $JS.API.STREAM.DELETE.KV_config _INBOX.31 0\r\n
\r\n
```

### 键值操作

**写入（Put）：** 向 `$KV.<bucket>.<key>` 发布消息，带 reply-to 可获取写入确认

```text
PUB $KV.config.db.host _INBOX.32 13\r\n
192.168.1.100\r\n

→ MSG _INBOX.32 1 64\r\n
{"stream":"KV_config","seq":1,"duplicate":false}\r\n
```

**读取（Get）：** 通过 Direct Get 读取指定 key 的最新值

```text
PUB $JS.API.DIRECT.GET.LAST.KV_config.$KV.config.db.host _INBOX.33 0\r\n
\r\n

→ HMSG _INBOX.33 1 128 141\r\n
NATS/1.0\r\n
Nats-Stream: KV_config\r\n
Nats-Subject: $KV.config.db.host\r\n
Nats-Sequence: 1\r\n
Nats-Time-Stamp: 2024-01-01T00:00:00Z\r\n
KV-Operation: \r\n
\r\n
192.168.1.100\r\n
```

**删除（Delete）：** 写入一条带 `KV-Operation: DEL` header 的标记消息

```text
HPUB $KV.config.db.host _INBOX.34 26 26\r\n
NATS/1.0\r\n
KV-Operation: DEL\r\n
\r\n
\r\n
```

**清除所有版本（Purge）：** 写入带 `KV-Operation: PURGE` 的标记消息，删除该 key 的所有历史版本

```text
HPUB $KV.config.db.host _INBOX.35 28 28\r\n
NATS/1.0\r\n
KV-Operation: PURGE\r\n
\r\n
\r\n
```

**监听变更（Watch）：** 通过创建 Consumer 订阅 key 或整个 bucket 的变更

```text
# 订阅单个 key 的变更
SUB $KV.config.db.host 1\r\n

# 订阅整个 bucket 所有 key 的变更
SUB $KV.config.> 2\r\n
```

变更通知以普通 `MSG` 形式推送，`KV-Operation` header 标识操作类型（空 = PUT，`DEL` = 删除，`PURGE` = 清除）。

## Object Store

Object Store 是基于 JetStream Stream 实现的对象存储，底层是名为 `OBJ_<bucket>` 的 Stream，支持存储任意大小的二进制对象（通过分块上传）。每个对象由两部分组成：

- **元数据**：存储在 `$OBJ.<bucket>.info.<object>` Subject
- **数据分块**：存储在 `$OBJ.<bucket>.chunks.<nonce>` Subject

### Object Bucket 管理

```text
# 创建 Bucket（创建名为 OBJ_files 的 Stream）
PUB $JS.API.STREAM.CREATE.OBJ_files _INBOX.40 96\r\n
{
  "name": "OBJ_files",
  "subjects": ["$OBJ.files.>"],
  "storage": "file",
  "num_replicas": 1,
  "max_age": 0
}\r\n

# 删除 Bucket
PUB $JS.API.STREAM.DELETE.OBJ_files _INBOX.41 0\r\n
\r\n
```

### 上传对象（Put）

上传分为两个步骤：先发送元数据，再分块发送数据。

第一步，发送元数据：

```text
HPUB $OBJ.files.info.myfile.txt _INBOX.42 0 192\r\n
NATS/1.0\r\n
\r\n
{
  "name": "myfile.txt",
  "description": "",
  "headers": null,
  "options": null,
  "nonce": "abc123",
  "bucket": "files",
  "chunks": 3,
  "size": 2097152
}\r\n
```

第二步，分块发送数据：

```text
# 第 1 块
PUB $OBJ.files.chunks.abc123 _INBOX.43 262144\r\n
<chunk data 1>\r\n

# 第 2 块
PUB $OBJ.files.chunks.abc123 _INBOX.43 262144\r\n
<chunk data 2>\r\n

# 最后一块（触发组装完成）
PUB $OBJ.files.chunks.abc123 _INBOX.43 131072\r\n
<chunk data 3>\r\n
```

### 下载对象（Get）

客户端发送 Get 请求，服务端流式推送分块数据到 deliver subject：

```text
PUB $OBJ.files _INBOX.44 96\r\n
{
  "name": "myfile.txt",
  "deliver_subject": "obj.deliver.myfile"
}\r\n

# 订阅接收数据
SUB obj.deliver.myfile 5\r\n

# 服务端推送分块
→ MSG obj.deliver.myfile 5 262144\r\n
<chunk data 1>\r\n

→ MSG obj.deliver.myfile 5 262144\r\n
<chunk data 2>\r\n

→ MSG obj.deliver.myfile 5 131072\r\n
<chunk data 3>\r\n
```

### 获取对象元数据（Info）

```text
PUB $OBJ.files _INBOX.45 32\r\n
{"name": "myfile.txt"}\r\n

→ MSG _INBOX.45 1 192\r\n
{
  "name": "myfile.txt",
  "nonce": "abc123",
  "bucket": "files",
  "chunks": 3,
  "size": 2097152,
  "digest": "sha-256=...",
  "deleted": false
}\r\n
```

### 删除对象（Delete）

```text
PUB $OBJ.files _INBOX.46 32\r\n
{"name": "myfile.txt"}\r\n

→ MSG _INBOX.46 1 16\r\n
{"success":true}\r\n
```

### 监听变更（Watch）

```text
# 订阅 bucket 内所有对象的变更（上传/删除）
SUB $OBJ.files.info.> 6\r\n
```

每次对象上传或删除时，服务端会向 `$OBJ.<bucket>.info.<object>` 发布更新后的元数据。

## JetStream Domain

服务端配置了 JetStream Domain 后，API Subject 前缀从 `$JS.API` 变为 `$JS.<domain>.API`：

```text
# 默认
PUB $JS.API.STREAM.INFO.mystream _INBOX.1 0

# 指定 domain
PUB $JS.hub.API.STREAM.INFO.mystream _INBOX.1 0
```

## 错误响应格式

所有 JetStream API 在出错时返回统一的 JSON 格式：

```json
{
  "type": "io.nats.jetstream.api.v1.stream_create_response",
  "error": {
    "code": 400,
    "err_code": 10058,
    "description": "stream name already in use"
  }
}
```

常见错误码：

| err_code | 说明 |
| -------- | ---- |
| 10039 | Stream 不存在 |
| 10014 | Consumer 不存在 |
| 10058 | Stream 名称已存在 |
| 10059 | Subject 已被其他 Stream 绑定 |
| 10071 | Consumer 名称已存在 |

客户端应检查响应 JSON 中是否存在 `error` 字段来判断操作是否成功。

## 与 Core NATS 的关系

JetStream 不引入任何新的线上协议命令，协议解析器无需任何修改：

- 普通 `PUB` 发出的消息，如果匹配了某个 Stream 的 Subject，会被自动捕获持久化，发布者不感知 JetStream 的存在
- 想要发布确认（Publish ACK），只需在 `PUB` 时带上 reply-to
- Consumer 收到的消息是标准的 `MSG` 或 `HMSG`，reply-to 字段是 ACK 地址
- 所有管理操作都是向 `$JS.API.*` Subject 发 JSON 请求

实现 JetStream 兼容的工作全部在业务层：识别特定 Subject 前缀、解析 JSON 请求体、执行对应操作、序列化结果通过 `MSG` 回复。
