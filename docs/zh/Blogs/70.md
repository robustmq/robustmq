# mq9：是什么

## 一句话

mq9 是 RobustMQ 专门为 AI Agent 设计的通信协议层，与 MQTT、Kafka、NATS、AMQP 并列，给 Agent 提供邮箱和广播能力。

![img](../../images/69-1.jpg)

---

## 解决什么问题

Agent 需要通信。但 Agent 不是服务——它是临时的，随时上下线，生命周期可能只有几秒。

今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。每个团队都在用 Redis、轮询数据库、自研队列绕过这个问题。

mq9 直接解决它：**发出去，对方上线自然收到。**

---

## 核心概念：邮箱

mq9 的核心概念来自现实世界的邮件系统。

**私人邮箱（INBOX）**：我知道对方地址，写好信投进去。对方在不在家不管，信在邮箱等着。分急件、普通件、广告。对方回家打开邮箱，急件先看。怕有遗漏，可以主动去邮局查一下还有没有信没取。

**公共广播（BROADCAST）**：小区公告栏，贴上去所有人能看。不需要知道谁会看，想看的自己来。公告栏贴了一个任务，好几个人都想接，但只能一个人干。

**邮箱过期**：长期没人用，邮局自动注销，信跟着一起清理。不需要手动销毁。

## 四个命令字

mq9 的全部客户端协议：

```
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # 申请邮箱，返回 mail_id
  │     └── QUERY.{mail_id}          # 查未读消息（兜底拉取）
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

```bash
# 申请邮箱
nats req '$mq9.AI.MAILBOX.CREATE' '{"type":"standard","ttl":3600}'
# → {"mail_id":"m-uuid-001","token":"tok-xxx"}

# 发消息到邮箱
nats pub '$mq9.AI.INBOX.m-uuid-001.normal' '{"from":"m-uuid-002","payload":"任务完成"}'

# 发紧急消息
nats pub '$mq9.AI.INBOX.m-uuid-001.urgent' '{"from":"m-uuid-003","payload":"异常告警"}'

# 查未读消息（兜底）
nats req '$mq9.AI.MAILBOX.QUERY.m-uuid-001' '{"token":"tok-xxx"}'

# 广播任务
nats pub '$mq9.AI.BROADCAST.task.available' '{"task_id":"t-001"}'

# 竞争消费
nats sub '$mq9.AI.BROADCAST.task.available' --queue task.workers
```

### 关键设计

**mail_id 不绑定 Agent 身份。** 一个 Agent 可以为不同任务申请不同的 mail_id，用完不管，TTL 自动清理。邮箱是通道级的，不是身份级的。

**两种邮箱类型。** `standard`：普通邮箱，消息累积。`latest`：状态邮箱，只保留最新一条，用于状态上报和能力声明。

**先存储后推送。** 消息到达先写存储，再推给在线订阅者。推送失败消息在存储等着，QUERY 兜底拉取。

**不创建新概念。** ACK 用 NATS 原生 reply-to，订阅用 NATS 原生 sub，竞争消费用 NATS 原生 queue group。

---

## 为什么不需要新 SDK

mq9 基于 NATS 协议。所有语言的 NATS 客户端——Go、Python、Rust、Java、JavaScript——直接就是 mq9 的客户端。不需要安装任何新东西，零生态成本。

`$mq9.AI.*` 的命名空间设计让协议本身成为文档。`$mq9.AI.INBOX.m-001.URGENT` ——mq9 协议，AI 场景，收件箱，指定通道，紧急优先级。一行字符串，五层信息。

---

## 存储分级

mq9 运行在 RobustMQ 统一存储层之上，每条消息按需选择存储级别：

**Memory** — 纯内存，最轻。协调信号，丢了重发。

**RocksDB** — 临时持久化。邮箱默认选择，TTL 到期自动清理。

**File Segment** — 长期持久化。审计日志，永久保留。

不是所有消息都值得三副本，也不是所有消息都可以丢。选择权交给用户。

---

## 与 RobustMQ 的关系

mq9 是 RobustMQ 的第五个原生协议，与 MQTT、Kafka、NATS、AMQP 共享同一套统一存储架构。部署一个 RobustMQ，mq9 的能力全部就位。IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储，零桥接，零复制。

---

## 在 NATS 生态中的定位

mq9 介于 Core pub/sub 和 JetStream 之间。Core NATS 太轻——没有持久化，离线就丢。JetStream 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义。mq9 在 pub/sub 的基础上加了可选持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。

JetStream 是通用持久化层，什么场景都想覆盖。mq9 只为 Agent 通信这一个场景设计。做的少，反而做得准。
