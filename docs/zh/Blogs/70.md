# mq9：是什么

## 一句话

mq9 是 RobustMQ 专门为 AI Agent 设计的通信层。给 Agent 提供邮箱和广播频道——发出去，对方上线自然收到。

---

## 解决什么问题

Agent 需要通信。但 Agent 不是服务——它是临时的，随时上下线，生命周期可能只有几秒。

今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。每个团队都在用 HTTP callback、Redis、轮询数据库、自研队列绕过这个问题。

mq9 直接解决它：**消息到达先写存储，对方在线实时推送，不在线消息等着，上线后全量收到。**

mq9 只管通信——消息怎么可靠送达。消息里装什么、业务语义是什么，mq9 不管。

---

## 两个核心概念

### 邮箱（INBOX）

我知道对方地址，写好信投进去。对方在不在家不管，信在邮箱等着。分急件、普通件、广告。对方回家打开邮箱，急件先看。

邮箱是私人的——别人往里发，只有主人能收。创建时声明 TTL，到期自动销毁，不需要手动删除。

### 广播频道（BROADCAST）

小区公告栏。任何人可以创建一个，创建后任何人都能往里贴公告，任何人都能来看。不需要知道谁会看，想看的自己来。公告栏贴了一个任务，好几个人都想接，只能一个人干。

广播频道也持久化、也有 TTL。离线的人上线后也能看到未过期的公告。

### 共同点

都需要创建，同一个 CREATE 命令。都持久化，都有 TTL 自动清理。每次订阅全量推送所有未过期消息，不追踪已读/未读。CREATE 是幂等的——重复创建不报错，TTL 以第一次为准。

区别只有一个：邮箱是私人的（有 mail_id、有 token、只有主人能收），广播频道是公共的（无 mail_id、无 token、任何人可发可收）。

### 系统公告板

mq9 内置三个永久广播频道，系统自动创建，永不过期：

- `BROADCAST.system.capability` — 能力公告：我能做什么
- `BROADCAST.system.status` — 状态公告：我还活着
- `BROADCAST.system.channel` — 频道公告：我建了一个新频道

Agent 上线订阅 `BROADCAST.system.*`，就能发现整个网络。不需要注册中心。

---

## 三个命令字

| 命令字 | 说明 |
|--------|------|
| `MAILBOX.CREATE` | 创建邮箱或广播频道 |
| `INBOX` | 点对点投递，持久化，三级优先级（urgent / normal / notify） |
| `BROADCAST` | 公共广播，持久化，任何人可发可订 |

没有 QUERY——订阅本身就是查询。没有 DELETE——TTL 自动清理。

协议详细设计见 [$mq9.AI.* 协议设计文档]。

---

## 关键设计

- **mail_id 不绑定 Agent 身份。** 一个 Agent 多个邮箱，用完不管，TTL 自动清理。
- **两种类型。** `standard`：消息累积。`latest`：只保留最新一条，用于状态上报。
- **先存储后推送。** 持久化是默认行为，不是可选项。
- **客户端去重。** 每条消息有 msg_id，服务端不追踪消费状态。
- **CREATE 幂等。** 重复创建静默成功，TTL 以第一次为准。不提供状态查询。

---

## 与 RobustMQ 的关系

mq9 是 RobustMQ 的第五个原生协议层，与 MQTT、Kafka、NATS、AMQP 并列，共享统一存储。部署一个 RobustMQ，mq9 的能力全部就位。

```bash
docker run -d -p 4222:4222 robustmq/robustmq:latest
```

IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储。

---

## 与 NATS 的关系

mq9 兼容 NATS 协议，所有语言的 NATS 客户端直接可用，零生态成本。

mq9 不是 NATS，是一个独立的 broker。定位介于 Core NATS（太轻，没有持久化）和 JetStream（太重，stream/consumer/offset）之间——加了持久化、优先级、TTL，不加 stream 和 consumer group。只为 Agent 通信设计，不做通用消息系统。
