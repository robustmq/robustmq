# mq9：是什么

## 一句话

mq9 是 RobustMQ 专门为 AI Agent 设计的通信层。给 Agent 提供邮箱——发出去，对方上线自然收到。

---

## 解决什么问题

Agent 需要通信。但 Agent 不是服务——它是临时的，随时上下线，生命周期可能只有几秒。

今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。每个团队都在用 HTTP callback、Redis、轮询数据库、自研队列绕过这个问题。

mq9 直接解决它：**消息到达先写存储，对方在线实时推送，不在线消息等着，上线后全量收到。**

mq9 只管通信——消息怎么可靠送达。消息里装什么、业务语义是什么，mq9 不管。

---

## 核心概念：邮箱

mq9 只有一个核心抽象：**邮箱（MAILBOX）**。

我知道对方地址，写好信投进去。对方在不在家不管，信在邮箱等着。分急件、普通件、低优先级。对方回家打开邮箱，急件先看。

邮箱有 TTL，到期自动销毁，消息随之清理。不需要手动删除。一个 Agent 可以同时持有多个邮箱——用完不管，TTL 自动清理。

**mail_id 不可猜测即安全边界。** mail_id 是系统生成的不可猜测字符串。知道 mail_id 就能发消息、能订阅，不知道就无从操作。没有 token，没有 ACL。

邮箱分两种：

| | 私有邮箱 | 公开邮箱 |
|---|---|---|
| mail_id | 系统生成，不可猜测 | 用户自定义，有意义的名字 |
| 可发现性 | 不公开 | 自动注册到 PUBLIC.LIST |
| 适合场景 | 点对点通信、结果回传 | 任务队列、公共频道 |

公开邮箱的 mail_id 就是地址，起一个有意义的名字——`task.queue`、`analytics.result`——比 UUID 更容易被其他 Agent 发现和理解。

**订阅语义：** 每次订阅全量推送所有未过期消息，后续新消息实时推送。不追踪已读/未读，没有消费位点，没有 QUERY 命令。

**CREATE 幂等：** 重复创建不报错，TTL 以第一次为准。

---

## 三件事

| 操作 | 说明 |
|------|------|
| `MAILBOX.CREATE` | 创建邮箱 |
| `MAILBOX.{mail_id}.{priority}` | 发消息到指定邮箱 |
| `MAILBOX.{mail_id}.*` | 订阅邮箱，全量接收所有未过期消息 |

没有 QUERY——订阅本身就是查询。没有 DELETE——TTL 自动清理。

**三个优先级：**

| 级别 | 语义 | 典型场景 |
|------|------|---------|
| high | 紧急，优先处理 | 任务中断、紧急指令 |
| normal | 常规通信 | 任务分发、结果返回 |
| low | 后台，不紧急 | 日志、状态上报 |

存储层保证优先级顺序，消费方无需自行排序。

协议详细设计见 [$mq9.AI.* 协议设计文档]。

---

## 发现公开邮箱

`$mq9.AI.PUBLIC.LIST` 是系统内置地址，broker 维护，不接受用户写入，TTL 永不过期。

`public: true` 的邮箱创建后自动注册，TTL 到期自动移除。订阅即全量推送——当前所有公开邮箱立即推送，后续增删实时推送。

```bash
nats sub '$mq9.AI.PUBLIC.LIST'
```

不需要注册中心，PUBLIC.LIST 本身就是目录。

---

## 关键设计

- **先存储后推送。** 持久化是默认行为，不是可选项。
- **mail_id 不绑定 Agent 身份。** 一个 Agent 多个邮箱，用完不管，TTL 自动清理。
- **客户端去重。** 每条消息有 msg_id，服务端不追踪消费状态。
- **CREATE 幂等。** 重复创建静默成功，TTL 以第一次为准，不提供状态查询。
- **禁止通配符订阅所有邮箱。** 订阅必须精确到 mail_id，broker 层强制拒绝。

---

## 与 RobustMQ 的关系

mq9 是 RobustMQ 的第五个原生协议层，与 MQTT、Kafka、NATS、AMQP 并列，共享统一存储。部署一个 RobustMQ，mq9 的能力全部就位。

```bash
docker run -d -p 4222:4222 robustmq/robustmq:latest
```

IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储。

---

## 与 NATS 的关系

mq9 兼容 NATS 协议，所有语言的 NATS 客户端直接可用，零生态成本。

mq9 不是 NATS，是一个独立的 broker。定位介于 Core NATS（太轻，没有持久化）和 JetStream（太重，stream/consumer/offset）之间——加了持久化、优先级、TTL，不加 stream 和 consumer group。只为 Agent 通信设计，不做通用消息系统。
