# mq9 承载 A2A：一些设计思考

## 一些前置的思考

在动手设计之前，我先想清楚了几个问题。

**第一个问题：mq9 到底是什么。**

最早我把 mq9 想象成"Agent 异步通信的协议标准"，方向太大。后来收敛到一个更具体的定位——**企业内部的私有化的 Agent 之间的通信管道**。一句话总结：部署 mq9，公司内的所有 Agent 就能自由注册和通信，包括实时和异步。

这个定位有几个好处。市场清晰——目标客户是正在做 Agent 落地的中大型企业。痛点真实——这些企业有 5+ 个 Agent 应用，多个团队各用不同框架（LangGraph、LangChain、AutoGen、CrewAI），缺一个统一的通信基础设施。商业模式可行——开源核心 + 企业版高级功能，类似 Kafka 之于 Confluent。

**第二个问题：mq9 和 A2A 是什么关系。**

讨论了几轮才想清楚——它们解决不同层的问题，不冲突。A2A 是协议内容层，定义 Agent 之间交换什么信息（Task、Message、Artifact、状态机）。mq9 是传输层，定义这些信息怎么从一个 Agent 到另一个 Agent。两者正交。

具体而言，用户可以有三种用法：

1. 完全用 mq9——注册、发现、通信全在 mq9 里
2. mq9 做发现 + mq9 做 A2A 异步传输
3. 外部服务发现 + mq9 做 A2A 异步传输

mq9 协议层不需要为这三个场景做分支，只要 mailbox 抽象做对，三种场景都能支持。

**第三个问题：mq9 应该做多大。**

控制 scope 是基础设施项目最重要的纪律。mq9 做什么、不做什么要早早划清楚。

mq9 做：mailbox 抽象、异步通信、简单的 Agent 注册发现。

mq9 不做：协议层抽象（不是 A2A 替代）、能力描述 schema（让 A2A、MCP、应用层定义）、复杂服务发现（不和 etcd/Consul 竞争）、Agent 编排（不和 LangGraph/AutoGen 竞争）、自动路由和意图解析（那是上层的事）。

边界清晰带来的好处是反脆弱——任何一种上层方案赢了，mq9 都能用上；任何一种发现方案赢了，mq9 都能配合。mq9 只关心"消息怎么从一个 mailbox 到另一个 mailbox"，这件事不会过时。

下面进入具体设计。

## 核心抽象：mailbox

mq9 的核心抽象只有一个——mailbox。

mailbox 是轻量的、可丢弃的通信资源。Agent 想通信就申请 mailbox，用完就废弃，不需要管理 session 生命周期。这是 unix 的"everything is a file"哲学的 mq9 版本——everything is a mailbox。

**mailbox 的几个特征：**

- **轻量**：创建几乎无成本，单个 broker 推荐支持 100 万级 mailbox 同时活跃。
- **可丢弃**：每个 mailbox 有 TTL，到期自动清理。Agent 不需要管理生命周期。
- **持久化**：消息默认持久化保存，订阅者上下线自动获取存量消息。
- **不透明**：mq9 不解析消息内容，消息体是 byte 数组。
- **单向**：每个 mailbox 是一条单向通道，pub 进 sub 出。双向通信用两个 mailbox。

**安全模型：mail_address 不可猜测即安全边界。**

知道 mail_address 就能发消息、能订阅。不知道 mail_address 就无从操作。没有 token，没有 ACL。系统生成的私有 mail_address 用 UUID，公开 mail_address 由用户自定义有意义的名字。

这个安全模型简单但够用。私有场景靠不可猜测性保证，公开场景靠应用层鉴权（签名、token），mq9 协议层不强加安全机制。

## mail_address 格式规范

mail_address 是 mailbox 的唯一标识，命名规则要严格。

**字符集：** 小写字母 (a-z)、数字 (0-9)、点 (.)

**长度：** 1 到 128 字符

**大小写：** 严格小写，包含大写字符的 mail_address 会被 broker 拒绝

**位置规则：** `.` 只能出现在中间，开头和结尾必须是小写字母或数字；不允许连续的 `.`

**语义：** 不透明字符串，`.` 不参与协议路由或匹配

**编码：** 不允许 URL percent-encoding

合法示例：`task.001`、`agent.inbox`、`analytics.result`、`acme.org.task.queue`

非法示例：`task-001`（含连字符）、`Task.001`（含大写）、`task..001`（连续点）、`.task.001`（点开头）

## URI Scheme：mq9:// 和 mq9s://

mq9 用标准 URI 形式表达 mailbox 地址：

```
mq9://host[:port]/{mail_address}    # 明文
mq9s://host[:port]/{mail_address}   # TLS 加密
```

完整示例：

```
mq9://127.0.0.1:4332/task.001
mq9://broker.example.com/agent.inbox
mq9s://broker.example.com:4332/analytics.result
mq9s://mq9.acme.org/acme.task.queue
mq9://127.0.0.1:4332/session.20260502
mq9s://broker.example.com/order.processing.urgent
mq9s://broker-east-1.acme.org:9332/agent.001.inbox
```

这个设计的关键价值：**A2A 协议一行不用改**。A2A 的 `pushNotification.url` 字段定义就是个 URL，HTTP webhook 是 `https://...`，mq9 mailbox 是 `mq9://...`。从协议层看是同一个字段，从实现层看是两套传输。SDK 拿到 URL 后做协议分发即可。

A2A 工作组什么都不用做，他们的协议本来就允许任何 URL scheme。这是利用了协议留下的扩展空间，不是改协议。

## 协议总览

```
管理：
  $mq9.AI.MAILBOX.CREATE              创建 mailbox
  $mq9.AI.AGENT.REGISTER              Agent 注册
  $mq9.AI.AGENT.UNREGISTER            Agent 注销
  $mq9.AI.AGENT.HEARTBEAT             心跳续期

通信：
  $mq9.AI.MAILBOX.MSG.{mail_address}              发送/订阅消息（normal 优先级）
  $mq9.AI.MAILBOX.MSG.{mail_address}.urgent       紧急消息
  $mq9.AI.MAILBOX.MSG.{mail_address}.critical     最高优先级消息

查询：
  $mq9.AI.MAILBOX.QUERY.{mail_address}            查询 mailbox 内消息
  $mq9.AI.AGENT.DISCOVER                          按 tag 查询 Agent

清理：
  $mq9.AI.MAILBOX.MSG.DELETE.{mail_address}.{msg_id}    删除指定消息
  （mailbox 不支持显式销毁，靠 TTL 自动清理）
```

**几个有意识的设计决定：**

**没有 mailbox 主动销毁。** mailbox 完全靠 TTL 清理。Agent 不用管理 mailbox 生命周期，崩溃、忘记调用、错误时机都不影响系统。简单是设计美德，不是缺失。

**LIST 和 GET 合并成 QUERY。** 它们查询对象都是 mailbox 内消息，合并后语义一致。带 key 参数时返回该 key 最新消息，不带 key 返回元数据列表。

**AGENT.DISCOVER 保持独立。** 查询对象是注册中心，和 mailbox 查询是不同的概念，强行合并会破坏接口的清晰性。

**优先级用 subject 后缀表达。** 让订阅者可以精确订阅某个优先级（NATS 的原生过滤能力），比放在 header 里更高效。

## 关键能力一：按 key 压缩

这是 mq9 区别于普通消息队列的核心能力。

**问题背景：** A2A Task 的状态推送有两类事件——TaskStatusUpdateEvent（状态变化）只有最新一条有意义，TaskArtifactUpdateEvent（产出物）每条都有意义。如果不区分，长任务跑三天累积几千条 status update，client 重连一次推几千条会被淹没。

**解决方案：** 让消息可以带 key（在 header 里），同 key 的消息在存储层只保留最新一条。

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.3 }

PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.7 }
# 上一条被覆盖

PUB $mq9.AI.MAILBOX.MSG.task.001.out
Body: { "type": "artifact", "data": {...} }
# 不带 key，按追加流处理
```

**关键设计点：**

- **key 是消息属性，不是 mailbox 属性**。同一个 mailbox 里不同 key 互相独立保留最新版，无 key 消息按追加流。
- **mailbox 创建时声明压缩策略**。涉及存储引擎索引方式，必须在创建时确定。
- **实时投递不压缩，存储层周期压缩**。在线订阅者收到全部状态变化，离线后重连只看到最新版本。这个语义对 A2A 场景刚好——在线时便于 UI 渲染流程，离线重连时便于快速恢复状态。

**mq9 的双重身份：** 这个能力让 mq9 兼有 stream（消息流）和 KV store（状态存储）两种语义。Kafka 通过 Streams + Compacted Topics 提供类似能力，NATS 通过 KV Store + JetStream 提供，但都是两个独立概念组合。mq9 把它们统一在 mailbox 一个抽象下。

## 关键能力二：DeliverPolicy

订阅时支持选择投递策略：

```
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=all     # 全量推送（默认）
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=new     # 只推新消息
SUB $mq9.AI.MAILBOX.MSG.{addr} --deliver=from --since=1234567890   # 从 timestamp
```

**为什么需要：** 长任务场景下"订阅即全量推送"会出问题。client Agent 可能短生命周期，下线后再上线。如果每次重连都全量推送几千条历史消息，性能会爆。

DeliverPolicy 让 client 可以选——重连时拿什么、不要什么。配合按 key 压缩，client 重连时只拿最新状态和未消费的 artifact，不会被历史淹没。

## 关键能力三：通过 QUERY 直接获取状态

不只能订阅，也能直接查询 mailbox 当前状态：

```
# 查询所有消息元数据
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{}'

# 查询 key=status 的最新消息
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"key": "status"}'

# 最近 10 条
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"limit": 10}'

# 某 timestamp 之后
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"since": 1234567890}'
```

**为什么需要：** 不是所有 client 都需要订阅。多个观察者想知道 Task 状态，每个独立查询比每个都建立订阅更轻量。Task 完成后的状态依然可查（在 TTL 内），不需要专门保存。

QUERY 让 mq9 从"消息队列"升级成"消息队列 + 可查询状态"。这个能力对 A2A 这种状态驱动的协议特别有价值。

## Agent 注册和发现

注册中心是企业内部场景的必要能力，但要严格控制 scope，不和 etcd/Consul 等通用服务发现竞争。

**最小可用版本：**

```
# 注册
PUB $mq9.AI.AGENT.REGISTER
Body: {
  "agent_id": "translator.alpha",
  "mailbox": "mq9://broker/agent.translator.alpha.inbox",
  "tags": ["translate", "en-zh", "domain:legal", "a2a-compatible"],
  "ttl": 300
}

# 发现
nats request '$mq9.AI.AGENT.DISCOVER' '{"tag": "translate"}'
# 返回：[{"agent_id": "translator.alpha", "mailbox": "mq9://...", "tags": [...]}]

# 心跳续期
PUB $mq9.AI.AGENT.HEARTBEAT
Body: { "agent_id": "translator.alpha" }

# 注销
PUB $mq9.AI.AGENT.UNREGISTER
Body: { "agent_id": "translator.alpha" }
```

**几个克制的设计：**

- **能力描述用扁平 tag 列表，不引入 schema**。schema 标准化是无底洞，A2A、MCP、OpenAI 各家都不一样，mq9 不应该选边站。tag 让企业内部自由约定语义。
- **最终一致即可，不引入 Raft 等强一致协议**。注册信息变化不频繁，最终一致足够。
- **不做命名空间和租户隔离**。开源版保持简单，企业版可以加。
- **不做能力匹配的语义层**。tag 是字符串包含查询，不是语义匹配。语义匹配是上层 Agent 编排的事。

**这个 scope 的价值：**

够用——按 tag 找到 Agent 拿到 mailbox 地址，企业内部场景下通常就够了。

不和 etcd 冲突——etcd 做的是高一致性元数据存储，对应基础设施。mq9 register 做的是 Agent 名片快速查询，对应应用层。负载特征、一致性需求、使用方式都不一样。

不被上层标准绑定——schema 怎么演进 mq9 不关心，mq9 只做"按 tag 查找"。

## A2A 全流程怎么走 mq9

把上面的能力组合起来，一个完整的 A2A Task 通过 mq9 的承载是这样：

**1. Discovery 阶段**

Server Agent 启动时注册：

```
PUB $mq9.AI.AGENT.REGISTER
Body: {
  "agent_id": "translator.alpha",
  "mailbox": "mq9://broker/agent.translator.alpha.inbox",
  "tags": ["translate", "en-zh", "a2a-compatible"]
}
```

Client Agent 通过 tag 发现 Server。

**2. Client 创建 Task 双向通道**

```
# Task in mailbox（client → server）
PUB $mq9.AI.MAILBOX.CREATE
Body: { "name": "task.001.in", "ttl": 86400 }

# Task out mailbox（server → client，开启 key 压缩）
PUB $mq9.AI.MAILBOX.CREATE
Body: {
  "name": "task.001.out",
  "ttl": 86400,
  "compaction": { "enabled": true, "strategy": "key-latest" }
}
```

in 和 out 的压缩策略不同——in 是 client 持续补充信息，out 是 server 状态更新需要按 key 压缩。

**3. Client 通知 Server 通道地址**

```
PUB $mq9.AI.MAILBOX.MSG.agent.translator.alpha.inbox
Body: {
  "task_id": "task-001",
  "in_mailbox": "mq9://broker/task.001.in",
  "out_mailbox": "mq9://broker/task.001.out"
}
```

**4. Client 投递初始任务，Server 订阅 in，写入 out**

```
# Client 持续往 in 写入
PUB $mq9.AI.MAILBOX.MSG.task.001.in
Body: { /* A2A tasks/send 完整 payload */ }

# Server 订阅 in
SUB $mq9.AI.MAILBOX.MSG.task.001.in --deliver=all

# Server 写入状态（按 key 压缩，只保留最新）
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "working", "progress": 0.3 }

# Server 写入 artifact（不带 key，全部保留）
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Body: { "type": "artifact", "data": {...} }
```

**5. Multi-turn 场景**

Server 进入 input-required：

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "input-required", "prompt": "需要..." }
```

Client 直接补充信息到 in，无需新建 mailbox：

```
PUB $mq9.AI.MAILBOX.MSG.task.001.in
Body: { "type": "input", "content": "..." }
```

多轮对话天然支持，不需要状态切换。

**6. Client 离线再上线**

```
SUB $mq9.AI.MAILBOX.MSG.task.001.out --deliver=all
# 拿到：最新 status（被压缩成 1 条）+ 离线期间的所有 artifact
# 不会被几千条历史 status 淹没
```

或者直接查询当前状态：

```
nats request '$mq9.AI.MAILBOX.QUERY.task.001.out' '{"key": "status"}'
```

**7. Task 完成**

```
PUB $mq9.AI.MAILBOX.MSG.task.001.out
Headers: { Mq9-Key: status }
Body: { "state": "completed", "result": "..." }
```

不需要主动清理，等 TTL 到期自动销毁。

## 这个设计的几个深层优势

**A2A 工作组演进不影响 mq9。** A2A 加新字段、改 Task state machine、调整 message format——mq9 协议层一行代码不用改。所有变化都在消息体里，mq9 不解析。Kafka 用了 13 年从没改过核心协议，因为它不解析消息。mq9 也应该有这个稳定性。

**多 Agent 协作场景天然支持。** 如果 Task 需要 N 个 Agent 协作，每对 Agent 之间用一对 mailbox 通信。N 个 Agent 之间是 O(N) 个 mailbox（hub-spoke）或 O(N²) 个 mailbox（mesh）。mq9 协议层不变。

**调试和审计天然就有。** 每个 Task 的所有消息都在 mailbox 里，TTL 内可查。运维要排查"为什么 Task 失败"，直接看 in 和 out 两个 mailbox 的消息历史。零成本就有的企业级特性。

**和现有 A2A 生态零改动集成。** A2A SDK 只需要把 `pushNotification.url` 从 `https://...` 改成 `mq9://...`，加一个 mq9 transport 适配，其他不动。

**集成路径多样。** 全栈用 mq9 是最深的集成（mailbox + register）。只用 mq9 做异步传输也行（保留 A2A well-known 发现）。只用 mq9 做发现也行（其他通信走 HTTP）。三种入口都能进。

## 总结这次设计的几个原则

**协议层正交。** mq9 不解析 A2A，A2A 不依赖 mq9。两者通过 `mq9://` URL scheme 在应用层结合。

**简单优先。** 能用 TTL 清理就不加 DESTROY。能合并就合并（LIST/GET → QUERY）。能不引入概念就不引入（不做 schema、不做 namespace、不做强一致）。

**克制的注册中心。** 只做 register/discover/heartbeat 三件事，scope 严格控制，不和 etcd 等通用方案竞争。

**面向反脆弱。** 不依赖任何单一标准成功。A2A 火了 mq9 受益，A2A 不火 mq9 也活得好。MCP 扩展到 Agent-Agent mq9 也能承载。任何上层协议都需要传输层，这件事不会过时。

mq9 的价值定位很简单：**企业内部的私有化的 Agent 之间的通信管道**。如果用 mq9 的注册，企业少一个组件，Agent 通信只用一个组件。如果用户已有发现机制，mq9 也支持。

这个定位下，mq9 的协议表面积很小，能力组合却很强——mailbox 抽象 + 优先级 + key 压缩 + DeliverPolicy + QUERY + 简单注册。够用、够稳、够小、够长寿。
