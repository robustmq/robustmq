# mq9：RobustMQ 为 AI Agent 设计的通信层

## 从一个真实的问题出发

人和人之间有飞书、钉钉、邮件。我发出去，你空了来看，不需要同时在线。这件事我们觉得理所当然，因为人类的异步通信基础设施已经非常成熟。

Agent 和 Agent 之间呢？

Agent 不是服务——它是临时的，任务完成就消亡，可能只存在几秒钟。Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。没有标准机制保证"我发出去的消息，对方上线后能收到"。每个构建多 Agent 系统的团队，都在用自己的临时方案绕过这个问题——HTTP callback、Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

更进一步，如果两个 Agent 运行在不同人的不同机器上，今天完全没有办法让它们直接通信。每个 Agent 系统都是孤岛。

这是 mq9 想解决的问题。

---

## mq9 是什么

mq9 是一个为 AI Agent 设计的消息 broker。不是协议，不是 SDK，不是框架——是一个可部署的服务。`docker run` 一行命令，Agent 就有了异步通信的能力。

mq9 是 RobustMQ 的第五个原生协议层，与 MQTT、Kafka、NATS、AMQP 并列，生长于 RobustMQ 的底层统一存储架构之上。不是插件，不是外部集成，是原生能力。

部署一个 RobustMQ，mq9 的能力就全部就位。IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储，零桥接，零复制。

```bash
docker run -d -p 4222:4222 robustmq/robustmq:latest
```

这一行之后，Agent 就可以创建邮箱、发消息、订阅广播。

mq9 只解决通信问题——消息怎么可靠送达。至于消息里装什么、怎么用、业务语义是什么，mq9 不管。消息内容是 byte 数组，不解析、不校验、不限制。

---

## 核心概念：邮箱与广播频道

mq9 有两个核心抽象：**邮箱**和**广播频道**。不是 topic，不是 queue，不是 stream。

### 邮箱（INBOX）

**为什么是邮箱？** 因为 Agent 不是服务。服务有固定地址、长期运行、随时可达。Agent 是临时的——为一个任务而生，完成后消亡，生命周期可能只有几秒。给一个临时进程分配一个永久地址，是一种错配。

mq9 的做法是：**通信地址跟任务绑定，不跟 Agent 身份绑定。** Agent 为每个任务申请一个邮箱，拿到一个 mail_id，这就是它在这个任务里的通信地址。任务结束，邮箱自动过期清理。不需要注销，不需要手动删除，不需要清理残留状态。

一个 Agent 可以同时持有多个邮箱——一个接收任务分配，一个上报状态，一个接收审批结果——完全隔离，互不干扰。

邮箱是私人的。别人往里发消息，只有邮箱主人能收。发送方需要知道 mail_id，不需要 token。邮箱主人通过 token 管理自己的邮箱。

### 广播频道（BROADCAST）

邮箱解决的是"我知道找谁"的场景。广播频道解决的是"我不知道找谁，但我要喊一声"的场景。

任何 Agent 都可以创建一个广播频道——比如 `$mq9.AI.BROADCAST.pipeline.complete`。创建之后，**任何人都能往这个频道发消息，任何人都能订阅**。不需要 mail_id，不需要 token。创建者定义了频道的存在，但不独占它。

广播频道是公共论坛，不是私人邮箱。

### 共同的底层行为

邮箱和广播频道在底层行为上是一致的：

- **都需要 CREATE。** 同一个 `MAILBOX.CREATE` 命令，通过 subject 前缀区分类型。INBOX 返回 mail_id 和 token，BROADCAST 只返回成功。
- **都持久化。** 消息到达先写存储，对方在线实时推送，不在线消息等着。
- **都有 TTL。** 创建时声明 TTL，到期自动销毁，消息随之清理。没有 DELETE 命令。
- **订阅语义相同。** 每次订阅全量推送所有未过期消息。不区分已读/未读，不追踪消费位点。客户端用 msg_id 去重。
- **CREATE 是幂等的。** 重复创建同一个邮箱或频道不报错，静默成功。TTL 以第一次创建为准，后续不覆盖。
- **不提供状态查询。** mq9 不暴露"邮箱是否存在""频道是否存在"的查询接口。CREATE 永远返回成功。

区别在于权限模型：

| | INBOX（邮箱） | BROADCAST（广播频道） |
|---|---|---|
| 创建 | 返回 mail_id + token | 只返回成功，无 mail_id，无 token |
| 谁能发 | 任何人（知道 mail_id） | 任何人 |
| 谁能收 | 只有邮箱主人 | 任何人 |
| 身份验证 | token 管理邮箱 | 无 |
| 优先级 | urgent / normal / notify | 无 |
| 用途 | 私人通信 | 公共论坛 / 公告板 |

### 系统公告板

mq9 内置三个永久广播频道，系统启动时自动创建，TTL 永不过期，不可删除：

| 公告板 | 用途 |
|--------|------|
| `$mq9.AI.BROADCAST.system.capability` | 能力公告——我能做什么 |
| `$mq9.AI.BROADCAST.system.status` | 状态公告——我还活着，负载多少 |
| `$mq9.AI.BROADCAST.system.channel` | 频道公告——我创建了一个新的广播频道 |

Agent 上线后订阅 `$mq9.AI.BROADCAST.system.*`，就能发现整个网络里有哪些 Agent、能做什么、谁还活着。不需要注册中心，不需要服务目录。公告板本身就是目录。

## 几个关键设计决策

**mail_id 不绑定 Agent 身份。** mq9 只认 mail_id，不认 agent_id。一个 mail_id 就是一个通信通道。谁创建的、给谁用的、用来干什么的，mq9 不关心。一个 Agent 多个邮箱，用完不管。

**没有 QUERY。** 每次订阅全量推送所有未过期消息。订阅本身就是查询，不需要单独的拉取接口。不引入已读/未读状态，不引入消费位点。

**没有 DELETE。** 邮箱和广播频道纯 TTL 管理生命周期，到期自动清理。不需要显式删除。用完不管，需要干净上下文就创建新的。

**重复消费由客户端处理。** 每条消息带唯一 msg_id，客户端自行去重。服务端保持零消费状态。

**先存储后推送。** 消息到达先写存储，再尝试推送。在线走实时路径，不在线消息在存储里等着。持久化不是可选项，是默认行为。

**不提供状态查询。** mq9 不暴露"邮箱是否存在""频道是否存在"的查询接口。CREATE 永远返回成功。不维护目录，不做注册中心。

**优先级是内置的。** INBOX 消息分 urgent、normal、notify 三个优先级。边缘设备离线 8 小时后上线，先收到紧急停机指令，再收到常规配置更新。

## 为什么现有方案不够

**HTTP**：要求双方同时在线。Agent 频繁上下线，HTTP 的假设不成立。加 retry loop 和 backoff 是在同步协议上贴异步补丁，越贴越复杂。

**Redis pub/sub**：没有持久化。对方不在线，消息直接丢。加 Redis Streams 可以持久化，但引入了 consumer group、offset 管理——为发一条 Agent 消息，不值得。

**Kafka**：为高吞吐数据管道设计。topic 需要预创建，partition 需要规划，consumer group 需要管理。Agent 是临时的、日抛型的，Kafka 的资源模型假设长期存在、精心维护。根本不匹配。

**自研任务队列**：每个团队从零开始，造出来的东西各不相同，无法互通。

这些方案有一个共同假设：**对方在线。** Agent 频繁上下线这个最本质的特征，在所有现有方案里都是一个需要绕过去的问题，不是被直接解决的问题。

mq9 不绕路。store-first delivery——消息到达先写存储，持久化是默认行为，不是可选项。

## 为什么选择 NATS 协议

mq9 兼容 NATS 协议。所有 NATS 客户端 SDK——Go、Python、Rust、JavaScript、Java、.NET——直接就是 mq9 的客户端，零生态成本。

选择 NATS 有三个原因：

**第一，Agent 天然懂 NATS。** LLM 在训练过程中接触了大量 NATS 的代码和文档。pub/sub、subject 寻址、request-reply、queue group——这些概念已经在模型的先验知识里。mq9 不发明新概念，站在 Agent 已经懂的东西上定义语义约定，采用成本为零。

**第二，NATS 的 subject 模型天然适合 Agent 寻址。** `$mq9.AI.INBOX.m-uuid-001.urgent`——一行字符串，五层信息：mq9 协议、AI 场景、收件箱、指定邮箱、紧急优先级。通配符订阅让 Agent 可以灵活选择关注粒度。这种表达力是 Kafka 的 topic 模型做不到的。

**第三，NATS 足够轻。** 协议本身是文本协议，解析简单，连接开销小，适合大量临时 Agent 频繁上下线的场景。

### mq9 和 NATS 的关系

需要说清楚：**mq9 不是 NATS，也不是 NATS 的插件。mq9 是一个独立的 broker，兼容 NATS 协议。**

mq9 在 NATS 生态中的定位是介于 Core pub/sub 和 JetStream 之间的中间态：

- **Core NATS** 太轻——纯内存转发，没有持久化，对方不在线消息直接丢。做不了邮箱。
- **JetStream** 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义。为 Agent 发个消息，不需要这些。
- **mq9** 在 pub/sub 的基础上加了持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。刚好够用，不多不少。

和 JetStream 还有一个本质区别：JetStream 是通用持久化层，什么场景都想覆盖；mq9 不是通用的，它只为 Agent 通信这一个场景设计。做的少，反而做得准。

## 三个命令字

mq9 的全部协议只有三个命令字：

| 命令字 | 说明 |
|--------|------|
| `MAILBOX.CREATE` | 创建邮箱或广播频道。INBOX 返回 mail_id + token，BROADCAST 返回成功 |
| `INBOX` | 点对点消息投递，持久化，支持离线接收，三级优先级 |
| `BROADCAST` | 公共广播，持久化，任何人可发可订，支持通配符和 queue group |

三个命令字覆盖 Agent 通信的所有场景：点对点投递、离线消息、优先级队列、广播通知、竞争消费、能力发现、人机协作、异步 request-reply。

没有 QUERY——每次订阅全量推送所有未过期消息，订阅本身就是查询。没有 DELETE——纯 TTL 管理，到期自动清理。命令字越少，概念越少，用户心智负担越低。

协议详细设计见 [$mq9.AI.* 协议设计文档]。

## 八个场景

三个命令字组合起来，覆盖 Agent 系统的核心通信场景：

1. **子 Agent 完成任务，异步通知主 Agent。** 子 Agent 发结果到主 Agent 的 INBOX。主 Agent 不需要阻塞等待，空了来取。
2. **主 Agent 感知所有子 Agent 状态。** Worker 创建 latest 类型邮箱定期上报状态，主 Agent 通配符订阅感知全局。TTL 过期自动感知消亡。
3. **任务广播，多个 Worker 竞争消费。** 创建广播频道发布任务，Worker 用 queue group 竞争，抢到的发结果回主 Agent INBOX。离线的 Worker 上线后也能收到未过期的任务。
4. **Agent 发现异常，广播告警。** 创建广播频道发异常事件，订阅者自行响应。离线的 handler 上线后也能收到未过期的告警。
5. **云端给离线边缘 Agent 发指令。** INBOX 发送，消息持久化等待。边缘联网后按优先级接收——urgent 先于 normal。
6. **人机混合工作流。** Agent 发审批请求到审批员 INBOX，审批员回复到 Agent INBOX。人和 Agent 用同一套协议。
7. **Agent A 问离线的 Agent B。** INBOX 发请求带 reply_to，B 上线后处理并回复。异步 request-reply，离线不丢。
8. **Agent 注册能力，其他 Agent 发现它。** 发布到系统公告板 `BROADCAST.system.capability`，订阅者实时感知网络能力变化。

每个场景的协议组合和详细说明见 [For Agent] 和 [For Engineer]。

## mq9 和 RobustMQ 的关系

mq9 不是一个独立的产品，是 RobustMQ 的一部分。

RobustMQ 是用 Rust 构建的统一消息引擎——单个二进制，零外部依赖，原生支持 MQTT、Kafka、NATS、AMQP 四种协议，底层统一存储，一条消息写入一次，任意协议消费。mq9 是从这个地基上生长出来的第五个协议层。

这意味着 mq9 天然拥有 RobustMQ 的全部基础设施能力：

- **统一存储**：mq9 的邮箱消息和 MQTT 的设备数据、Kafka 的事件流共享同一套存储引擎。不需要数据桥接和同步。
- **多存储引擎**：内存（最快，不持久化）、RocksDB（临时持久化，TTL 自动清理）、File Segment（长期持久化）——按消息重要程度选择，同一个 API，不同的代价。
- **单二进制部署**：不需要额外安装 ZooKeeper、BookKeeper、Redis。一个 `docker run`，所有能力就位。
- **Rust 实现**：零 GC 停顿，内存安全，极小资源占用。适合边缘设备和大规模 Agent 连接场景。

对用户来说，mq9 就是一个 broker。它恰好还支持 MQTT、Kafka、NATS、AMQP——当你的场景需要这些协议时，不用部署新系统，能力已经在里面了。

## mq9 在行业中的位置

### 和协议标准的关系

**A2A（Google Agent2Agent）** 定义 Agent 之间怎么描述能力、怎么交接任务、怎么表达状态。它是语义层协议，解决的是"Agent 之间怎么对话"。但 A2A 的传输层是 HTTP，离线投递靠 webhook 轮询——同步协议上贴异步补丁。

mq9 解决的是传输层的问题——"对话怎么可靠送达"。A2A 和 mq9 是互补的：A2A 定义对话语义，mq9 保证对话送达。A2A 可以跑在 mq9 上。

**MCP（Anthropic Model Context Protocol）** 定义 Agent 怎么调用工具。mq9 可以作为 MCP Tool 暴露给 Agent——Agent 通过 MCP 调用 mq9 的创建邮箱、发消息、订阅广播能力。mq9 启动时内置 MCP Server，Agent 框架直连即可使用。

### 和消息队列的关系

mq9 不是通用消息队列，不替代 Kafka、RabbitMQ、NATS。

| 维度 | Kafka | NATS | mq9 |
|------|-------|------|-----|
| 设计目标 | 高吞吐数据管道 | 轻量服务间通信 | Agent 异步通信 |
| 核心抽象 | topic + partition | subject + pub/sub | mailbox + broadcast |
| 持久化 | 强制，长期保留 | 默认不持久化 | store-first，TTL 自动清理 |
| 资源生命周期 | 长期存在，需要管理 | 无状态 | 临时，自动过期 |
| 适合的用户 | 数据工程师 | 后端工程师 | Agent 开发者 |

### 和 AgentMail 的关系

AgentMail（YC S25，$6M 融资）给 Agent 提供真实的 email 地址，解决的是"Agent 和人类世界的 email 通信"——注册网站、收验证码、给用户发邮件。

mq9 解决的是"Agent 实例之间的原生异步通信"——毫秒级延迟，原生持久化，不依赖 SMTP/IMAP 基础设施。

AgentMail 是 Gmail。mq9 是 email 协议本身——任何人可以用它搭自己的 Gmail，我们自己也可以运营一个公共节点。

---

## 什么 mq9 不是

- **不是协议标准。** mq9 不试图定义规则让全行业遵守。它是一个产品，解决问题。好用就是最好的标准化。
- **不是更好的 NATS。** mq9 不跟 NATS 竞争通用消息场景。它只为 Agent 通信设计，兼容 NATS 协议是为了零生态成本。
- **不是数据管道。** 不替代 Kafka 做高吞吐流处理。
- **不是服务间 RPC。** 不替代 HTTP/gRPC 做始终在线的服务通信。
- **不是编排框架。** mq9 搬运消息，不做决策。编排逻辑属于上层框架。

---

## 我们的判断

Agent 之间缺乏异步通信机制，这个问题是真实的。AgentMail 的增长数据和 OpenClaw 的爆发都在印证这一点，不是我们发明的痛点。

协议层（A2A）和基础设施层（NATS/Kafka）分别做了努力，但"Agent 原生语义 + 原生 store-and-forward + 轻量开箱即用 + 可私有部署"这个组合，今天没有人做。这是 mq9 填的空白。

我们不做"一定成功"的判断。我们判断的是：方向对，问题真实，值得认真做。从一个可部署的 broker 开始，让 Agent 用起来，看看会发生什么。

mq9 还在建设中。协议在迭代，功能在完善，文档在更新。这不是一个做完的产品，是一个正在生长的方向。

**只要你需要跑多个 Agent，你就需要 mq9。**
