# mq9：RobustMQ 为 AI Agent 设计的通信层

## 从一个真实的问题出发

人和人之间有飞书、钉钉、邮件。我发出去，你空了来看，不需要同时在线。这件事我们觉得理所当然，因为人类的异步通信基础设施已经非常成熟。

Agent 和 Agent 之间呢？

Agent 不是服务——它是临时的，任务完成就消亡，可能只存在几秒钟。今天，Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。没有标准机制保证"我发出去的消息，对方上线后能收到"。每个构建多 Agent 系统的团队，都在用自己的临时方案绕过这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

这是 mq9 想解决的问题。

---

## mq9 是什么

mq9 是 RobustMQ 专门为 AI Agent 通信设计的协议层，与 MQTT、Kafka、NATS、AMQP 并列，生长于 RobustMQ 的底层统一存储架构之上。

RobustMQ 是用 Rust 构建的统一消息引擎——单个二进制，零外部依赖，原生支持 MQTT、Kafka、NATS、AMQP 四种协议，底层统一存储，一条消息写入一次，任意协议消费。这是 RobustMQ 三年打磨的地基。

mq9 是从这个地基上自然生长出来的第五个协议层。不是插件，不是外部集成，是原生协议能力。就像 MQTT 专门为 IoT 设计，Kafka 专门为数据流管道设计，mq9 专门为 Agent 通信设计。

**部署一个 RobustMQ，mq9 的能力就全部就位。** IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储，零桥接，零复制。

---

## 为什么当前方案都不够好

**HTTP** 是同步的。调用方必须等待，对方必须在线。Agent 频繁上下线，HTTP 的假设根本不成立。

**Redis pub/sub** 没有持久化。对方不在线，消息直接丢，无法保证送达。

**Kafka** 太重。为大批量持久化数据流设计，没有 Agent 身份概念，没有临时邮箱，没有优先级队列，topic 的创建需要管理员操作。Agent 是日抛型的，Kafka 的设计假设是资源长期存在、需要精心维护，两者根本不匹配。

**自研任务队列** 每个团队重复造轮子，造出来各不相同，无法互通。

这些方案有一个共同的缺陷：**都假设对方在线。** Agent 频繁上下线的特点，在所有现有方案里都是一个需要绕过去的问题，不是被直接解决的问题。

---

## Agent 通信的本质特征

理解 mq9 的设计，需要先理解 Agent 通信跟传统服务通信有什么本质不同。

**消息小，但连接多，关系网状。** 一条 Agent 消息通常只有几十到几百字节——一个任务指令，一个状态更新，一个结果通知。挑战不在单条消息的大小，而在连接数量和关系的复杂度。一个 Orchestrator 同时跟几十个 Worker 通信，每个 Worker 可能又在跟其他 Agent 通信，形成复杂的网状拓扑。

**点对点解决不了网状问题。** A 要跟 B、C、D、E 通信，B 也要跟 C、D 通信——点对点需要维护 n² 个连接，任何节点的加入或退出都需要更新所有相关连接。广播 + 通配符订阅才是网状通信的正确模型。

**Agent 是临时的，生命周期不可预测。** 任务完成就消亡，资源不够被杀掉，网络抖动断线，版本更新重启。在线时间可能只有几秒钟。通信系统必须为这个特点专门设计，不能假设对方一直在线。

这三个特征，决定了 mq9 的三个核心原语。

## 三个核心原语

### 邮箱（Inbox）

邮箱解决"我知道要找谁，但不知道对方在不在线"的问题。

每个 Agent 有自己的邮箱地址。发件方直接发到对方邮箱，消息持久化等待。对方什么时候上线，什么时候来取，发件方完全不需要管。

```bash
# 申请邮箱，调用即有身份
nats req '$mq9.AI.MAILBOX.CREATE' '{}'
# → {"mail_id": "agt-uuid-001", "token": "tok-xxx"}

# 发消息到另一个 Agent 邮箱
nats pub '$mq9.AI.INBOX.agt-uuid-002.NORMAL' \
  '{"from":"agt-uuid-001","type":"task_result","payload":"..."}'

# 订阅自己的邮箱
nats sub '$mq9.AI.INBOX.agt-uuid-001.*'
```

邮箱支持三个优先级：`URGENT`、`NORMAL`、`NOTIFY`。同优先级保证 FIFO，紧急消息先处理，不会被普通任务淹没。

邮箱是临时的。Agent 注册时声明 TTL，心跳续期，超时自动销毁。消息随邮箱一起清理。Agent 完全不需要关心资源管理。

### 广播（Broadcast）

广播解决"我不知道谁需要，但需要的人应该知道"的问题。

发布方完全不关心消费方，发出去就完了。订阅方自己决定关心什么，用通配符表达兴趣。新 Agent 加入，直接订阅相关 subject，不需要通知任何人。

```bash
# 广播任务可用
nats pub '$mq9.AI.BROADCAST.task.available' \
  '{"task_id":"t-001","reply_to":"$mq9.AI.INBOX.agt-001.NORMAL"}'

# 订阅所有任务事件
nats sub '$mq9.AI.BROADCAST.task.*'

# 订阅所有域的异常事件
nats sub '$mq9.AI.BROADCAST.*.anomaly'

# 竞争消费——同一条消息只有一个 Worker 处理
nats sub '$mq9.AI.BROADCAST.task.available' --queue task.workers
```

广播默认不持久化——时效性的，错过就错过。重要广播显式开启持久化：

```bash
nats pub '$mq9.AI.BROADCAST.system.shutdown' '{}' --persist --ttl 3600
```

### 状态上报（Status）

状态上报解决"主 Agent 怎么感知所有子 Agent 的存在和状态"的问题。

每个 Agent 定期把自己的状态写到 `$mq9.AI.STATUS.{mail_id}`，使用 retain=latest 语义——只保留最新一条。主 Agent 用通配符订阅一行代码感知全局。

```bash
# Agent 上报状态（含能力声明）
nats pub '$mq9.AI.STATUS.agt-uuid-002' \
  '{"status":"running","load":0.6,"capabilities":["data.analysis"]}'

# 主 Agent 感知所有子 Agent
nats sub '$mq9.AI.STATUS.*'
```

新 Agent 加入不需要注册，消亡不需要注销——超过 TTL 没有上报，状态自动过期，主 Agent 感知消亡。这是去中心化的 Agent 编排基础，不需要任何中心化的注册服务。

## `$mq9.AI.*` 命名空间

mq9 的所有通信都通过 `$mq9.AI.*` 命名空间组织。这个设计把品牌、场景、语义三件事用一个字符串说清楚：

```
$mq9.AI.MAILBOX.CREATE              # 创建邮箱
$mq9.AI.INBOX.{id}.URGENT           # 收件箱 - 紧急
$mq9.AI.INBOX.{id}.NORMAL           # 收件箱 - 普通
$mq9.AI.INBOX.{id}.NOTIFY           # 收件箱 - 通知
$mq9.AI.STATUS.{id}                 # 状态上报
$mq9.AI.BROADCAST.{domain}.{event}  # 广播
```

`$mq9` 小写，表明归属——这是 RobustMQ 的能力。`.AI.` 大写，表明场景——为 AI Agent 设计。后面的关键字大写，表明语义——`INBOX`、`STATUS`、`BROADCAST`，看到 subject 就懂意图。

未来 `$mq9.IOT.*`、`$mq9.EDGE.*` 也可以并列扩展，但 `.AI.` 是现在的焦点。

这个设计有一个被低估的好处：**协议本身就是文档。** `$mq9.AI.INBOX.agt-001.URGENT` ——mq9 协议，AI 场景，收件箱，指定 Agent，紧急优先级。一行字符串，五层信息，不需要查文档。

## 调用即存在，无需任何配置

mq9 的使用门槛趋近于零。

没有注册流程，没有配置文件，没有管理员操作。一行命令，Agent 就有了身份和邮箱。任何语言的 NATS 客户端直接就是 mq9 的客户端——NATS 生态有多大，mq9 的接入生态就有多大，不需要构建任何新的 SDK。

这不是设计上的偷懒，而是一个深思熟虑的决策。给 Agent 用的基础设施，必须贴合已被训练进模型的心智模型。NATS 的 pub/sub、subject 寻址、queue group——这些概念 Agent 早就懂了。mq9 不发明新概念，站在 Agent 已经懂的东西上定义语义约定，采用成本为零。

---

## 存储分级：选择权比绝对保障更重要

传统消息队列的假设是每条消息都很宝贵，必须多副本持久化。但 Agent 通信的消息"价值密度"差异极大——一条协调信号和一条审计记录，需要的保障级别完全不同。

mq9 基于 RobustMQ 的统一存储层，提供三种存储级别：

**Memory**——纯内存，最轻，不持久化。Agent 之间的协调信号、心跳、临时通知。发完就没了，丢了重发。延迟最低，成本为零。

**RocksDB**——临时持久化。消息落盘，等对方上线取走，TTL 到期自动清理。邮箱的默认选择。平衡了可靠性和成本。

**File Segment**——长期持久化。审计日志、关键事件记录、合规数据。永久保留，可回溯。

同一个 mq9 实例，同一套 API，不同的消息走不同的存储。不是所有消息都值得三副本，也不是所有消息都可以丢。**让用户按需选择，系统按配置执行。**

## 八个真实场景

三个原语组合起来，覆盖了 Agent 系统的核心通信场景。

**场景一：异步任务回传。** 子 Agent 完成任务后发结果到 `$mq9.AI.INBOX.{主agent}.NORMAL`，主 Agent 不需要阻塞等待，空了来取。

**场景二：全局状态感知。** 一行订阅 `$mq9.AI.STATUS.*`，新 Worker 加入自动感知，Worker 消亡自动感知，不需要维护任何 Agent 列表。

**场景三：任务广播与竞争消费。** 主 Agent 广播到 `$mq9.AI.BROADCAST.task.available`，有能力的 Worker 自己来抢，queue group 保证每个任务只被一个 Worker 处理，不需要分布式锁。

**场景四：异常告警广播。** 某个 Agent 发现异常，广播到 `$mq9.AI.BROADCAST.{domain}.anomaly`，有处理能力的 Agent 自动响应，发布方不需要知道谁在处理。

**场景五：边缘离线消息积压。** 云端给边缘 Agent 发指令到 `$mq9.AI.INBOX.{edge}.URGENT`，边缘不在线消息持久化等待，联网后按优先级处理。

**场景六：异步请求-响应。** Agent 向另一个 Agent 提问，对方不在线请求在邮箱等着，上线处理完通过 reply_to 发回。correlation_id 保证请求响应配对。

**场景七：能力注册与发现。** Agent 把能力声明写入 `$mq9.AI.STATUS.{id}`，其他 Agent 通过订阅状态或广播查询 `$mq9.AI.BROADCAST.capability.query` 发现有该能力的 Agent。

**场景八：人机混合工作流。** Agent 发审批请求到人类审批员的 `$mq9.AI.INBOX.{approver}.URGENT`，审批完结果发回 Agent 邮箱，流程继续。人类审批员和 Agent 使用完全相同的协议——mq9 的通信模型不区分参与者是人还是 Agent。

## 为什么 mq9 适合 AI

单独看每一个原语都不新鲜：优先级队列消息队列早就有，广播 pub/sub 到处都是，轻量 NATS 已经做到了。

但三个组合在一起，专门为 Agent 的特点设计，就变成了别人没有的东西。

Agent 是异步的——邮箱解决，消息等着，上线按优先级处理。Agent 是不知道彼此的——广播解决，发出去，关心的人自己来。Agent 是临时的——轻量解决，调用即有，到期即消，系统自动回收。

三个原语覆盖了 Agent 通信的所有基本模式，八个场景没有一个落在覆盖范围之外。

最关键的地方不是功能多，而是**足够轻**。三个原语，NATS 原生 SDK，几行代码。采用成本趋近于零——不是技术门槛，而是采用门槛。接入的 Agent 越多，每个 Agent 能找到的协作对象越多，整个系统的价值越大。这是网络效应。

## 单机即可，按需升级

大多数 Agent 通信场景不需要高可用——消息丢了重试就好，任务失败重派就好。Agent 的容错模型跟传统系统完全不同，Agent 天然可以重试。

mq9 单机部署就能满足大量需求。一台普通云主机，`docker run` 一行命令，Agent 邮箱就起来了。不需要搭集群，不需要运维经验。需要高可用时，切集群，接口不变，Agent 无感知。

一键启动，运维简单，协议简单，各种模式。

---

## 它是它

mq9 看起来像消息队列，像 pub/sub，像 JetStream，像谁都有点像。但它是它。

拆开看，mq9 用到的每一个能力都不新鲜。但这个组合，为 Agent 场景专门设计，用邮箱语义统一表达，今天没有人做过。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

不需要跟任何人比较。用的人自然懂。

mq9 在 NATS 生态中的定位是介于 Core pub/sub 和 JetStream 之间的中间态。Core NATS 太轻——没有持久化，离线就丢，做不了邮箱。JetStream 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义，为 Agent 发个消息不需要这些。mq9 在 pub/sub 的基础上加了可选持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。刚好够用，不多不少。和 JetStream 还有一个本质区别：JetStream 是通用持久化层，什么场景都想覆盖；mq9 不是通用的，它只为 Agent 通信这一个场景设计命令字和功能。做的少，反而做得准。
