# mq9：RobustMQ 为 AI Agent 设计的通信层

## 从一个真实的问题出发

人和人之间有飞书、钉钉、邮件。我发出去，你空了来看，不需要同时在线。这件事我们觉得理所当然，因为人类的异步通信基础设施已经非常成熟。

Agent 和 Agent 之间呢？

Agent 不是服务——它是临时的，任务完成就消亡，可能只存在几秒钟。今天，Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。没有标准机制保证"我发出去的消息，对方上线后能收到"。每个构建多 Agent 系统的团队，都在用自己的临时方案绕过这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

这是 mq9 想解决的问题。

---

## mq9 是什么

mq9 是 RobustMQ 专门为 AI Agent 通信设计的协议层，与 MQTT、Kafka、NATS、AMQP 并列，生长于 RobustMQ 的底层统一存储架构之上。

RobustMQ 是用 Rust 构建的统一消息引擎——单个二进制，零外部依赖，原生支持 MQTT、Kafka、NATS、AMQP 四种协议，底层统一存储，一条消息写入一次，任意协议消费。这是 RobustMQ 三年打磨的地基。

mq9 是从这个地基上自然生长出来的第五个协议层。不是插件，不是外部集成，是原生协议能力。就像 MQTT 专门为 IoT 设计，Kafka 专门为数据流管道设计，mq9 专门为 Agent 通信设计。

**部署一个 RobustMQ，mq9 的能力就全部就位。** IoT 设备通过 MQTT 发数据，分析系统通过 Kafka 消费，Agent 通过 mq9 协作——同一个 broker，同一份存储，零桥接，零复制。

### 在 NATS 生态中的定位

mq9 在 NATS 生态中的定位是介于 Core pub/sub 和 JetStream 之间的中间态。Core NATS 太轻——没有持久化，离线就丢，做不了邮箱。JetStream 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义，为 Agent 发个消息不需要这些。mq9 在 pub/sub 的基础上加了可选持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。刚好够用，不多不少。

和 JetStream 还有一个本质区别：JetStream 是通用持久化层，什么场景都想覆盖；mq9 不是通用的，它只为 Agent 通信这一个场景设计命令字和功能。做的少，反而做得准。

## 核心概念：邮箱

mq9 的核心概念来自现实世界的邮件系统：

- **开通邮箱** — 去邮局注册，拿到地址。可以是普通邮箱（存所有信），也可以是公告栏（只留最新一张）。→ `MAILBOX.CREATE`
- **寄信** — 知道对方地址，写好投进去。对方在不在家不管，信在邮箱等着。分急件、普通件、广告。→ `INBOX.{mail_id}.{priority}`
- **收信** — 回家打开邮箱，信自动送到手上。急件先看，普通件后看，广告最后。→ 在线时服务端推送
- **去邮局查信** — 怕有遗漏，主动去问"我还有没有信没取"。→ `MAILBOX.QUERY.{mail_id}`
- **回信** — 信里有对方回信地址，写好寄回去。→ NATS 原生 reply-to
- **邮箱过期** — 长期没人用，邮局自动注销，信跟着一起清理。→ TTL 自动销毁
- **小区广播** — 公告栏，贴上去所有人能看。不需要知道谁会看，想看的自己来。→ `BROADCAST.{domain}.{event}`
- **只看某类公告** — 我只关心安全类公告。→ 通配符订阅
- **抢活** — 公告栏贴了一个任务，好几个人都想接，但只能一个人干。→ queue group

现实邮箱能做的事，mq9 全覆盖。现实邮箱做不到的事（优先级分级、通配符订阅、竞争消费），mq9 也有。

### 关键设计决策

**mail_id 不绑定 Agent 身份。** mq9 只认 mail_id，不认 agent_id。一个 mail_id 就是一个通信通道，谁创建的、给谁用的、用来干什么的，mq9 不关心。一个 Agent 可以为不同任务申请不同的 mail_id，用完不管，TTL 自动清理。

**邮箱不需要显式删除。** 纯 TTL 管理生命周期。创建时声明 TTL，到期自动销毁，消息随邮箱一起清理。客户端语义越简单越好。

**INBOX 是私人邮箱，BROADCAST 是公共广播。** 两种不同的通信模式：知道找谁用 INBOX（需要 CREATE 拿到 mail_id），不知道找谁用 BROADCAST（不需要任何前置操作，直接发直接订阅）。

**先存储后推送。** 消息到达先写存储层，再尝试推送给在线订阅者。推送成功走实时路径，推送失败消息在存储里等着，Agent 可通过 QUERY 兜底拉取。

**不创建新概念。** ACK 复用 NATS 原生 reply-to，订阅复用 NATS 原生 sub 语义，竞争消费复用 NATS 原生 queue group。

## 为什么当前方案都不够好

**HTTP** 是同步的。调用方必须等待，对方必须在线。Agent 频繁上下线，HTTP 的假设根本不成立。

**Redis pub/sub** 没有持久化。对方不在线，消息直接丢，无法保证送达。

**Kafka** 太重。为大批量持久化数据流设计，没有临时邮箱概念，没有优先级队列，topic 的创建需要管理员操作。Agent 是日抛型的，Kafka 的设计假设是资源长期存在、需要精心维护，两者根本不匹配。

**自研任务队列** 每个团队重复造轮子，造出来各不相同，无法互通。

这些方案有一个共同的缺陷：**都假设对方在线。** Agent 频繁上下线的特点，在所有现有方案里都是一个需要绕过去的问题，不是被直接解决的问题。

## 四个命令字

mq9 的全部客户端协议：

```
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # 申请邮箱，返回 mail_id
  │     └── QUERY.{mail_id}          # 查询未读消息（兜底拉取）
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

`MAILBOX.CREATE` — 开邮箱，拿到 mail_id。`MAILBOX.QUERY` — 查邮箱，兜底拉取。`INBOX` — 私人邮箱，点对点通信。`BROADCAST` — 公共广播，按频道喊。

四个命令字，覆盖八个场景，ACK 和订阅全部复用 NATS 原生能力。

### MAILBOX.CREATE — 申请邮箱

```bash
REQ $mq9.AI.MAILBOX.CREATE
{
  "type": "standard",        # standard（普通邮箱）| latest（只保留最新一条）
  "ttl": 3600                # 邮箱生存时间（秒）
}

# 响应
{
  "mail_id": "m-uuid-001",
  "token": "tok-xxx",
  "inbox": "$mq9.AI.INBOX.m-uuid-001"
}
```

邮箱不绑定 Agent 身份。一个 Agent 可以为不同任务申请不同的 mail_id。用完不管，TTL 自动清理。

邮箱有两种类型：`standard` 是普通邮箱，消息累积，按优先级存储；`latest` 是状态邮箱，只保留最新一条，新消息覆盖旧消息，用于状态上报、能力声明等场景。

### MAILBOX.QUERY — 查询邮箱

```bash
REQ $mq9.AI.MAILBOX.QUERY.{mail_id}
{
  "token": "tok-xxx"
}

# 响应
{
  "mail_id": "m-uuid-001",
  "unread": 5,
  "messages": [ ... ]
}
```

QUERY 是推送的兜底机制。正常情况下在线订阅者通过 SUB 实时收到推送。QUERY 用于：推送可能遗漏时主动拉取、Agent 上线后检查离线期间的积压消息、确认邮箱是否还有未处理的消息。

### INBOX — 点对点邮箱

```bash
# 发送消息到邮箱
PUB $mq9.AI.INBOX.{mail_id}.{priority}
{
  "from": "m-uuid-002",
  "type": "task_result",
  "correlation_id": "req-uuid-001",
  "reply_to": "$mq9.AI.INBOX.m-uuid-002.normal",
  "payload": { ... },
  "ts": 1234567890
}

# 订阅邮箱（所有优先级）
SUB $mq9.AI.INBOX.{mail_id}.*

# 只订阅紧急消息
SUB $mq9.AI.INBOX.{mail_id}.urgent
```

| 优先级 | 说明 | 默认持久化 | 建议 TTL |
|--------|------|-----------|---------|
| urgent | 紧急消息，优先处理 | true | 86400s |
| normal | 普通消息，顺序处理 | true | 3600s |
| notify | 通知，后台处理 | false | — |

### BROADCAST — 公共广播

广播不需要 CREATE，不需要任何前置操作。直接发，直接订阅。

```bash
# 发布广播
PUB $mq9.AI.BROADCAST.{domain}.{event}
{
  "from": "m-uuid-004",
  "type": "event_type",
  "payload": { ... },
  "ts": 1234567890
}

# 订阅单个域的所有事件
SUB $mq9.AI.BROADCAST.system.*

# 订阅所有域的特定事件
SUB $mq9.AI.BROADCAST.*.anomaly

# 订阅所有广播
SUB $mq9.AI.BROADCAST.#

# 竞争消费
SUB $mq9.AI.BROADCAST.task.available  queue_group=task.workers
```

`{domain}.{event}` 由用户自定义，mq9 只规定前缀 `$mq9.AI.BROADCAST.`。广播默认不持久化，重要广播显式设置 persist=true。

## 八个真实场景

四个命令字组合起来，覆盖 Agent 系统的核心通信场景。

**场景一：子 Agent 完成任务，异步通知主 Agent。** 主 Agent CREATE 邮箱，子 Agent 完成后 PUB 到主 Agent 的 INBOX。主 Agent 不需要阻塞等待，空了来取。

**场景二：主 Agent 感知所有子 Agent 状态。** Worker 申请 type=latest 的状态邮箱，定期 PUB 状态。主 Agent 订阅感知全局，新 Worker 加入自动感知，TTL 过期自动感知消亡。

**场景三：任务广播，多个 Worker 竞争消费。** 主 Agent BROADCAST 任务，Worker 用 queue group 竞争消费，抢到的 Worker 发结果到主 Agent INBOX。

**场景四：Agent 发现异常，广播告警。** BROADCAST 异常事件，订阅了该域的 Agent 自动响应。发布方不需要知道谁在处理。

**场景五：云端给离线边缘 Agent 发指令。** PUB 到边缘 Agent 的 INBOX（urgent/normal 分优先级），边缘不在线消息持久化等待，联网后按优先级取。QUERY 兜底拉取确保不遗漏。

**场景六：人机混合工作流。** Agent PUB 审批请求到人类审批员 INBOX，审批员处理后 PUB 结果回 Agent INBOX。人类和 Agent 用完全相同的协议。

**场景七：Agent A 向 Agent B 提问，B 可能不在线。** A PUB 请求到 B 的 INBOX 带 reply_to 和 correlation_id，B 上线处理后 PUB 响应到 A 的 INBOX。异步 request-reply，离线不丢。

**场景八：Agent 注册能力，其他 Agent 发现它。** Agent 把能力声明 PUB 到自己的 latest 邮箱，其他 Agent 订阅感知，或通过 BROADCAST.capability.query 主动查询。去中心化，零额外服务。

## 存储分级

mq9 运行在 RobustMQ 统一存储层之上，提供三种存储能力：

**Memory** — 纯内存，最轻，不持久化。Agent 协调信号、心跳、临时通知。丢了重发。

**RocksDB** — 临时持久化。消息落盘，等对方上线取走，TTL 到期自动清理。邮箱的默认选择。

**File Segment** — 长期持久化。审计日志、关键事件，永久保留。

一条协调信号走 Memory，一条任务指令走 RocksDB，一条审计记录走 File Segment。同一个系统，同一个 API，不同的代价。不是所有消息都值得三副本，也不是所有消息都可以丢。让用户按需选择。

## 调用即存在，零生态成本

没有注册流程，没有配置文件，没有管理员操作。一行命令，Agent 就有了邮箱。

任何语言的 NATS 客户端直接就是 mq9 的客户端——Go、Python、Rust、Java、JavaScript，所有 SDK 开箱即用。不需要构建任何新的 SDK，零生态成本。

这不是偷懒，是深思熟虑的决策。给 Agent 用的基础设施，必须贴合已被训练进模型的心智模型。NATS 的 pub/sub、subject 寻址、queue group——这些概念 Agent 早就懂了。mq9 不发明新概念，站在 Agent 已经懂的东西上定义语义约定，采用成本为零。

`$mq9.AI.*` 的命名空间设计，让协议本身成为文档。`$mq9.AI.INBOX.m-001.URGENT` ——mq9 协议，AI 场景，收件箱，指定通道，紧急优先级。一行字符串，五层信息，看到 subject 就懂语义。

## 单机即可，按需升级

大多数 Agent 通信场景不需要高可用——消息丢了重试就好，任务失败重派就好。Agent 天然可以重试。

mq9 单机部署就能满足大量需求。一台普通云主机，`docker run` 一行命令，Agent 邮箱就起来了。不需要搭集群，不需要运维经验。需要高可用时，切集群，接口不变，Agent 无感知。

一键启动，运维简单，协议简单，各种模式。

---

## 它是它

mq9 看起来像消息队列，像 pub/sub，像 JetStream，像谁都有点像。但它是它。

拆开看，mq9 用到的每一个能力都不新鲜。但这个组合，为 Agent 场景专门设计，用邮箱语义统一表达，今天没有人做过。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

不需要跟任何人比较。用的人自然懂。
