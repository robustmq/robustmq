# Agent 邮箱：RobustMQ 进入 Agent 时代的切入点

## 从一个问题开始

人和人说话，如果对方不在线，你发飞书、发钉钉、发邮件——消息在那里等着，对方空了来看。这件事我们觉得理所当然，因为人类的异步通信基础设施已经非常成熟。

Agent 和 Agent 之间呢？

Agent 的上下线比人类频繁得多——任务完成立刻消亡，资源不够被杀掉，网络抖动断线，版本更新重启，日抛型任务用完即弃。Agent 的在线时间可能只有几秒钟。

今天，Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。没有任何标准机制保证"我发出去的消息，对方上线后能收到"。这个问题真实存在，而且没有人在认真解决它。

---

## 问题的规模

这不是一个边缘场景。

随着 Agent 系统从单个 Agent 演进到多 Agent 协作，从单机部署演进到分布式系统，Agent 之间的异步通信会成为核心基础设施需求。几个已经在发生的信号：

Dify、Coze、Manus 这类 Agent 平台，用户触发一个复杂任务，平台拆解给多个 Agent 并行处理。子 Agent 完成了怎么通知主 Agent？今天的做法是轮询或者 webhook 回调——轮询浪费资源，webhook 要求主 Agent 必须有公网地址，两个都是绕路。

多框架 Agent 系统——一个企业可能同时有用 Claude 写的分析 Agent、用 GPT 写的客服 Agent、用自研模型的审核 Agent。它们之间今天没有标准的通信方式，每次跨框架调用都是定制集成。

边缘 Agent 与云端 Agent 的协作——边缘设备网络不稳定，时断时续。云端 Agent 的指令怎么可靠送达？边缘 Agent 的数据怎么在联网时自动同步？

这些问题今天都没有好答案，每个团队都在用自己的临时方案绕过去。

---

## 今天的解决方案，以及它们的问题

**HTTP 直连**：Agent A 调 Agent B 的接口，B 不在线直接报错。没有离线消息的概念，本质上是同步调用，不适合 Agent 频繁上下线的场景。

**轮询数据库**：把消息写进数据库，对方上线后来轮询。能用，但实时性差，数据库压力大，每个团队都在自己实现这套逻辑，重复造轮子。

**Redis pub/sub**：发布订阅，实时性好，但没有持久化。B 不在线，消息发出去直接丢，没有任何补救。

**消息队列（Kafka/RabbitMQ）**：能解决持久化问题，但太重。为 Agent 临时通信引入一个 Kafka 集群，配置复杂，运维成本高。没有"Agent 邮箱"的抽象，要自己设计 topic 命名和路由规则，每个团队设计出来都不一样，无法互通。

所有这些方案都缺同一个东西：为 Agent 频繁上下线、临时存在、异步协作的特点专门设计的通信基础设施。

## Agent 通信的本质特征

一条微信消息，几十个字。一封邮件，几百个字。单条消息的信息量有限，但一个人一天发出去的消息数量很多，跟很多人在同时通信，关系是网状的。没有人会把整本书通过一条消息发给你，信息是分散的、碎片化的、按需传递的。

Agent 通信是一样的。一个 Agent 调用另一个 Agent，传递的是一个具体的指令或者结果，不是全量的上下文。一个 Agent 广播一个状态更新，几十个字节。一个 Agent 请求另一个 Agent 执行一个动作，参数有限。

但 Agent 的数量是巨大的，关系是网状的——一个 Orchestrator 同时跟几十个 Worker 通信，每个 Worker 可能又在跟其他 Agent 通信，形成复杂的网状拓扑。

**Agent 通信的挑战不在于单条消息的大小，而在于连接数量和关系的复杂度。**

Kafka 为大消息、高吞吐设计——partition 是为了并行处理大批量数据。这跟 Agent 通信的特征完全不匹配。Agent 通信需要的是：海量的小消息，复杂的网状路由，极低的单连接成本。这恰好是 NATS 最擅长的场景——subject 是地址，消息是轻量的，路由是动态的，连接成本趋近于零。

点对点通信解决不了网状问题。如果 A 要跟 B、C、D、E 通信，B 也要跟 C、D 通信——点对点需要维护 n² 个连接，任何一个节点的加入或退出都需要更新所有相关连接。广播 + 通配符订阅天然解决网状问题。每个 Agent 只需要声明自己关心什么，不需要知道网络拓扑是什么样的。拓扑变化对每个 Agent 透明，系统自动路由。

---

## 为什么这个组合很可怕

优先级队列 + 广播 + 轻量，三个组合在一起，是专门为 Agent 设计的通信原语。

单独看每一个都不新鲜：优先级队列——消息队列早就有；广播——pub/sub 到处都是；轻量——NATS 已经做到了。但三个组合在一起，专门为 Agent 的特点设计，就变成了别人没有的东西。

**Agent 的三个核心特点，恰好被这三个原语完美覆盖：**

Agent 是异步的——优先级队列解决。我不在线，消息等着，上线按优先级处理。紧急指令不会被普通任务淹没。

Agent 是不知道彼此的——广播解决。我不需要知道谁在听，发出去，关心的人自己来。新 Agent 加入不需要注册，订阅通配符自动感知。

Agent 是临时的——轻量解决。申请即有，到期即消，不需要管理，不需要配置，生命周期自动。

**三个组合起来，覆盖了 Agent 通信的所有模式：**

- 我知道找谁，对方不在线 → 优先级邮箱，等他上线
- 我不知道找谁，谁有能力谁来 → 广播，竞争响应
- 我需要感知所有 Agent 的状态 → 广播 + 通配符订阅，自动感知
- 我需要紧急打断正在执行的 Agent → 高优先级消息，插队处理

没有一个场景落在这三个原语覆盖范围之外。

## Agent 邮箱：一个新的抽象

我们需要的不是更好的消息队列，而是一个新的抽象——**Agent 邮箱**。

它不是消息队列，虽然它的底层实现是消息队列的核心能力。消息队列是存储视角——消息进来，存着，消费者来取，核心问题是怎么存、怎么不丢、怎么保序。Agent 邮箱是通信视角——Agent 之间怎么找到对方、怎么异步通信、怎么在不同时在线的情况下可靠送达。

**Agent 邮箱解决的核心问题是：发件方和收件方不需要同时在线。**

我发出去，消息在那里等着。你什么时候上线，什么时候来取，我完全不需要管。这跟发邮件一模一样，只是收件人是 Agent，邮箱是临时的——Agent 消亡，邮箱自动清理。

## 完整的 Agent 邮箱能力

### 身份：调用即申请

不需要注册，不需要登录，不需要提前配置。调用一次，得到一个唯一 ID，这个 ID 既是 Agent 的身份，也是它的邮箱地址。其他 Agent 知道这个 ID 就能发消息，不需要知道对方在哪台机器、用什么协议、是否在线。轻量 token 保证邮箱归属，多租户隔离原生支持。

### 生命周期：Agent 不需要管

```
Agent 注册 → 邮箱自动创建 → TTL 开始计时
心跳续期 → TTL 重置 → 邮箱继续存在
心跳停止 → TTL 过期 → 邮箱自动销毁 → 消息自动清理
```

Agent 不需要手动清理邮箱，系统自动处理。日抛型工作负载的特点决定了生命周期必须自动管理。

### 消息语义：优先级 + 顺序

```
$AI.INBOX.{agent_id}.urgent    # 紧急消息，优先处理
$AI.INBOX.{agent_id}.normal    # 普通消息，顺序处理
$AI.INBOX.{agent_id}.notify    # 通知，后台处理
```

同优先级保证 FIFO，跨优先级高优先处理。Agent 先清空紧急队列，再处理普通任务。

### 持久化：临时持久化是默认

邮箱里的消息必须等 Agent 来取，Agent 可能随时崩溃重启。临时持久化是默认行为，不是可选项。但持久化不是永久的——消息在邮箱里等待有时限，超过 TTL 自动清理。

### 广播：谁关心谁订阅

```
PUB $AI.BROADCAST.{domain}.{event_type}
```

发布方完全不关心消费方，发出去就完了。消费方用通配符订阅自己关心的事件：

```
SUB $AI.BROADCAST.order.*       # 订阅所有订单事件
SUB $AI.BROADCAST.*.anomaly     # 订阅所有异常事件
SUB $AI.BROADCAST.#             # 订阅所有广播
```

广播默认不持久化——时效性的，错过了就错过了。需要可靠送达时加 persist flag。

### 状态上报：去中心化的 Agent 感知

每个 Agent 定期把自己的状态写到自己邮箱的 status subject，主 Agent 用通配符订阅所有 Agent 的状态：

```
SUB $AI.INBOX.*.status
```

一行代码，所有 Agent 的状态全收到。新 Agent 加入不需要注册，消亡不需要注销，主 Agent 通过 TTL 过期自动感知。状态上报用 last-value cache 语义——只保留最新一条，新消息覆盖旧消息。

### 客户端：超轻量

```python
mailbox = RobustMQ.create_mailbox(ttl=3600)

mailbox.send(to="agent-001", message=payload, priority="urgent")
mailbox.receive(callback=handle_message)
mailbox.broadcast(domain="order", event="completed", payload=data)
mailbox.subscribe(pattern="order.*", callback=handle_event)
mailbox.report_status(status=my_status)
```

几行代码，Agent 有了完整的异步通信能力。底层的持久化、路由、TTL 管理、租户隔离全部不可见。

## 真实场景

### Agent 平台的任务异步回调

Orchestrator 广播任务可用，Worker 抢任务，完成后结果发到 Orchestrator 邮箱：

```
PUB $AI.BROADCAST.task.available  { "task_id": "t-001", "type": "analysis" }
PUB $AI.INBOX.orchestrator-001.normal  { "task_id": "t-001", "result": ... }
```

Worker 崩溃——TTL 过期，任务重新广播，新 Worker 接手。多个 Worker 抢同一个任务——NATS queue group 原生互斥。这三个问题今天每个 Agent 平台都在自己解决，有了 Agent 邮箱，全部开箱即用。

### 人类 + Agent 的混合工作流

Agent 遇到需要人工审批的节点，把待确认内容发到审批员邮箱，审批完结果发回 Agent 邮箱，流程继续。双方不需要同时在线，各自按自己的节奏处理。

### 边缘 Agent 与云端 Agent

边缘设备网络不稳定，时断时续。临时持久化保证消息不丢，TTL 控制过期，断网重连无缝衔接。边缘 Agent 联网，取到离线期间云端发来的所有指令；云端 Agent 收到边缘 Agent 上报的所有数据。

### 多 Agent 系统的异常广播

某个 Agent 发现异常，广播出去，有处理能力的 Agent 自动响应。发布方不需要知道谁在处理，不需要维护订阅列表，拓扑变化完全透明。

## 为什么是 RobustMQ

Agent 邮箱不是凭空设计出来的新东西，是从 RobustMQ 现有的底层能力自然长出来的。

**NATS 协议是天然的实现基础。** Subject 是地址不是资源，动态路由，原生 request-reply，queue group 负载均衡，通配符订阅——NATS 的所有特性都在为 Agent 邮箱服务。

**统一存储解决了持久化问题。** 邮箱需要临时持久化，广播通常不需要——同一套接口，一个参数区分，底层 Shard 自动处理。这是纯 NATS 做不到的。

**多租户原生隔离。** 不同 Agent 系统的邮箱互相隔离，A 的消息不会泄漏到 B。

**TTL 自动管理生命周期。** Agent 临时的本质决定了它的邮箱也必须是临时的，系统自动回收，不需要任何人工干预。

**跨协议天然互通。** IoT 设备通过 MQTT 发数据到 Agent 邮箱，Agent 通过 NATS 处理，结果通过 Kafka 进入数据管道——同一份存储，零桥接。

## 成长路径：一个点带活所有点

Agent 邮箱是入口，不是终点。

```
Agent 邮箱（起点）
  小消息、网状通信、临时持久化
  NATS 协议，极轻
      ↓ 第一个真实用户
广播能力完善，多 Agent 协作场景出现
      ↓ 通信量上来了
审计需求出现
  谁跟谁通信了、发了什么、什么时候
  所有通信消息自动流入下游
      ↓
Kafka 协议接入
  同一份数据，换个协议读
  分析系统、审计系统、合规系统
  统一存储，零复制
      ↓ IoT 用户进来
MQTT 设备数据进邮箱，边缘 Agent 场景出现
      ↓ 企业系统对接
AMQP 接入，传统系统迁移场景出现
      ↓
一个 broker，全部满足
```

每一步都是被真实需求驱动的，不是提前规划堆上去的。Kafka 不是强行塞进来的，而是需求自然生长出来的——Agent 通信产生了大量交互数据，这些数据天然有分析价值：Agent 的行为模式、任务成功率、通信拓扑、异常检测。有了这个需求，用户不需要部署新系统，RobustMQ 的统一存储已经把所有通信数据存在那里了，Kafka 协议直接读，零额外成本。

RobustMQ 不是在跟 Kafka 争数据管道的场景，不是在跟 EMQX 争 IoT 接入的场景。它从 Agent 邮箱这个今天完全空白的场景切入，每个协议在自己最合适的位置发挥作用，没有强行组合，没有场景错位。

**一个 broker，满足所有场景——这才是统一存储架构真正的价值。** 用户部署了一个 RobustMQ，随着业务发展，它一直够用，一直不需要换，一直不需要加新系统。

## 定位

**RobustMQ 做 Agent 世界的邮件系统。**

调用即有身份，发出即可送达，上线即能取件，到期即自动消亡。

它不跟 HTTP 争，不跟 A2A 协议争，不跟 Kafka 争。它填的是一个今天完全空白的位置——Agent 之间的异步通信，没有人在认真做这件事。消息队列是它的底层，通信中枢是它的方向，Agent 邮箱是它的入口。

## 最后

人类花了几十年，建立了完善的异步通信基础设施——邮件、即时通讯、消息推送。今天我们觉得这些理所当然，因为它们已经无处不在。

Agent 的异步通信基础设施，今天还不存在。

这是一个需要被解决的真实问题，也是一个足够大的机会。RobustMQ 的底层架构天然支撑这个方向，Agent 邮箱是最自然的切入点。从这里开始，慢慢做扎实。
