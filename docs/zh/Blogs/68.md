# $AI.* 协议设计文档

## 概述

`$AI.*` 是 RobustMQ 为 Agent 异步通信设计的 subject 命名空间约定。基于 NATS 协议，运行在 RobustMQ 统一存储层之上。

核心解决的问题是：**Agent 之间的异步通信，发件方和收件方不需要同时在线。**

这不是一个新协议，而是在 NATS subject 上定义的一套语义约定。Agent 看到 subject 就能理解消息的意图，不需要查文档，不需要额外配置。

---

## 场景概述

Agent 通信有三个本质特征：Agent 是异步的，上下线不可预测；Agent 是不知道彼此的，关系是网状的；Agent 是临时的，生命周期可能只有几秒。

围绕这三个特征，`$AI.*` 覆盖以下八个核心场景：

**场景一：子 Agent 完成任务，异步通知主 Agent**
主 Agent 分发任务后不阻塞等待，子 Agent 完成后发消息到主 Agent 邮箱，主 Agent 空了来取。

**场景二：主 Agent 感知所有子 Agent 状态**
每个子 Agent 定期上报自己的状态，主 Agent 通过通配符订阅自动感知所有 Agent 的上线、运行、消亡。

**场景三：任务广播，谁有能力谁来接**
主 Agent 广播任务，多个 Worker 竞争消费，抢到任务的 Worker 发确认和结果到主 Agent 邮箱。

**场景四：Agent 发现异常，广播告警**
某个 Agent 发现异常，广播事件，订阅了该域的 Agent 自动响应，发现方不需要知道谁在处理。

**场景五：边缘 Agent 离线消息积压**
云端 Agent 给边缘 Agent 发指令，边缘 Agent 不在线消息持久化等待，联网后按优先级取消息。

**场景六：人机混合工作流**
Agent 遇到需要人工判断的节点，发审批请求到人类审批员邮箱，审批员空了处理，结果发回 Agent 邮箱继续执行。

**场景七：Agent 请求另一个 Agent，等待响应**
Agent A 发请求到 Agent B 邮箱，B 不在线请求等着，B 上线处理完发响应到 A 邮箱，A 通过 correlation_id 匹配。

**场景八：Agent 注册能力，其他 Agent 发现它**
Agent 启动时把能力声明写入状态，其他 Agent 通过订阅状态或广播查询发现有该能力的 Agent。

## 场景说明

---

### 场景一：子 Agent 完成任务，异步通知主 Agent

**问题**

主 Agent 拆解任务分发给子 Agent 并行执行，子 Agent 完成后需要通知主 Agent。主 Agent 不能阻塞等待，需要异步处理其他事情，子 Agent 完成了再来取结果。

**流程**

```
主 Agent 创建邮箱，得到 agent_id
主 Agent 分发任务给子 Agent，告知 reply_to
子 Agent 处理任务
子 Agent 完成，发消息到主 Agent 邮箱
主 Agent 订阅邮箱，取到结果继续执行
```

**核心需求**：点对点，异步，临时持久化，主 Agent 不需要一直在线等。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| REQ | `$AI.MAILBOX.CREATE` | 主 Agent 申请邮箱，得到 agent_id |
| SUB | `$AI.INBOX.{主agent_id}.*` | 主 Agent 订阅自己邮箱 |
| PUB | `$AI.INBOX.{主agent_id}.normal` | 子 Agent 完成，发结果 |

三个操作，两个 subject，覆盖完整场景。主 Agent 不需要在线等，子 Agent 不需要知道主 Agent 在哪台机器。

---

### 场景二：主 Agent 感知所有子 Agent 状态

**问题**

Orchestrator 管理多个 Worker Agent，需要实时掌握每个 Worker 的状态。不能靠轮询，不能维护静态列表，新 Worker 加入要自动感知，Worker 消亡也要自动感知。

**流程**

```
Worker Agent 启动，定期上报状态
主 Agent 用通配符订阅所有 Agent 的状态
新 Worker 加入，主 Agent 自动收到其状态
Worker 超过 TTL 没有上报，状态过期，主 Agent 感知消亡
```

**核心需求**：一对多，主动上报，通配符订阅，去中心化感知，retain=latest。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.STATUS.{子agent_id}` | 子 Agent 定期上报状态，retain=latest，ttl=60 |
| SUB | `$AI.STATUS.*` | 主 Agent 订阅所有子 Agent 状态 |
| REQ | `$AI.STATUS.{子agent_id}` | 主 Agent 按需查询某个子 Agent 最新状态 |

两个 subject，主 Agent 一行订阅掌握全局。新 Agent 加入自动感知，TTL 过期自动感知消亡，不需要任何注册或注销操作。

---

### 场景三：任务广播，谁有能力谁来接

**问题**

主 Agent 有一批任务需要分发，但不知道哪些 Worker 可用，不想维护 Worker 列表。广播任务出去，有能力且空闲的 Worker 自己来抢，保证每个任务只被一个 Worker 处理。

**流程**

```
主 Agent 广播任务到 $AI.BROADCAST.task.available
Worker 订阅该 subject，queue group 保证竞争消费
抢到任务的 Worker 发确认到主 Agent 邮箱
Worker 完成，结果发到主 Agent 邮箱
```

**核心需求**：广播分发，竞争消费，queue group 互斥，结果回传邮箱。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.BROADCAST.task.available` | 主 Agent 广播任务 |
| SUB | `$AI.BROADCAST.task.available` queue_group | Worker 竞争消费，只有一个抢到 |
| PUB | `$AI.INBOX.{主agent_id}.normal` | Worker 发确认和结果到主 Agent 邮箱 |

三个操作，两个 subject。queue_group 是关键——保证同一个任务只被一个 Worker 处理，NATS 原生支持，不需要分布式锁。

---

### 场景四：Agent 发现异常，广播告警

**问题**

某个 Agent 发现了系统异常，需要通知所有关心这个事件的 Agent。但不知道谁需要被通知，也不需要等待响应。发出去，有能力处理的 Agent 自己响应。

**流程**

```
Agent 发现异常
广播到 $AI.BROADCAST.{domain}.{event}
订阅了该 domain 的 Agent 自动收到
各自按自己的逻辑响应
发现方不需要知道谁在处理
```

**核心需求**：事件广播，订阅方自决，发布方无感知，通配符多级订阅。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.BROADCAST.{domain}.{event}` | Agent 广播异常事件 |
| SUB | `$AI.BROADCAST.system.*` | 订阅某个域的所有事件 |
| SUB | `$AI.BROADCAST.*.anomaly` | 订阅所有域的异常事件 |
| SUB | `$AI.BROADCAST.#` | 订阅所有广播 |

一个 PUB，多种 SUB 模式。发布方只管发，订阅方自己决定关心什么粒度。通配符三级覆盖单域、跨域、全局三种订阅需求。

---

### 场景五：边缘 Agent 离线消息积压

**问题**

边缘设备网络不稳定，时断时续。云端 Agent 需要给边缘 Agent 发指令，边缘 Agent 可能不在线。指令不能丢，边缘 Agent 联网后要能取到所有离线消息，并按优先级处理。

**流程**

```
云端 Agent 发指令到边缘 Agent 邮箱，持久化存储
边缘 Agent 不在线，消息在邮箱等待
边缘 Agent 联网，订阅自己邮箱
按优先级取消息：urgent 先于 normal 先于 notify
处理完发响应到云端 Agent 邮箱
```

**核心需求**：离线消息，优先级队列，断网重连无缝衔接，不同 TTL 对应不同重要度。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.INBOX.{边缘agent_id}.urgent` | 云端发紧急指令，persist=true，ttl=86400 |
| PUB | `$AI.INBOX.{边缘agent_id}.normal` | 云端发普通数据，persist=true，ttl=3600 |
| SUB | `$AI.INBOX.{边缘agent_id}.*` | 边缘 Agent 联网后订阅，按优先级取消息 |
| PUB | `$AI.INBOX.{云端agent_id}.normal` | 边缘 Agent 上报遥测数据 |

同一个 INBOX subject，不同优先级后缀，不同 TTL。边缘 Agent 联网后 urgent 先于 normal 先于 notify，断网期间消息不丢。

---

### 场景六：人机混合工作流

**问题**

Agent 在自动化流程中遇到需要人工判断的节点。Agent 不能无限等待，人类审批员也不需要实时在线。双方通过邮箱异步协作，各自按自己的节奏处理。

**流程**

```
Agent 处理到人工节点
发审批请求到人类审批员邮箱，带上 reply_to 和 deadline
Agent 继续处理其他任务，不阻塞
审批员空了来处理
审批结果发回 Agent 邮箱
Agent 收到结果，继续执行流程
```

**核心需求**：人机异步协作，双向邮箱，correlation_id 配对，deadline 控制超时。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.INBOX.{审批员_id}.urgent` | Agent 发审批请求，带 reply_to 和 correlation_id |
| SUB | `$AI.INBOX.{审批员_id}.*` | 审批员订阅自己邮箱 |
| PUB | `$AI.INBOX.{agent_id}.urgent` | 审批员发结果到 reply_to 地址 |
| SUB | `$AI.INBOX.{agent_id}.*` | Agent 订阅自己邮箱，收到结果继续执行 |

人类审批员和 Agent 使用完全相同的协议——邮箱没有区别，不需要为人类单独设计接口。correlation_id 保证 Agent 能匹配哪个审批对应哪个请求。

### 场景七：Agent 请求另一个 Agent，等待响应

**问题**

Agent A 需要 Agent B 提供某个能力。这是 request-reply 模式，但不要求 B 必须在线。B 离线时请求在邮箱等着，B 上线处理完再回。异步的请求响应。

**流程**

```
Agent A 发请求到 Agent B 邮箱，携带 correlation_id 和 reply_to
Agent B 可能不在线，消息持久化等待
Agent B 上线，取到请求处理
Agent B 发响应到 Agent A 的 reply_to 地址
Agent A 通过 correlation_id 匹配请求和响应
```

**核心需求**：异步 request-reply，correlation_id 配对，B 不在线请求等待，超时降级。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.INBOX.{B的agent_id}.normal` | A 发请求到 B 邮箱，带 correlation_id 和 reply_to |
| SUB | `$AI.INBOX.{B的agent_id}.*` | B 订阅自己邮箱，取到请求处理 |
| PUB | `$AI.INBOX.{A的agent_id}.normal` | B 发响应到 A 的 reply_to 地址 |
| SUB | `$AI.INBOX.{A的agent_id}.*` | A 订阅自己邮箱，通过 correlation_id 匹配响应 |

两个邮箱，四个操作，完整的异步 request-reply。跟 HTTP 的本质区别：B 不在线请求不会失败，消息在邮箱等着。A 也不会阻塞，继续处理其他事情，响应来了再匹配。

---

### 场景八：Agent 注册能力，其他 Agent 发现它

**问题**

一个复杂 Agent 系统里，各种专业 Agent 提供不同的能力。Agent A 需要某个能力时，怎么知道谁能提供？不能靠硬编码，不能靠中心化的注册服务，需要去中心化的能力发现机制。

**流程**

```
Agent 启动，把能力声明写入 $AI.STATUS.{agent_id}
其他 Agent 需要某个能力时：
  方式一：订阅 $AI.STATUS.*，本地过滤有该能力的 Agent
  方式二：广播能力查询，有能力的 Agent 响应到发现方邮箱
找到提供能力的 Agent 后，衔接场景七发起请求
```

**核心需求**：去中心化能力发现，复用 STATUS subject，两种发现方式按场景选择。

**协议组合**

方式一：被动发现（已在监听状态的 Agent）

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.STATUS.{agent_id}` | Agent 上报状态，capabilities 字段包含能力声明 |
| SUB | `$AI.STATUS.*` | 发现方订阅所有状态，本地过滤有目标能力的 Agent |

方式二：主动发现（临时需要某个能力）

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$AI.BROADCAST.capability.query` | 发现方广播能力查询，带 reply_to |
| SUB | `$AI.BROADCAST.capability.query` | 有能力的 Agent 订阅，空闲时响应 |
| PUB | `$AI.INBOX.{发现方_id}.normal` | 有能力的 Agent 响应到发现方邮箱 |

发现之后衔接场景七，向目标 Agent 邮箱发请求。能力声明复用 STATUS subject，不引入新 subject，协议空间保持最小。

## 协议设计

---

### Subject 空间

```
$AI.
  ├── MAILBOX.
  │     └── CREATE                    # 申请邮箱，返回 agent_id
  │
  ├── INBOX.
  │     └── {agent_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  ├── STATUS.
  │     └── {agent_id}               # 状态上报 + 能力声明
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

---

### 基础概念

**agent_id**：每个 Agent 的唯一身份标识，也是它的邮箱地址。通过 `$AI.MAILBOX.CREATE` 申请，系统返回全局唯一 ID。调用即申请，不需要注册。

**TTL**：每个邮箱和消息都有 TTL，跟 Agent 的生命周期绑定。Agent 活着心跳续期，Agent 消亡 TTL 过期自动清理。

**persist**：消息的持久化策略。`$AI.INBOX.*` 默认持久化，`$AI.BROADCAST.*` 默认不持久化，可显式覆盖。

**priority**：邮箱支持三个优先级：urgent、normal、notify。同优先级保证 FIFO，跨优先级高优先处理。

**token**：轻量身份验证。申请邮箱时返回 token，发消息时携带，保证邮箱归属，不做复杂权限控制。

**retain=latest**：每个 subject 只保留最新一条消息，新消息覆盖旧消息。用于状态上报场景。

---

### $AI.MAILBOX.CREATE — 申请邮箱

```bash
# 请求
REQ $AI.MAILBOX.CREATE
{
  "ttl": 3600                          # 邮箱生存时间（秒）
}

# 响应
{
  "agent_id": "agt-uuid-001",          # 全局唯一 ID
  "token": "tok-xxx",                  # 身份 token
  "inbox": "$AI.INBOX.agt-uuid-001"    # 邮箱地址
}
```

---

### $AI.INBOX.{agent_id}.{priority} — 点对点邮箱

```bash
# 发送消息到邮箱
PUB $AI.INBOX.{agent_id}.{priority}
{
  "from": "agt-uuid-002",              # 发件方，必填
  "type": "task_result",              # 消息类型，必填
  "correlation_id": "req-uuid-001",   # 请求响应配对，按需填写
  "reply_to": "$AI.INBOX.agt-uuid-A.normal",  # 回复地址，按需填写
  "payload": { ... },                 # 业务数据
  "ts": 1234567890                    # 发送时间戳（毫秒），必填
}
persist=true   ttl=3600

# 订阅邮箱（订阅所有优先级）
SUB $AI.INBOX.{agent_id}.*

# 订阅邮箱（只订阅紧急消息）
SUB $AI.INBOX.{agent_id}.urgent
```

**priority 取值**

| 值 | 说明 | 默认持久化 | 建议 TTL |
|----|------|-----------|---------|
| urgent | 紧急消息，优先处理 | true | 86400s |
| normal | 普通消息，顺序处理 | true | 3600s |
| notify | 通知，后台处理 | false | — |

---

### $AI.STATUS.{agent_id} — 状态上报与能力声明

```bash
# 上报状态（含能力声明）
PUB $AI.STATUS.{agent_id}
{
  "agent_id": "agt-uuid-002",
  "status": "running",                 # online | running | idle | offline
  "task": "analysis-001",             # 当前任务，无则 null
  "load": 0.6,                        # 负载 0-1
  "capabilities": [
    {
      "name": "data.analysis",
      "version": "1.0",
      "description": "数据分析能力",
      "input_schema": { ... },
      "output_schema": { ... }
    }
  ],
  "ts": 1234567890
}
retain=latest   ttl=60

# 订阅所有 Agent 状态
SUB $AI.STATUS.*

# 查询某个 Agent 最新状态
REQ $AI.STATUS.{agent_id}
→ RESP { 最新状态 }
```

---

### $AI.BROADCAST.{domain}.{event} — 事件广播

```bash
# 发布广播事件
PUB $AI.BROADCAST.{domain}.{event}
{
  "from": "agt-uuid-004",
  "type": "event_type",
  "severity": "high",                  # low | medium | high | critical，按需填写
  "payload": { ... },
  "ts": 1234567890
}
persist=false   # 默认不持久化，重要广播显式设置 persist=true

# 订阅单个域的所有事件
SUB $AI.BROADCAST.system.*

# 订阅所有域的特定事件
SUB $AI.BROADCAST.*.anomaly

# 订阅所有广播
SUB $AI.BROADCAST.#

# 竞争消费（同一条消息只有一个订阅者收到）
SUB $AI.BROADCAST.task.available   queue_group=task.workers
```

**domain 建议命名**

| domain | 说明 |
|--------|------|
| system | 系统级事件，如异常、重启、扩缩容 |
| task | 任务相关，如 available、completed、failed |
| capability | 能力发现，如 query、response |
| data | 数据变更事件 |
| {自定义} | 业务域自定义 |

---

### 消息通用字段规范

所有 `$AI.*` 消息建议包含以下通用字段：

```json
{
  "from": "agent_id",           # 发件方，必填
  "type": "message_type",       # 消息类型，必填
  "ts": 1234567890,             # 发送时间戳（毫秒），必填
  "correlation_id": "uuid",     # 请求响应配对，request/response 必填
  "reply_to": "$AI.INBOX...",   # 回复地址，需要响应时填写
  "deadline": 1234567890,       # 期望处理截止时间，按需填写
  "payload": { ... }            # 业务数据，按需填写
}
```

**type 建议取值**

| type | 说明 | 适用场景 |
|------|------|---------|
| task_result | 任务结果 | 场景一、三 |
| request | 请求 | 场景七 |
| response | 响应 | 场景七 |
| approval_request | 审批请求 | 场景六 |
| approval_result | 审批结果 | 场景六 |
| task_accepted | 任务确认 | 场景三 |
| capability_response | 能力响应 | 场景八 |
| command | 指令 | 场景五 |
| telemetry | 遥测数据 | 场景五 |

## 协议总览表格

| Subject | 方向 | 默认持久化 | TTL | 特殊参数 | 适用场景 |
|---------|------|-----------|-----|---------|---------|
| `$AI.MAILBOX.CREATE` | REQ/RESP | — | — | — | 所有场景初始化 |
| `$AI.INBOX.{id}.urgent` | PUB/SUB | true | 86400s | correlation_id, reply_to | 一、五、六、七 |
| `$AI.INBOX.{id}.normal` | PUB/SUB | true | 3600s | correlation_id, reply_to | 一、三、七、八 |
| `$AI.INBOX.{id}.notify` | PUB/SUB | false | — | — | 一 |
| `$AI.STATUS.{id}` | PUB/SUB/REQ | true | 60s | retain=latest | 二、八 |
| `$AI.BROADCAST.task.available` | PUB/SUB | false | 300s | queue_group, reply_to | 三 |

## 设计原则

**语义即地址**：看到 subject 就能理解消息的意图，不需要查文档。`$AI.INBOX.*` 是邮箱，`$AI.STATUS.*` 是状态，`$AI.BROADCAST.*` 是广播，层次清晰，不混用。

**最小化 subject 空间**：四个顶层 subject 覆盖八个场景，没有冗余，没有重叠。新增场景优先复用现有 subject，不轻易扩展命名空间。

**持久化按需选择**：不强制所有消息持久化，根据消息语义和业务重要性，由发送方显式声明。

**去中心化**：没有中心化的注册服务，没有中心化的路由表。Agent 通过 subject 直接寻址，通过广播和状态上报实现去中心化的服务发现。

**向后兼容**：`$AI.*` 的 subject 语义一旦发布不轻易修改。新增能力通过新增 subject 实现，不修改已有 subject 的语义。
