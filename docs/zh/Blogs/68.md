# $AI.* 协议设计文档

## 概述

`$AI.*` 是 RobustMQ 为 Agent 异步通信设计的 subject 命名空间约定。基于 NATS 协议，运行在 RobustMQ 统一存储层之上。

核心解决的问题是：**Agent 之间的异步通信，发件方和收件方不需要同时在线。**

这不是一个新协议，而是在 NATS subject 上定义的一套语义约定。Agent 看到 subject 就能理解消息的意图，不需要查文档，不需要额外配置。

---

## Subject 空间总览

```
$AI.
  ├── MAILBOX.
  │     └── CREATE                    # 申请邮箱，返回 agent_id
  │
  ├── INBOX.
  │     └── {agent_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  ├── STATUS.
  │     └── {agent_id}               # 状态上报 + 能力声明
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

---

## 基础概念

### agent_id

每个 Agent 的唯一身份标识，也是它的邮箱地址。通过 `$AI.MAILBOX.CREATE` 申请，系统返回一个全局唯一的 ID。Agent 不需要注册，不需要登录，调用即申请。

### TTL

每个邮箱和消息都有 TTL。TTL 跟 Agent 的生命周期绑定——Agent 活着心跳续期，Agent 消亡 TTL 过期自动清理。Agent 不需要手动管理资源。

### persist

消息的持久化策略。默认根据 subject 类型决定：`$AI.INBOX.*` 默认持久化，`$AI.BROADCAST.*` 默认不持久化。可以通过参数显式覆盖。

### priority

邮箱支持三个优先级：`urgent`、`normal`、`notify`。同优先级保证 FIFO，跨优先级高优先处理。

### token

轻量身份验证。申请邮箱时返回 token，发消息时携带，系统验证 token 合法性保证邮箱归属。不做复杂权限控制，只确认"你是你声称的那个 Agent"。

## 八个场景及协议设计

---

### 场景一：子 Agent 完成任务，异步通知主 Agent

**问题描述**

主 Agent 拆解任务分发给子 Agent 并行执行，子 Agent 完成后需要通知主 Agent。主 Agent 不能一直阻塞等待，需要异步处理其他事情，子 Agent 完成了再来取结果。

**使用流程**

```
主 Agent 创建邮箱 → 得到 agent_id
主 Agent 分发任务给子 Agent，告知 reply_to
子 Agent 处理任务
子 Agent 完成，发消息到主 Agent 邮箱
主 Agent 空了订阅邮箱，取到结果继续执行
```

**协议设计**

```bash
# Step 1：主 Agent 创建邮箱
REQ $AI.MAILBOX.CREATE
{
  "ttl": 3600
}
→ RESP
{
  "agent_id": "agt-uuid-001",
  "token": "tok-xxx"
}

# Step 2：子 Agent 完成，发结果到主 Agent 邮箱
PUB $AI.INBOX.agt-uuid-001.normal
{
  "from": "agt-uuid-002",
  "task_id": "t-001",
  "type": "task_result",
  "result": { ... },
  "ts": 1234567890
}

# Step 3：主 Agent 订阅自己邮箱
SUB $AI.INBOX.agt-uuid-001.*
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `from` | 发件方 agent_id，收件方可以知道是谁发的 |
| `task_id` | 任务标识，主 Agent 用来匹配哪个任务完成了 |
| `type` | 消息类型，`task_result` 表示这是任务结果 |
| persist | 默认 true，子 Agent 发消息时主 Agent 可能不在线 |
| ttl | 跟主 Agent 邮箱 TTL 一致，主 Agent 销毁消息自动清理 |

---

### 场景二：主 Agent 感知所有子 Agent 状态

**问题描述**

一个 Orchestrator 管理多个 Worker Agent，需要实时掌握每个 Worker 的状态——是否在线、当前负载、正在执行什么任务。不能靠轮询，不能维护静态列表，新 Worker 加入要自动感知，Worker 消亡也要自动感知。

**使用流程**

```
Worker Agent 启动 → 定期上报状态到 $AI.STATUS.{agent_id}
主 Agent 用通配符订阅 $AI.STATUS.*
新 Worker 加入 → 主 Agent 自动收到其状态
Worker 超过 TTL 没有上报 → 状态过期 → 主 Agent 感知消亡
```

**协议设计**

```bash
# Worker Agent 定期上报状态（心跳间隔建议 30s）
PUB $AI.STATUS.agt-uuid-002
{
  "agent_id": "agt-uuid-002",
  "status": "running",          # online | running | idle | offline
  "task": "analysis-001",       # 当前执行的任务，没有则为 null
  "load": 0.6,                  # 负载 0-1
  "capabilities": ["data.analysis", "data.visualization"],
  "ts": 1234567890
}
retain=latest  ttl=60

# 主 Agent 订阅所有 Worker 状态
SUB $AI.STATUS.*

# 主 Agent 查询某个 Worker 的最新状态
REQ $AI.STATUS.agt-uuid-002
→ RESP { 最新状态 }
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `retain=latest` | 每个 subject 只保留最新一条，新消息覆盖旧消息 |
| `ttl=60` | 60 秒没有上报，状态自动过期，主 Agent 感知消亡 |
| `status` | online=刚上线，running=执行中，idle=空闲，offline=即将下线 |
| `capabilities` | 该 Agent 能提供的能力列表，供场景八的能力发现使用 |

**注意**：新订阅时立刻收到每个 Worker 的最新状态，不需要等下一次上报。这是 `retain=latest` 的作用。

### 场景三：任务广播，谁有能力谁来接

**问题描述**

主 Agent 有一批任务需要分发，但不知道哪些 Worker 可用，也不想维护 Worker 列表。广播任务出去，有能力且空闲的 Worker 自己来抢，保证每个任务只被一个 Worker 处理。

**使用流程**

```
主 Agent 广播任务到 $AI.BROADCAST.task.available
Worker 订阅该 subject，queue group 保证竞争消费
抢到任务的 Worker 发确认到主 Agent 邮箱
Worker 完成，结果发到主 Agent 邮箱
```

**协议设计**

```bash
# 主 Agent 广播任务
PUB $AI.BROADCAST.task.available
{
  "task_id": "t-001",
  "type": "analysis",
  "priority": "normal",
  "payload": { ... },
  "reply_to": "$AI.INBOX.agt-uuid-001.normal",
  "deadline": 1234567890,
  "ttl": 300
}

# Worker 订阅任务广播（queue group 保证只有一个 Worker 抢到）
SUB $AI.BROADCAST.task.available  queue_group=task.workers

# Worker 抢到任务，发确认到主 Agent 邮箱
PUB $AI.INBOX.agt-uuid-001.normal
{
  "from": "agt-uuid-003",
  "type": "task_accepted",
  "task_id": "t-001",
  "ts": 1234567890
}

# Worker 完成，发结果到主 Agent 邮箱
PUB $AI.INBOX.agt-uuid-001.normal
{
  "from": "agt-uuid-003",
  "type": "task_result",
  "task_id": "t-001",
  "result": { ... },
  "ts": 1234567890
}
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `queue_group` | NATS 原生竞争消费，同一条消息只有一个订阅者收到 |
| `reply_to` | 告知 Worker 结果发到哪个邮箱 |
| `deadline` | 任务截止时间，Worker 超时不处理主 Agent 可以重新广播 |
| `ttl` | 任务广播的存活时间，超时没有 Worker 接手自动失效 |

### 场景四：Agent 发现异常，广播告警

**问题描述**

某个 Agent 发现了系统异常——内存泄漏、服务不可用、数据异常。它需要通知所有关心这个事件的 Agent，但不知道谁需要被通知，也不需要等待响应。发出去，有能力处理的 Agent 自己响应。

**使用流程**

```
Agent 发现异常
广播到 $AI.BROADCAST.{domain}.{event}
订阅了该 domain 的 Agent 自动收到
各自按自己的逻辑响应
发现方不需要知道谁在处理
```

**协议设计**

```bash
# Agent 广播异常告警
PUB $AI.BROADCAST.system.anomaly
{
  "from": "agt-uuid-004",
  "severity": "high",           # low | medium | high | critical
  "type": "memory_leak",
  "affected": "worker-cluster-B",
  "detail": { ... },
  "ts": 1234567890
}
persist=true  ttl=3600          # 高严重度告警需要持久化

# 订阅所有系统级事件
SUB $AI.BROADCAST.system.*

# 订阅所有域的异常事件
SUB $AI.BROADCAST.*.anomaly

# 订阅所有广播（慎用，流量大）
SUB $AI.BROADCAST.#
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `domain` | 业务域，如 system、order、payment、device |
| `event` | 事件类型，如 anomaly、completed、failed、updated |
| `severity` | 严重程度，消费方可以按严重程度过滤处理 |
| `persist` | 高严重度告警建议 true，保证离线 Agent 上线后也能收到 |

**通配符规则**

```
$AI.BROADCAST.system.*     # 所有系统事件
$AI.BROADCAST.*.anomaly    # 所有域的异常事件
$AI.BROADCAST.#            # 所有广播（所有域所有事件）
```

### 场景五：边缘 Agent 离线消息积压

**问题描述**

边缘设备网络不稳定，时断时续。云端 Agent 需要给边缘 Agent 发指令，但边缘 Agent 可能不在线。指令不能丢，边缘 Agent 联网后要能取到所有离线期间的消息，并按优先级处理。

**使用流程**

```
云端 Agent 发指令到边缘 Agent 邮箱，持久化存储
边缘 Agent 不在线，消息在邮箱等待
边缘 Agent 联网，订阅自己邮箱
按优先级取消息：urgent 先于 normal 先于 notify
处理完发响应到云端 Agent 邮箱
```

**协议设计**

```bash
# 云端 Agent 发紧急指令（高优先级，持久化等待 24h）
PUB $AI.INBOX.edge-agt-007.urgent
{
  "from": "cloud-agt-001",
  "type": "command",
  "command": "update_config",
  "payload": { "config": { ... } },
  "reply_to": "$AI.INBOX.cloud-agt-001.normal",
  "ts": 1234567890
}
persist=true  ttl=86400

# 云端 Agent 发普通数据同步（持久化等待 1h）
PUB $AI.INBOX.edge-agt-007.normal
{
  "from": "cloud-agt-001",
  "type": "data_sync",
  "payload": { ... },
  "ts": 1234567890
}
persist=true  ttl=3600

# 边缘 Agent 联网，订阅自己邮箱
SUB $AI.INBOX.edge-agt-007.*

# 边缘 Agent 上报遥测数据到云端邮箱
PUB $AI.INBOX.cloud-agt-001.normal
{
  "from": "edge-agt-007",
  "type": "telemetry",
  "data": { ... },
  "ts": 1234567890
}
persist=true  ttl=3600
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `persist=true` | 必须显式开启，保证消息在边缘 Agent 离线期间不丢 |
| `ttl` | 不同优先级设置不同 TTL，紧急指令等待更久 |
| 优先级顺序 | 联网后先处理 urgent，再处理 normal，最后 notify |

**与 MQTT 的结合**

边缘场景下，IoT 设备本身通过 MQTT 协议接入 RobustMQ。MQTT 消息和 `$AI.INBOX.*` 消息共享同一套统一存储，零桥接，边缘设备数据直接流入 Agent 邮箱。

### 场景六：人机混合工作流

**问题描述**

Agent 在自动化流程中遇到需要人工判断的节点——合同金额超过阈值、模糊情况无法自动决策、合规需要人工审核。Agent 不能无限等待，人类审批员也不需要实时在线。双方通过邮箱异步协作，各自按自己的节奏处理。

**使用流程**

```
Agent 处理到人工节点
发审批请求到人类审批员邮箱，带上 reply_to 和 deadline
Agent 继续处理其他任务，不阻塞
审批员空了来处理（可能几分钟，可能几小时）
审批结果发回 Agent 邮箱
Agent 收到结果，继续执行流程
```

**协议设计**

```bash
# Agent 发审批请求到人类审批员邮箱
PUB $AI.INBOX.human-approver-001.urgent
{
  "from": "contract-agt-001",
  "type": "approval_request",
  "correlation_id": "req-uuid-001",
  "reply_to": "$AI.INBOX.contract-agt-001.urgent",
  "content": "合同金额超过阈值，需要人工审批",
  "context": {
    "contract_id": "c-xxx",
    "amount": 500000,
    "threshold": 100000,
    "detail": { ... }
  },
  "deadline": 1234567890,
  "ts": 1234567890
}
persist=true  ttl=86400

# 审批员处理完，发结果到 Agent 邮箱
PUB $AI.INBOX.contract-agt-001.urgent
{
  "from": "human-approver-001",
  "type": "approval_result",
  "correlation_id": "req-uuid-001",
  "approved": true,
  "comment": "金额合理，批准",
  "ts": 1234567890
}

# Agent 订阅自己邮箱，收到结果继续执行
SUB $AI.INBOX.contract-agt-001.*
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `correlation_id` | 请求响应配对标识，Agent 可能有多个并发审批请求 |
| `reply_to` | 审批员不需要知道 Agent 的任何细节，只需要发到这个地址 |
| `deadline` | 审批截止时间，超时 Agent 走默认逻辑（自动拒绝或升级） |
| `human-approver-001` | 人类审批员也有邮箱，跟 Agent 邮箱完全一致，协议无区别 |

### 场景七：Agent 请求另一个 Agent，等待响应

**问题描述**

Agent A 需要 Agent B 提供某个能力——数据分析、内容生成、外部 API 调用。这是一个 request-reply 模式，但跟 HTTP 不同：不要求 B 必须在线，B 离线时请求在邮箱等着，B 上线处理完再回。异步的请求响应。

**使用流程**

```
Agent A 发请求到 Agent B 邮箱
  携带 correlation_id 和 reply_to
Agent B 可能不在线，消息持久化等待
Agent B 上线，取到请求，处理
Agent B 发响应到 Agent A 的 reply_to 地址
Agent A 通过 correlation_id 匹配请求和响应
```

**协议设计**

```bash
# Agent A 发请求到 Agent B 邮箱
PUB $AI.INBOX.agt-uuid-B.normal
{
  "from": "agt-uuid-A",
  "type": "request",
  "correlation_id": "req-uuid-001",
  "reply_to": "$AI.INBOX.agt-uuid-A.normal",
  "capability": "data.analysis",
  "payload": {
    "dataset": "...",
    "config": { ... }
  },
  "timeout": 3600,
  "ts": 1234567890
}
persist=true  ttl=3600

# Agent B 处理完，发响应到 Agent A 邮箱
PUB $AI.INBOX.agt-uuid-A.normal
{
  "from": "agt-uuid-B",
  "type": "response",
  "correlation_id": "req-uuid-001",
  "status": "success",           # success | failed | timeout
  "result": { ... },
  "ts": 1234567890
}

# Agent A 订阅自己邮箱，通过 correlation_id 匹配响应
SUB $AI.INBOX.agt-uuid-A.*
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `correlation_id` | 请求响应配对的关键，Agent A 可能同时发出多个请求 |
| `reply_to` | Agent B 不需要知道 A 的任何信息，只需要发到这个地址 |
| `timeout` | Agent A 期望多久内得到响应，超时走降级逻辑 |
| `capability` | 请求的能力标识，Agent B 根据这个决定如何处理 |
| `status` | 响应状态，failed 时 result 里包含错误信息 |

**与 HTTP request-reply 的核心区别**

HTTP 要求 B 必须在线，否则立刻报错。`$AI.*` 的 request-reply 是异步的——B 不在线请求在邮箱等着，B 上线处理完再回。这在 Agent 频繁上下线的场景里是根本性的差异。

### 场景八：Agent 注册能力，其他 Agent 发现它

**问题描述**

一个复杂的 Agent 系统里，各种专业 Agent 提供不同的能力——数据分析、内容生成、代码执行、外部 API 调用。Agent A 需要某个能力时，怎么知道谁能提供？不能靠硬编码，不能靠中心化的注册服务，需要去中心化的能力发现机制。

**使用流程**

```
Agent 启动，把自己的能力声明写入 $AI.STATUS.{agent_id}
其他 Agent 需要某个能力时有两种方式发现：
  方式一：订阅 $AI.STATUS.*，本地过滤有该能力的 Agent
  方式二：广播能力查询，有能力的 Agent 响应到发现方邮箱
找到提供能力的 Agent 后，衔接场景七发起请求
```

**协议设计**

```bash
# Agent 注册能力（复用 STATUS subject，状态里包含能力声明）
PUB $AI.STATUS.agt-uuid-B
{
  "agent_id": "agt-uuid-B",
  "status": "idle",
  "capabilities": [
    {
      "name": "data.analysis",
      "version": "1.0",
      "description": "数据分析能力",
      "input_schema": { ... },
      "output_schema": { ... }
    },
    {
      "name": "data.visualization",
      "version": "1.0",
      "description": "数据可视化能力"
    }
  ],
  "load": 0.3,
  "ts": 1234567890
}
retain=latest  ttl=60

# 方式一：订阅所有 Agent 状态，本地过滤
SUB $AI.STATUS.*
# 本地过滤 capabilities 包含 data.analysis 的 Agent

# 方式二：广播能力查询
PUB $AI.BROADCAST.capability.query
{
  "from": "agt-uuid-A",
  "capability": "data.analysis",
  "reply_to": "$AI.INBOX.agt-uuid-A.normal",
  "ts": 1234567890
}

# 有该能力的 Agent 响应（只有空闲的才响应，避免过载）
PUB $AI.INBOX.agt-uuid-A.normal
{
  "from": "agt-uuid-B",
  "type": "capability_response",
  "capability": "data.analysis",
  "load": 0.3,
  "available": true,
  "ts": 1234567890
}

# 发现 Agent B 有能力后，衔接场景七发起请求
PUB $AI.INBOX.agt-uuid-B.normal
{
  "from": "agt-uuid-A",
  "type": "request",
  "correlation_id": "req-uuid-002",
  "reply_to": "$AI.INBOX.agt-uuid-A.normal",
  "capability": "data.analysis",
  "payload": { ... }
}
```

**关键参数**

| 参数 | 说明 |
|------|------|
| `capabilities` | 能力列表，每个能力包含名称、版本、输入输出 schema |
| `load` | 当前负载，发现方可以选择负载低的 Agent |
| `available` | 是否可以接受新任务，负载高时可以返回 false |
| 方式一 vs 方式二 | 方式一适合已经在监听状态的主 Agent；方式二适合临时需要某个能力的场景 |

## 消息通用字段规范

所有 `$AI.*` 消息都建议包含以下通用字段：

```json
{
  "from": "agent_id",          # 发件方，必填
  "type": "message_type",      # 消息类型，必填
  "ts": 1234567890,            # 发送时间戳（毫秒），必填
  "correlation_id": "uuid",    # 请求响应配对，request/response 必填
  "reply_to": "$AI.INBOX...",  # 回复地址，需要响应时填写
  "ttl": 3600,                 # 消息存活时间（秒），可选
  "payload": { ... }           # 业务数据，可选
}
```

## 持久化策略总结

| Subject | 默认持久化 | 建议 TTL | 说明 |
|---------|-----------|---------|------|
| `$AI.INBOX.*.urgent` | true | 86400s | 紧急消息必须持久化 |
| `$AI.INBOX.*.normal` | true | 3600s | 普通消息持久化等待 |
| `$AI.INBOX.*.notify` | false | — | 通知类消息可以丢失 |
| `$AI.STATUS.*` | true | 60s | retain=latest，只保留最新 |
| `$AI.BROADCAST.*` | false | — | 广播默认不持久化 |
| `$AI.BROADCAST.*` persist=true | true | 自定义 | 重要广播显式开启 |

---

## 八个场景与 Subject 对应关系

```
场景一  子 Agent 通知主 Agent       →  $AI.INBOX.{id}.normal
场景二  主 Agent 感知所有状态       →  $AI.STATUS.*
场景三  任务广播竞争消费            →  $AI.BROADCAST.task.available
场景四  异常告警广播                →  $AI.BROADCAST.{domain}.anomaly
场景五  边缘离线消息积压            →  $AI.INBOX.{id}.urgent  persist=true
场景六  人机混合审批流              →  $AI.INBOX.{id}.urgent  + reply_to + correlation_id
场景七  Agent 请求等待响应          →  $AI.INBOX.{id}.normal  + correlation_id
场景八  能力注册与发现              →  $AI.STATUS.{id}  +  $AI.BROADCAST.capability.query
```

---

## 设计原则

**语义即地址。** 看到 subject 就能理解消息的意图，不需要查文档。`$AI.INBOX.*` 是邮箱，`$AI.STATUS.*` 是状态，`$AI.BROADCAST.*` 是广播，层次清晰，不混用。

**最小化 subject 空间。** 四个顶层 subject 覆盖八个场景，没有冗余，没有重叠。新增场景优先考虑复用现有 subject，不轻易扩展命名空间。

**持久化按需选择。** 不强制所有消息持久化，不强制所有消息不持久化。根据消息的语义和业务重要性，由发送方显式声明。

**去中心化。** 没有中心化的注册服务，没有中心化的路由表。Agent 通过 subject 直接寻址，通过广播和状态上报实现去中心化的服务发现。

**向后兼容。** `$AI.*` 的 subject 语义一旦发布不轻易修改。新增能力通过新增 subject 实现，不修改已有 subject 的语义。
