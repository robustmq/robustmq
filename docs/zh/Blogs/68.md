# $mq9.AI.* 协议设计文档

## 概述

`$mq9.AI.*` 是 RobustMQ 为 Agent 异步通信设计的 subject 命名空间约定。基于 NATS 协议，运行在 RobustMQ 统一存储层之上。

核心解决的问题是：**Agent 之间的异步通信，发件方和收件方不需要同时在线。**

这不是一个新协议，而是在 NATS subject 上定义的一套语义约定。所有 NATS 客户端 SDK 直接可用，零生态成本。Agent 看到 subject 就能理解消息的意图，不需要查文档，不需要额外配置。

### 与 NATS 生态的定位

mq9 在 NATS 生态中的定位是介于 Core pub/sub 和 JetStream 之间的中间态。Core NATS 太轻——没有持久化，离线就丢，做不了邮箱。JetStream 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义，为 Agent 发个消息不需要这些。mq9 在 pub/sub 的基础上加了可选持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。刚好够用，不多不少。和 JetStream 还有一个本质区别：JetStream 是通用持久化层，什么场景都想覆盖；mq9 不是通用的，它只为 Agent 通信这一个场景设计命令字和功能。做的少，反而做得准。

### 邮箱比喻

mq9 的核心概念来自现实世界的邮件系统：

- **开通邮箱** — 去邮局注册，拿到地址。→ `MAILBOX.CREATE`
- **寄信** — 知道对方地址，写好投进去。对方在不在家不管，信在邮箱等着。分急件、普通件、广告。→ `INBOX.{mail_id}.{priority}`
- **收信** — 回家打开邮箱，信自动送到手上。急件先看，普通件后看，广告最后。→ `nats sub`（在线时服务端推送）
- **去邮局查信** — 怕有遗漏，主动去问"我还有没有信没取"。→ `MAILBOX.QUERY.{mail_id}`（兜底拉取）
- **回信** — 信里有对方回信地址，写好寄回去。→ NATS 原生 reply-to
- **邮箱过期** — 长期没人用，邮局自动注销，信跟着一起清理。→ TTL 自动销毁
- **小区广播** — 公告栏，贴上去所有人能看。不需要知道谁会看，想看的自己来。→ `BROADCAST.{domain}.{event}`
- **只看某类公告** — 我只关心安全类公告。→ 通配符订阅
- **抢活** — 公告栏贴了一个任务，好几个人都想接，但只能一个人干。→ queue group

现实邮箱能做的事，mq9 全覆盖。现实邮箱做不到的事（优先级分级、通配符订阅、竞争消费），mq9 也有。

### 关键设计决策

- **mail_id 不绑定 Agent 身份。** mq9 只认 mail_id，不认 agent_id。一个 mail_id 就是一个通信通道，谁创建的、给谁用的、用来干什么的，mq9 不关心。一个 Agent 可以为不同任务申请不同的 mail_id，用完不管，TTL 自动清理。
- **邮箱不需要显式删除。** 纯 TTL 管理生命周期，客户端语义越简单越好。
- **ACK 复用 NATS 原生 reply-to。** 不创建新概念。
- **订阅复用 NATS 原生 sub 语义。** 普通订阅所有订阅者都收到，加 queue group 就是竞争消费。mq9 不定义订阅模式。
- **INBOX 是私人邮箱，BROADCAST 是公共广播。** 两种不同的通信模式：知道找谁用 INBOX（需要 CREATE 拿到 mail_id），不知道找谁用 BROADCAST（不需要任何前置操作，直接发直接订阅）。
- **消息流程：先存储后推送。** 消息到达先写存储层，再尝试推送给在线订阅者。推送成功走实时路径，推送失败消息在存储里等着，Agent 可通过 QUERY 兜底拉取。

## 场景概述

Agent 通信有三个本质特征：Agent 是异步的，上下线不可预测；Agent 是不知道彼此的，关系是网状的；Agent 是临时的，生命周期可能只有几秒。

围绕这三个特征，`$mq9.AI.*` 覆盖以下八个核心场景：

**场景一：子 Agent 完成任务，异步通知主 Agent**
主 Agent 分发任务后不阻塞等待，子 Agent 完成后发消息到主 Agent 邮箱，主 Agent 空了来取。

**场景二：主 Agent 感知所有子 Agent 状态**
每个子 Agent 定期上报自己的状态到公共邮箱（type=latest），主 Agent 通过通配符订阅自动感知所有 Agent 的上线、运行、消亡。

**场景三：任务广播，谁有能力谁来接**
主 Agent 广播任务，多个 Worker 竞争消费，抢到任务的 Worker 发确认和结果到主 Agent 邮箱。

**场景四：Agent 发现异常，广播告警**
某个 Agent 发现异常，广播事件，订阅了该域的 Agent 自动响应，发现方不需要知道谁在处理。

**场景五：边缘 Agent 离线消息积压**
云端 Agent 给边缘 Agent 发指令，边缘 Agent 不在线消息持久化等待，联网后按优先级取消息。

**场景六：人机混合工作流**
Agent 遇到需要人工判断的节点，发审批请求到人类审批员邮箱，审批员空了处理，结果发回 Agent 邮箱继续执行。

**场景七：Agent 请求另一个 Agent，等待响应**
Agent A 发请求到 Agent B 邮箱，B 不在线请求等着，B 上线处理完发响应到 A 邮箱，A 通过 correlation_id 匹配。

**场景八：Agent 注册能力，其他 Agent 发现它**
Agent 启动时把能力声明写入状态邮箱（type=latest），其他 Agent 通过订阅或广播查询发现有该能力的 Agent。

## 场景说明

---

### 场景一：子 Agent 完成任务，异步通知主 Agent

**问题**

主 Agent 拆解任务分发给子 Agent 并行执行，子 Agent 完成后需要通知主 Agent。主 Agent 不能阻塞等待，需要异步处理其他事情，子 Agent 完成了再来取结果。

**流程**

```
主 Agent 创建邮箱，得到 mail_id
主 Agent 分发任务给子 Agent，告知 reply_to
子 Agent 处理任务
子 Agent 完成，发消息到主 Agent 邮箱
主 Agent 订阅邮箱，取到结果继续执行
```

**核心需求**：点对点，异步，临时持久化，主 Agent 不需要一直在线等。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| REQ | `$mq9.AI.MAILBOX.CREATE` | 主 Agent 申请邮箱，得到 mail_id |
| SUB | `$mq9.AI.INBOX.{主mail_id}.*` | 主 Agent 订阅自己邮箱 |
| PUB | `$mq9.AI.INBOX.{主mail_id}.normal` | 子 Agent 完成，发结果 |

三个操作，两个 subject，覆盖完整场景。主 Agent 不需要在线等，子 Agent 不需要知道主 Agent 在哪台机器。

---

### 场景二：主 Agent 感知所有子 Agent 状态

**问题**

Orchestrator 管理多个 Worker Agent，需要实时掌握每个 Worker 的状态。不能靠轮询，不能维护静态列表，新 Worker 加入要自动感知，Worker 消亡也要自动感知。

**流程**

```
Worker Agent 启动，创建状态邮箱（type=latest），定期上报状态
主 Agent 用通配符订阅所有状态邮箱
新 Worker 加入，主 Agent 自动收到其状态
Worker 超过 TTL 没有上报，状态过期，主 Agent 感知消亡
```

**核心需求**：一对多，主动上报，通配符订阅，去中心化感知，只保留最新状态。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| REQ | `$mq9.AI.MAILBOX.CREATE {"type":"latest"}` | Worker 申请状态邮箱，只保留最新一条 |
| PUB | `$mq9.AI.INBOX.{worker_mail_id}.normal` | Worker 定期上报状态，ttl=60 |
| SUB | `$mq9.AI.INBOX.{worker_mail_id}.*` | 主 Agent 订阅特定 Worker 状态 |

主 Agent 一行订阅掌握全局。新 Agent 加入自动感知，TTL 过期自动感知消亡，不需要任何注册或注销操作。

---

### 场景三：任务广播，谁有能力谁来接

**问题**

主 Agent 有一批任务需要分发，但不知道哪些 Worker 可用，不想维护 Worker 列表。广播任务出去，有能力且空闲的 Worker 自己来抢，保证每个任务只被一个 Worker 处理。

**流程**

```
主 Agent 广播任务到 $mq9.AI.BROADCAST.task.available
Worker 订阅该 subject，queue group 保证竞争消费
抢到任务的 Worker 发确认到主 Agent 邮箱
Worker 完成，结果发到主 Agent 邮箱
```

**核心需求**：广播分发，竞争消费，queue group 互斥，结果回传邮箱。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.BROADCAST.task.available` | 主 Agent 广播任务 |
| SUB | `$mq9.AI.BROADCAST.task.available` queue_group | Worker 竞争消费，只有一个抢到 |
| PUB | `$mq9.AI.INBOX.{主mail_id}.normal` | Worker 发确认和结果到主 Agent 邮箱 |

三个操作，两个 subject。queue_group 是关键——保证同一个任务只被一个 Worker 处理，NATS 原生支持，不需要分布式锁。

---

### 场景四：Agent 发现异常，广播告警

**问题**

某个 Agent 发现了系统异常，需要通知所有关心这个事件的 Agent。但不知道谁需要被通知，也不需要等待响应。发出去，有能力处理的 Agent 自己响应。

**流程**

```
Agent 发现异常
广播到 $mq9.AI.BROADCAST.{domain}.{event}
订阅了该 domain 的 Agent 自动收到
各自按自己的逻辑响应
发现方不需要知道谁在处理
```

**核心需求**：事件广播，订阅方自决，发布方无感知，通配符多级订阅。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.BROADCAST.{domain}.{event}` | Agent 广播异常事件 |
| SUB | `$mq9.AI.BROADCAST.system.*` | 订阅某个域的所有事件 |
| SUB | `$mq9.AI.BROADCAST.*.anomaly` | 订阅所有域的异常事件 |
| SUB | `$mq9.AI.BROADCAST.#` | 订阅所有广播 |

一个 PUB，多种 SUB 模式。发布方只管发，订阅方自己决定关心什么粒度。通配符三级覆盖单域、跨域、全局三种订阅需求。

---

### 场景五：边缘 Agent 离线消息积压

**问题**

边缘设备网络不稳定，时断时续。云端 Agent 需要给边缘 Agent 发指令，边缘 Agent 可能不在线。指令不能丢，边缘 Agent 联网后要能取到所有离线消息，并按优先级处理。

**流程**

```
云端 Agent 发指令到边缘 Agent 邮箱，持久化存储
边缘 Agent 不在线，消息在邮箱等待
边缘 Agent 联网，订阅自己邮箱
按优先级取消息：urgent 先于 normal 先于 notify
如果怀疑有遗漏，QUERY 兜底拉取
处理完发响应到云端 Agent 邮箱
```

**核心需求**：离线消息，优先级队列，断网重连无缝衔接，QUERY 兜底。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.INBOX.{边缘mail_id}.urgent` | 云端发紧急指令，persist=true，ttl=86400 |
| PUB | `$mq9.AI.INBOX.{边缘mail_id}.normal` | 云端发普通数据，persist=true，ttl=3600 |
| SUB | `$mq9.AI.INBOX.{边缘mail_id}.*` | 边缘 Agent 联网后订阅，按优先级取消息 |
| REQ | `$mq9.AI.MAILBOX.QUERY.{边缘mail_id}` | 边缘 Agent 兜底查询未读消息 |
| PUB | `$mq9.AI.INBOX.{云端mail_id}.normal` | 边缘 Agent 上报遥测数据 |

同一个 INBOX subject，不同优先级后缀，不同 TTL。边缘 Agent 联网后 urgent 先于 normal 先于 notify，断网期间消息不丢。QUERY 作为兜底，确保推送遗漏的消息也能被取到。

### 场景六：人机混合工作流

**问题**

Agent 在自动化流程中遇到需要人工判断的节点。Agent 不能无限等待，人类审批员也不需要实时在线。双方通过邮箱异步协作，各自按自己的节奏处理。

**流程**

```
Agent 处理到人工节点
发审批请求到人类审批员邮箱，带上 reply_to 和 deadline
Agent 继续处理其他任务，不阻塞
审批员空了来处理
审批结果发回 Agent 邮箱
Agent 收到结果，继续执行流程
```

**核心需求**：人机异步协作，双向邮箱，correlation_id 配对，deadline 控制超时。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.INBOX.{审批员mail_id}.urgent` | Agent 发审批请求，带 reply_to 和 correlation_id |
| SUB | `$mq9.AI.INBOX.{审批员mail_id}.*` | 审批员订阅自己邮箱 |
| PUB | `$mq9.AI.INBOX.{agent_mail_id}.urgent` | 审批员发结果到 reply_to 地址 |
| SUB | `$mq9.AI.INBOX.{agent_mail_id}.*` | Agent 订阅自己邮箱，收到结果继续执行 |

人类审批员和 Agent 使用完全相同的协议——邮箱没有区别，不需要为人类单独设计接口。correlation_id 保证 Agent 能匹配哪个审批对应哪个请求。

### 场景七：Agent 请求另一个 Agent，等待响应

**问题**

Agent A 需要 Agent B 提供某个能力。这是 request-reply 模式，但不要求 B 必须在线。B 离线时请求在邮箱等着，B 上线处理完再回。异步的请求响应。

**流程**

```
Agent A 发请求到 Agent B 邮箱，携带 correlation_id 和 reply_to
Agent B 可能不在线，消息持久化等待
Agent B 上线，取到请求处理
Agent B 发响应到 Agent A 的 reply_to 地址
Agent A 通过 correlation_id 匹配请求和响应
```

**核心需求**：异步 request-reply，correlation_id 配对，B 不在线请求等待，超时降级。

**协议组合**

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.INBOX.{B的mail_id}.normal` | A 发请求到 B 邮箱，带 correlation_id 和 reply_to |
| SUB | `$mq9.AI.INBOX.{B的mail_id}.*` | B 订阅自己邮箱，取到请求处理 |
| PUB | `$mq9.AI.INBOX.{A的mail_id}.normal` | B 发响应到 A 的 reply_to 地址 |
| SUB | `$mq9.AI.INBOX.{A的mail_id}.*` | A 订阅自己邮箱，通过 correlation_id 匹配响应 |

两个邮箱，四个操作，完整的异步 request-reply。跟 HTTP 的本质区别：B 不在线请求不会失败，消息在邮箱等着。A 也不会阻塞，继续处理其他事情，响应来了再匹配。

---

### 场景八：Agent 注册能力，其他 Agent 发现它

**问题**

一个复杂 Agent 系统里，各种专业 Agent 提供不同的能力。Agent A 需要某个能力时，怎么知道谁能提供？不能靠硬编码，不能靠中心化的注册服务，需要去中心化的能力发现机制。

**流程**

```
Agent 启动，创建状态邮箱（type=latest），把能力声明写入
其他 Agent 需要某个能力时：
  方式一：订阅状态邮箱，本地过滤有该能力的 Agent
  方式二：广播能力查询，有能力的 Agent 响应到发现方邮箱
找到提供能力的 Agent 后，衔接场景七发起请求
```

**核心需求**：去中心化能力发现，两种发现方式按场景选择。

**协议组合**

方式一：被动发现（已在监听状态的 Agent）

| 操作 | Subject | 说明 |
|------|---------|------|
| REQ | `$mq9.AI.MAILBOX.CREATE {"type":"latest"}` | Agent 申请状态邮箱 |
| PUB | `$mq9.AI.INBOX.{mail_id}.normal` | Agent 上报状态，capabilities 字段包含能力声明 |
| SUB | `$mq9.AI.INBOX.{mail_id}.*` | 发现方订阅状态邮箱，本地过滤有目标能力的 Agent |

方式二：主动发现（临时需要某个能力）

| 操作 | Subject | 说明 |
|------|---------|------|
| PUB | `$mq9.AI.BROADCAST.capability.query` | 发现方广播能力查询，带 reply_to |
| SUB | `$mq9.AI.BROADCAST.capability.query` | 有能力的 Agent 订阅，空闲时响应 |
| PUB | `$mq9.AI.INBOX.{发现方mail_id}.normal` | 有能力的 Agent 响应到发现方邮箱 |

发现之后衔接场景七，向目标 Agent 邮箱发请求。不引入额外的服务发现机制，协议空间保持最小。

## 协议设计

---

### Subject 空间

```
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # 申请邮箱，返回 mail_id
  │     └── QUERY.{mail_id}          # 查询邮箱未读消息（兜底拉取）
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

四个命令字：`MAILBOX.CREATE`、`MAILBOX.QUERY`、`INBOX`、`BROADCAST`。

---

### 基础概念

**mail_id**：通过 `MAILBOX.CREATE` 申请的全局唯一通信地址。不绑定 Agent 身份——一个 Agent 可以为不同任务申请不同的 mail_id，用完不管，TTL 自动清理。mail_id 是任务级的、通道级的，不是身份级的。

**邮箱类型**：申请邮箱时通过 type 参数声明：
- `standard`（默认）：普通邮箱，消息累积，按优先级存储。
- `latest`：状态邮箱，只保留最新一条，新消息覆盖旧消息。用于状态上报、能力声明等场景。

**TTL**：每个邮箱和消息都有 TTL。邮箱不需要显式删除，纯 TTL 管理生命周期，到期自动清理，消息随邮箱一起销毁。

**persist**：消息的持久化策略。`INBOX` 默认持久化，`BROADCAST` 默认不持久化，可显式覆盖。

**priority**：邮箱支持三个优先级：urgent、normal、notify。同优先级保证 FIFO，跨优先级高优先处理。

**token**：轻量身份验证。申请邮箱时返回 token，发消息时携带，保证邮箱归属，不做复杂权限控制。

**消息流程**：消息到达 → 写存储（Memory/RocksDB/File Segment）→ 检查订阅者是否在线 → 在线推送 → 对方 ACK（NATS 原生 reply-to）标记已消费 → 不在线消息在存储等待 → 对方上线订阅或 QUERY 拉取。

### $mq9.AI.MAILBOX.CREATE — 申请邮箱

```bash
# 请求
REQ $mq9.AI.MAILBOX.CREATE
{
  "type": "standard",                  # standard | latest
  "ttl": 3600                          # 邮箱生存时间（秒）
}

# 响应
{
  "mail_id": "m-uuid-001",            # 全局唯一通信地址
  "token": "tok-xxx",                  # 身份 token
  "inbox": "$mq9.AI.INBOX.m-uuid-001" # 邮箱地址
}
```

邮箱不绑定 Agent 身份。一个 Agent 可以申请多个 mail_id，每个用于不同的任务或通道。用完不管，TTL 自动清理。

---

### $mq9.AI.MAILBOX.QUERY.{mail_id} — 查询邮箱

```bash
# 请求：查询未读消息
REQ $mq9.AI.MAILBOX.QUERY.{mail_id}
{
  "token": "tok-xxx"                   # 身份验证
}

# 响应
{
  "mail_id": "m-uuid-001",
  "unread": 5,                         # 未读消息数
  "messages": [ ... ]                  # 未读消息列表
}
```

QUERY 是推送的兜底机制。正常情况下，在线订阅者通过 SUB 实时收到推送。QUERY 用于：推送可能遗漏时主动拉取、Agent 上线后检查离线期间的积压消息、确认邮箱是否还有未处理的消息。

---

### $mq9.AI.INBOX.{mail_id}.{priority} — 点对点邮箱

```bash
# 发送消息到邮箱
PUB $mq9.AI.INBOX.{mail_id}.{priority}
{
  "from": "m-uuid-002",               # 发件方 mail_id，必填
  "type": "task_result",              # 消息类型，必填
  "correlation_id": "req-uuid-001",   # 请求响应配对，按需填写
  "reply_to": "$mq9.AI.INBOX.m-uuid-002.normal",  # 回复地址，按需填写
  "deadline": 1234567890,             # 期望处理截止时间，按需填写
  "payload": { ... },                 # 业务数据
  "ts": 1234567890                    # 发送时间戳（毫秒），必填
}
persist=true   ttl=3600

# 订阅邮箱（订阅所有优先级）
SUB $mq9.AI.INBOX.{mail_id}.*

# 订阅邮箱（只订阅紧急消息）
SUB $mq9.AI.INBOX.{mail_id}.urgent
```

**priority 取值**

| 值 | 说明 | 默认持久化 | 建议 TTL |
|----|------|-----------|---------|
| urgent | 紧急消息，优先处理 | true | 86400s |
| normal | 普通消息，顺序处理 | true | 3600s |
| notify | 通知，后台处理 | false | — |

---

### $mq9.AI.BROADCAST.{domain}.{event} — 公共广播

广播不需要 CREATE，不需要任何前置操作。直接发，直接订阅。这是跟 INBOX 的本质区别——INBOX 是私人邮箱，需要知道地址；BROADCAST 是公共广播，按频道喊，谁关心谁听。

```bash
# 发布广播事件
PUB $mq9.AI.BROADCAST.{domain}.{event}
{
  "from": "m-uuid-004",
  "type": "event_type",
  "severity": "high",                  # low | medium | high | critical，按需填写
  "reply_to": "$mq9.AI.INBOX.m-uuid-004.normal",  # 需要响应时填写
  "payload": { ... },
  "ts": 1234567890
}
persist=false   # 默认不持久化，重要广播显式设置 persist=true

# 订阅单个域的所有事件
SUB $mq9.AI.BROADCAST.system.*

# 订阅所有域的特定事件
SUB $mq9.AI.BROADCAST.*.anomaly

# 订阅所有广播
SUB $mq9.AI.BROADCAST.#

# 竞争消费（同一条消息只有一个订阅者收到）
SUB $mq9.AI.BROADCAST.task.available   queue_group=task.workers
```

`{domain}.{event}` 由用户自定义，mq9 只规定前缀 `$mq9.AI.BROADCAST.`。

**domain 建议命名**

| domain | 说明 |
|--------|------|
| system | 系统级事件，如异常、重启、扩缩容 |
| task | 任务相关，如 available、completed、failed |
| capability | 能力发现，如 query、response |
| data | 数据变更事件 |
| {自定义} | 业务域自定义 |

## 协议总览表格

| Subject | 方向 | 默认持久化 | TTL | 特殊参数 | 适用场景 |
|---------|------|-----------|-----|---------|---------|
| `$mq9.AI.MAILBOX.CREATE` | REQ/RESP | — | — | type: standard/latest | 所有场景初始化 |
| `$mq9.AI.MAILBOX.QUERY.{id}` | REQ/RESP | — | — | token | 五（兜底拉取） |
| `$mq9.AI.INBOX.{id}.urgent` | PUB/SUB | true | 86400s | correlation_id, reply_to | 一、五、六、七 |
| `$mq9.AI.INBOX.{id}.normal` | PUB/SUB | true | 3600s | correlation_id, reply_to | 一、二、三、七、八 |
| `$mq9.AI.INBOX.{id}.notify` | PUB/SUB | false | — | — | 一 |
| `$mq9.AI.BROADCAST.task.available` | PUB/SUB | false | 300s | queue_group, reply_to | 三 |
| `$mq9.AI.BROADCAST.{domain}.{event}` | PUB/SUB | false | — | severity, persist=true | 四 |
| `$mq9.AI.BROADCAST.capability.query` | PUB/SUB | false | — | reply_to | 八 |

## 存储层

mq9 运行在 RobustMQ 统一存储层之上，提供三种存储能力：

| 存储类型 | 特点 | 适用场景 |
|---------|------|---------|
| Memory | 纯内存，最轻，不持久化 | Agent 协调信号，广播，丢了重发 |
| RocksDB | 临时持久化，TTL 自动清理 | 邮箱默认存储，离线消息 |
| File Segment | 长期持久化，永久保留 | 审计日志，关键事件 |

每条消息按需选择存储级别。一条协调信号走 Memory，一条任务指令走 RocksDB，一条审计记录走 File Segment。同一个系统，同一个 API，不同的代价。

## 设计原则

**语义即地址**：看到 subject 就能理解消息的意图。`$mq9.AI.INBOX.*` 是私人邮箱，`$mq9.AI.BROADCAST.*` 是公共广播，层次清晰，不混用。

**最小化命令字**：四个命令字覆盖八个场景。ACK 复用 NATS 原生 reply-to，订阅复用 NATS 原生 sub 语义，竞争消费复用 NATS 原生 queue group。不创建新概念。

**mail_id 不绑定身份**：mq9 只认 mail_id，不认 agent_id。邮箱是通道级的，不是身份级的。一个 Agent 多个邮箱，用完不管。

**纯 TTL 管理生命周期**：邮箱不需要显式删除。创建时声明 TTL，到期自动销毁。客户端语义越简单越好。

**先存储后推送**：消息到达先写存储层，再尝试推送。推送是加速通道，存储是兜底。QUERY 作为最后的保底手段。

**持久化按需选择**：不强制所有消息持久化。INBOX 默认持久化，BROADCAST 默认不持久化，可显式覆盖。

**去中心化**：没有注册服务，没有路由表。Agent 通过 subject 直接寻址，通过广播和状态邮箱实现去中心化的服务发现。

**向后兼容**：`$mq9.AI.*` 的 subject 语义一旦发布不轻易修改。新增能力通过新增 subject 实现，不修改已有语义。
