# 我们的答案：mq9

上一篇文章《消息队列的下一个十年：忘了 Kafka 吧》里，我提了几个判断：append only log 不是通信的全部，S3 和数据湖叙事是这个框架里的尽头，底层必须原子化，评价体系要从"量"转向"多"。

那篇文章只提问题，不提方案。这篇聊我们的解法。

---

## 出发点

RobustMQ 做了三年，最初解决的是多协议割裂的问题——统一存储层，协议只是读写接口，一条消息写入一次，MQTT、Kafka、NATS、AMQP 各自按自己的语义读取。

做着做着，发现这个架构有一个副产品：**新增一个协议，代价极轻。** 存储层不动，协议只是加一层语义解析。

mq9 就是从这个地基上长出来的第五个协议——专门为 AI Agent 通信设计。

---

## 为什么是邮箱

上一篇我说，通信的形态远比 append only log 丰富——邮箱、广播、最新值、请求响应、临时通道。mq9 选了其中最基本的一个：**邮箱。**

为什么？因为 Agent 不是服务。服务是长期在线的，你发请求它就响应。Agent 是临时的，可能只活几秒，随时上下线。今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。

每个构建多 Agent 系统的团队都在用临时方案绕这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

mq9 直接解决它：我发出去，你空了来取，不需要同时在线。不是消息队列，不是流，是邮件系统。

这个邮箱比喻不是修辞，是设计本身：

- 开通邮箱，拿到地址 → `MAILBOX.CREATE`
- 寄信，对方在不在家不管，信在邮箱等着 → `INBOX.{mail_id}.{priority}`
- 收信，急件先看，普通件后看 → 在线时服务端推送
- 怕有遗漏，去邮局查一下 → `MAILBOX.QUERY`
- 回信，信里有对方地址 → NATS 原生 reply-to
- 长期没人用，邮局自动注销 → TTL 自动销毁
- 小区公告栏，贴上去所有人能看 → `BROADCAST`
- 公告栏贴了一个活，只能一个人接 → queue group

## 四个命令字

mq9 的全部客户端协议：

```
$mq9.AI.
  ├── MAILBOX.
  │     ├── CREATE                    # 申请邮箱，返回 mail_id
  │     └── QUERY.{mail_id}          # 查未读消息（兜底拉取）
  │
  ├── INBOX.
  │     └── {mail_id}.
  │           ├── urgent              # 紧急消息
  │           ├── normal              # 普通消息
  │           └── notify              # 通知
  │
  └── BROADCAST.
        └── {domain}.
              └── {event}            # 事件广播
```

四个命令字，覆盖八个场景。ACK 用 NATS 原生 reply-to，订阅用 NATS 原生 sub，竞争消费用 NATS 原生 queue group。不创建新概念。

几个关键的设计决策：

**mail_id 不绑定 Agent 身份。** mq9 只认 mail_id，不认 agent_id。一个 mail_id 就是一个通信通道，谁创建的、给谁用的、用来干什么的，mq9 不关心。一个 Agent 可以为不同任务申请不同的 mail_id，用完不管，TTL 自动清理。

**两种邮箱类型。** CREATE 时声明：`standard` 是普通邮箱，消息累积；`latest` 是状态邮箱，只保留最新一条，用于状态上报和能力声明。

**邮箱不需要显式删除。** 纯 TTL 管理生命周期。客户端语义越简单越好。

**INBOX 是私人邮箱，BROADCAST 是公共广播。** 知道找谁用 INBOX——需要 CREATE 拿到 mail_id。不知道找谁用 BROADCAST——不需要任何前置操作，直接发直接订阅。

**先存储后推送。** 消息到达先写存储层，再推给在线订阅者。推送是加速通道，存储是兜底。QUERY 作为最后的保底手段。

## 八个真实场景

四个命令字组合起来，能解决什么？

**异步任务回传。** 主 Agent CREATE 邮箱，子 Agent 完成后 PUB 到主 Agent 的 INBOX。不需要阻塞等待，空了来取。

**全局状态感知。** Worker 申请 type=latest 的状态邮箱，定期 PUB 状态。主 Agent 订阅感知全局，新 Worker 加入自动感知，TTL 过期自动感知消亡。

**任务广播与竞争消费。** 主 Agent BROADCAST 任务，Worker 用 queue group 竞争消费，抢到的 Worker 发结果到主 Agent INBOX。

**异常告警广播。** BROADCAST 异常事件，订阅了该域的 Agent 自动响应。发布方不需要知道谁在处理。

**边缘离线消息积压。** PUB 到边缘 Agent 的 INBOX（urgent/normal 分优先级），边缘不在线消息持久化等待，联网后按优先级取。QUERY 兜底拉取确保不遗漏。

**异步请求-响应。** A PUB 请求到 B 的 INBOX 带 reply_to 和 correlation_id，B 上线处理后 PUB 响应到 A 的 INBOX。离线不丢。

**能力注册与发现。** Agent 把能力声明 PUB 到自己的 latest 邮箱，其他 Agent 订阅感知，或通过 BROADCAST.capability.query 主动查询。去中心化，零额外服务。

**人机混合工作流。** Agent PUB 审批请求到人类审批员 INBOX，审批员处理后 PUB 结果回 Agent INBOX。人类和 Agent 用完全相同的协议，流程不中断。

## 不是 append only，是 pub/sub

上一篇我说 Kafka 的一切都来自 append only log 这个设计决策。mq9 选了另一条路：pub/sub 模型，基于 NATS 协议。

这个选择决定了 mq9 跟 Kafka 系是完全不同的东西。在邮箱语义下，append only log 的每一个假设都是多余的：消息不需要顺序回放，需要按优先级取；不需要 offset 管理，需要 TTL 自动清理；不需要 partition，需要动态 subject；不需要三副本持久化每一条消息，需要按场景选择存储级别。

mq9 在 NATS 生态中的定位是介于 Core pub/sub 和 JetStream 之间的中间态。Core NATS 太轻——没有持久化，离线就丢。JetStream 太重——stream、consumer、offset、replay，一整套对标 Kafka 的语义。mq9 在 pub/sub 的基础上加了可选持久化、优先级、TTL 自动管理，但不引入 stream、consumer group、offset 这些重概念。刚好够用，不多不少。

基于 NATS 还带来一个重要的好处：**零生态成本。** 所有语言的 NATS SDK 开箱即用，不需要构建任何新的 SDK。`$mq9.AI.*` 的命名空间设计让协议本身成为文档——`$mq9.AI.INBOX.{id}.URGENT`，一行字符串，五层信息，看到 subject 就懂语义。

## 底层原子，上层组合

上一篇我花了很大篇幅讲一个判断：底层必须是原子的，上层按场景组合封装。mq9 就是这个判断的实践。

RobustMQ 的存储层提供三种原子能力：

**Memory** — 纯内存，最轻，不持久化。Agent 之间的协调信号，发完就没了，丢了重发。

**RocksDB** — 临时持久化。消息落盘，等对方上线取走，TTL 到期自动清理。邮箱的默认选择。

**File Segment** — 长期持久化。审计日志、关键事件，永久保留。

一条协调信号走内存，一条任务指令走 RocksDB，一条审计记录走 File Segment。同一个系统，同一个 API，不同的代价。每条消息按需选择，不是所有消息都值得三副本，也不是所有消息都可以丢。

上一篇说的"选择权比绝对保障更重要"，这就是落地的样子。

---

## 总结

mq9 看起来像消息队列，像 pub/sub，像 JetStream，像谁都有点像。但它是它——拆开看每个能力都不新鲜，组合在一起为 Agent 场景专门设计，用邮箱语义统一表达，今天没有人做过。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

**只要你需要跑很多 Agent，你就需要 mq9。** 一键启动，运维简单，四个命令字，各种模式。

上一篇我说，底层越原子，上层越自由。这篇是实践：RobustMQ 三年打磨的统一存储是原子，mq9 是在原子能力上组合出来的、为 Agent 场景定制的通信层。

问题提完了，这是我们的答案。还在路上，慢慢干。
