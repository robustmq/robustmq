# 我们的答案：mq9

上一篇文章《消息队列的下一个十年：忘了 Kafka 吧》里，我提了几个判断：append only log 不是通信的全部，S3 和数据湖叙事是这个框架里的尽头，底层必须原子化，评价体系要从"量"转向"多"。

那篇文章只提问题，不提方案。这篇聊我们的解法。

---

## 出发点

RobustMQ 做了三年，最初解决的是多协议割裂的问题——统一存储层，协议只是读写接口，一条消息写入一次，MQTT、Kafka、NATS、AMQP 各自按自己的语义读取。

做着做着，发现这个架构有一个副产品：**新增一个协议，代价极轻。** 存储层不动，协议只是加一层语义解析。

mq9 就是从这个地基上长出来的第五个协议——专门为 AI Agent 通信设计。

---

## 为什么是邮箱

上一篇我说，通信的形态远比 append only log 丰富——邮箱、广播、最新值、请求响应、临时通道。mq9 选了其中最基本的一个：**邮箱。**

为什么？因为 Agent 不是服务。服务是长期在线的，你发请求它就响应。Agent 是临时的，可能只活几秒，随时上下线。今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。

每个构建多 Agent 系统的团队都在用临时方案绕这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

mq9 直接解决它：我发出去，你空了来取，不需要同时在线。不是消息队列，不是流，是邮件系统。

---

## 三个原语

邮箱语义展开，就是三个核心原语。

**邮箱（Inbox）**——每个 Agent 有自己的邮箱地址，发件方直接发到对方邮箱，消息持久化等待。支持三个优先级：`URGENT`、`NORMAL`、`NOTIFY`，紧急消息先处理。邮箱随 Agent 生命周期自动管理，TTL 到期自动销毁。

**广播（Broadcast）**——发布方不关心消费方，发出去就完了。订阅方自己决定关心什么，用通配符表达兴趣。支持 queue group 竞争消费，同一条消息只被一个 Worker 处理。

**状态上报（Status）**——每个 Agent 定期上报状态，只保留最新一条。主 Agent 用通配符一行订阅感知全局。新 Agent 加入不需要注册，消亡不需要注销，超过 TTL 状态自动过期。

三个原语，覆盖 Agent 通信的所有基本模式。

---

## 八个真实场景

三个原语组合起来，能解决什么？

**异步任务回传。** 子 Agent 完成任务，结果发到主 Agent 邮箱，主 Agent 不需要阻塞等待，空了来取。

**全局状态感知。** 一行订阅 `$mq9.AI.STATUS.*`，新 Worker 加入自动感知，消亡自动感知，不需要维护任何 Agent 列表。

**任务广播与竞争消费。** 主 Agent 广播任务，有能力的 Worker 自己来抢，queue group 保证一个任务只被一个 Worker 处理，不需要分布式锁。

**异常告警广播。** 某个 Agent 发现异常，广播出去，有处理能力的 Agent 自动响应，发布方不需要知道谁在处理。

**边缘离线消息积压。** 云端给边缘 Agent 发指令，边缘不在线消息持久化等待，联网后按优先级处理。

**异步请求-响应。** Agent 向另一个 Agent 提问，对方不在线请求在邮箱等着，上线处理完通过 reply_to 发回，correlation_id 保证配对。

**能力注册与发现。** Agent 把能力声明写入状态，其他 Agent 通过订阅或广播查询发现有该能力的 Agent。

**人机混合工作流。** Agent 发审批请求到人类审批员邮箱，审批完结果发回 Agent 邮箱。人类和 Agent 使用完全相同的协议，流程不中断。

---

## 不是 append only，是 pub/sub

上一篇我说 Kafka 的一切都来自 append only log 这个设计决策。mq9 选了另一条路：pub/sub 模型，基于 NATS 协议。

这个选择决定了 mq9 跟 Kafka 系是完全不同的东西。在邮箱语义下，append only log 的每一个假设都是多余的：消息不需要顺序回放，需要按优先级取；不需要 offset 管理，需要 TTL 自动清理；不需要 partition，需要动态 subject；不需要三副本持久化每一条消息，需要按场景选择存储级别。

基于 NATS 还带来一个重要的好处：**零生态成本。** NATS 的客户端生态直接就是 mq9 的客户端生态，Go、Python、Rust、Java、JavaScript 所有语言的 SDK 开箱即用。不需要构建任何新的 SDK。

`$mq9.AI.*` 的命名空间设计，让协议本身成为文档。`$mq9.AI.INBOX.{id}.URGENT` ——mq9 的协议，AI 场景，收件箱，指定 Agent，紧急优先级。一行字符串，五层信息，看到 subject 就懂语义。

---

## 底层原子，上层组合

上一篇我花了很大篇幅讲一个判断：底层必须是原子的，上层按场景组合封装。mq9 就是这个判断的实践。

RobustMQ 的存储层提供三种原子能力：

**Memory**——纯内存，最轻，不持久化。Agent 之间的协调信号，发完就没了，丢了重发。

**RocksDB**——临时持久化。消息落盘，等对方上线取走，TTL 到期自动清理。邮箱的默认选择。

**File Segment**——长期持久化。审计日志、关键事件，永久保留。

一条协调信号走内存，一条任务指令走 RocksDB，一条审计记录走 File Segment。同一个系统，同一个 API，不同的代价。每条消息按需选择，不是所有消息都值得三副本，也不是所有消息都可以丢。

上一篇说的"选择权比绝对保障更重要"，这就是落地的样子。

---

## 总结

mq9 看起来像消息队列，像 pub/sub，像 JetStream，像谁都有点像。但它是它——拆开看每个能力都不新鲜，组合在一起为 Agent 场景专门设计，用邮箱语义统一表达，今天没有人做过。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

**只要你需要跑很多 Agent，你就需要 mq9。** 一键启动，运维简单，协议简单，各种模式。

上一篇我说，底层越原子，上层越自由。这篇是实践：RobustMQ 三年打磨的统一存储是原子，mq9 是在原子能力上组合出来的、为 Agent 场景定制的通信层。

问题提完了，这是我们的答案。还在路上，慢慢干。
