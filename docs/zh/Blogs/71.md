# 下一代基础软件的核心是多，不是量

最近读了黄东旭两篇文章——《如何做 AI Agent 喜欢的基础软件》和《正式介绍一下最近的工作：db9.ai》。读完之后，有一个判断越来越清晰：**下一代基础软件的核心竞争力，不是量，是多。**

这篇文章是我作为 RobustMQ 作者的一些思考，不一定对，但是真实的。

---

## 这一代基础软件在解决什么问题

Kafka 诞生于 LinkedIn，解决的是大规模日志收集的问题——每秒百万级消息，持久化，顺序消费，高吞吐。这是"量"的问题。

Redpanda 用 C++ 重写了 Kafka，去掉了 JVM 的 GC 抖动，性能提升数倍。这仍然是"量"的问题，只是在同一个维度上做得更好。

这一代基础软件的评价体系是：吞吐、延迟、可靠性。这套评价体系背后有一个假设——消息是宝贵的，每一条都不能丢，系统要为最极端的流量场景设计。

这个假设在大数据、数据管道、实时流处理的场景里是对的。但它不是普世真理。

---

## 黄东旭看到了什么

黄东旭在 TiDB Cloud 上观察到一个数据：每天新创建的集群里，超过 90% 是由 AI Agent 直接创建的。

这不是预测，是今天正在发生的事情。

Agent 使用基础软件的方式跟人类开发者完全不同。Agent 快速创建资源，试一把，不行就丢掉，换个方向重来。这是日抛型的工作负载——实例的生命周期可能只有几秒钟，创建和销毁的频率极高，数量爆炸性增长。

这带来了一个根本性的问题：**传统基础软件假设"一个集群很宝贵"，但 Agent 需要的是"实例是便宜的，随开随用"。**

他做 db9 就是在回答这个问题——每个 Agent 有自己独立的 Postgres + 文件系统，底层是虚拟化共享资源，成本是传统方案的 1/100。不是把 Postgres 做得更快，而是让 Postgres 能支撑百亿级的租户数量。

从"量"到"多"，这是范式的切换，不是参数的优化。

---

## RobustMQ 的起点

我做 RobustMQ 的初衷，是想解决多协议割裂的问题。

今天的消息基础设施是协议孤岛：IoT 设备用 MQTT，数据管道用 Kafka，企业系统用 RabbitMQ，每套系统独立运维，数据需要桥接和转发，复杂度随规模指数级增长。

RobustMQ 从架构底层选了一条不同的路：**统一存储层，协议只是读写接口**。一条消息写入一次，MQTT、Kafka、NATS、AMQP 各自按自己的语义读取。写一次，任意协议消费，数据不复制，存储不冗余。

这个设计出发点是多协议互通，但在做的过程中，我慢慢意识到它的真正价值不只是互通。

---

## 真正的价值：协议只是语义，存储才是本质

传统的多协议方案——包括 FlowMQ——走的是路由 + 桥接的路子。消息进来，路由引擎分发到不同的存储后端，每个协议维护自己的数据副本。这是写放大，消费端越多，代价越高。

RobustMQ 的统一存储层把这个问题从根上解决了。但更重要的是，它带来了一个副产品：**新增一个协议，在架构层面的代价是很轻的。**

1 个协议、5 个协议、100 个协议，存储层不需要动，协议只是加一层解析。这个性质，在工业边缘场景里尤其重要——工厂里的协议生态极度碎片化，OPC-UA、Modbus、CoAP、MQTT-SN，每个厂商一套。传统方案只能堆桥接，运维复杂度随协议数量线性叠加。RobustMQ 的架构在这里有结构性优势。

---

## Kafka 和 Redpanda 的天花板

我很尊重 Kafka 和 Redpanda，它们解决的问题是真实的，工程质量也很高。但它们的天花板是清晰的。

Kafka 的核心抽象是 topic 和 partition。这套抽象为高吞吐的数据管道设计——topic 是长期存在的资源，partition 是并行处理的单元，消费是 pull 模型。这套假设在大数据场景里是对的。

但这套假设决定了 Kafka 无法回答 Agent 时代的问题：
- 没有 Agent 身份的概念
- 没有临时邮箱，topic 创建需要管理员操作
- 没有优先级队列
- 没有 TTL 自动清理
- 为"量"设计，无法支撑海量轻量临时通道

Redpanda 用 C++ 重写了 Kafka，性能更好，运维更简单。但它继承了 Kafka 的全部架构假设。在"量"的维度，Redpanda 做得比 Kafka 好。但在"多"的维度，它跟 Kafka 是同一条路。

这不是说 Kafka 和 Redpanda 没有价值，而是说它们解决的是上一个时代的问题。新的时代需要新的答案。

---

## 语义变了

这是我最近想得最清楚的一件事。

传统消息队列的核心评价指标：高吞吐、低延迟、高可靠、多副本。这套指标背后是一个假设——每条消息都很宝贵，丢了就丢了，无法重来。

Agent 时代这个假设失效了。Agent 可以重试，可以重新生成，可以换个方向再来。一条临时任务消息丢了——重发一次就好。消息的"价值密度"降低了，为每一条消息付出三副本的代价变得不划算。

真正重要的不是高可靠，而是**选择权**：
- 这条消息值得持久化吗？——不值得，走内存，最轻
- 这条消息需要可靠传递吗？——需要，加一层持久化
- 这条消息需要长期保留吗？——需要，走 File Segment

不同的消息，不同的语义，不同的代价。Agent 按需选择，系统按配置执行。

RobustMQ 的存储分级——Memory、RocksDB、File Segment——本来是为性能和成本权衡设计的。但在这个框架下，它的意义更深：这是在给每一条消息提供可靠性的选择权。

---

## mq9：从多协议到 Agent 通信

顺着这个思路，RobustMQ 做了 mq9——专门为 AI Agent 设计的通信协议层，与 MQTT、Kafka、NATS、AMQP 并列，共享同一套统一存储。

mq9 解决的核心问题是：**Agent 之间的异步通信，发件方和收件方不需要同时在线。**

Agent 不是服务，它是临时的，频繁上下线。今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。每个构建多 Agent 系统的团队都在用自己的临时方案绕过这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

mq9 给每个 Agent 一个邮箱：

```bash
# 申请邮箱
nats req '$mq9.AI.MAILBOX.CREATE' '{}'

# 发消息，对方不在线也没关系
nats pub '$mq9.AI.INBOX.{agent_id}.NORMAL' '{"payload":"..."}'

# 广播，谁关心谁订阅
nats pub '$mq9.AI.BROADCAST.task.available' '{"task_id":"t-001"}'
```

`$mq9.AI.*` 的命名空间设计，把品牌、场景、语义三件事用一个字符串说清楚了。基于 NATS 协议，不需要构建任何新的 SDK，NATS 的生态直接就是 mq9 的生态。

**只要你需要跑很多 Agent，你就需要 mq9。**

低延迟、高可靠、高性能、大吞吐——这些在这个场景里不是必要的。必要的是：轻量、易用、支持海量租户、生命周期自动管理。这就是"多"的逻辑。

---

## 跟 db9 的关系

黄东旭做 db9，解决 Agent 的存储问题。我做 mq9，解决 Agent 的通信问题。

```
db9    # Agent 存储层：数据怎么存，怎么查
mq9    # Agent 通信层：消息怎么传，怎么协作
```

两个互补，不竞争。Agent 系统同时需要两者——通过 mq9 传递消息协调行动，通过 db9 存储状态和上下文。

这也说明了一件事：Agent 时代的基础设施栈正在从零开始被建立。存储层、通信层、计算层——每一层都需要重新设计，因为 Agent 的工作方式跟人类开发者根本不同。

db9 和 mq9 只是开始。

---

## 它是它

mq9 看起来像消息队列，看起来像 pub/sub，看起来像 JetStream，看起来像谁都有点像。但它是它，不是任何人的变体。

拆开看，mq9 用到的每一个能力都不新鲜：

- pub/sub——NATS 有
- 持久化——Kafka 有
- 通配符订阅——NATS 有
- 轻量——NATS 有
- TTL——Redis 有
- 无成本创建删除——NATS subject 有
- 优先级队列——RabbitMQ 有

但这个组合，为 Agent 场景专门设计，今天没有人做过。

**mq9 是 pub/sub 模型，不是 append-only 模型。** 这一个选择，就决定了它跟 Kafka 系是完全不同的东西。Kafka 的一切——高吞吐、顺序性、可回放——都来自 append-only log 这个设计决策。这个模型对数据管道是完美的，但天然不适合邮箱语义：消息不需要顺序回放，需要按优先级取；不需要 offset 管理，需要 TTL 自动清理；不需要 partition，需要动态 subject。

mq9 把 pub/sub、持久化、模糊订阅、轻量、TTL、优先级、无成本创建删除糅合在一起，用邮箱的语义把它们统一表达出来。每一个能力都在为同一个目标服务：让 Agent 之间的异步通信，像发邮件一样自然。

不需要跟任何人比较，不需要解释"比 Kafka 好在哪里"，不需要解释"跟 JetStream 的区别是什么"。它就是它——为 Agent 设计的通信层，用的人自然懂。

## 三个可能被问到的问题

做基础软件，方向对了还不够，还要能回答质疑。以下三个问题是我预判会被问到的，提前给出我们的答案。

**问题一：mq9 的第一个真实用户在哪里？**

会有的。慢慢做，慢慢等。

基础软件不是靠第一天就有用户证明价值的，是靠方向正确、持续积累。MQTT 等了二十年，Kafka 等了十年。今天构建多 Agent 系统的团队正在用 Redis、轮询数据库、自研队列绕过这个问题——这些人就是 mq9 的潜在用户，他们的痛点今天就存在。

等 mq9 有了可运行的 demo，等第一个团队用它解决了真实问题，那一天会是真正的转折点。在那之前，把代码做扎实，把文档写清楚，让有这个问题的人能找到答案。

**问题二："多"的量级你想到多少？**

这个问题本身问错了维度。百亿、十亿，只是数字，数字不重要。重要的是架构有没有这个能力。

RobustMQ 的统一存储 + 轻量 subject + TTL 自动管理，这套架构天然支持水平扩展。上限不是架构约束，是资源约束。资源可以加，架构不能改。三年打磨的统一存储层，给了我们在这个维度上肆意挥洒的底气——想支撑多少租户，加资源就好，不需要重新设计。

这就是为什么地基要打扎实。地基对了，上面能建多高是工程问题，不是架构问题。

**问题三：mq9 跟 NATS JetStream 的本质差异是什么？**

这是最值得说清楚的一个问题。

NATS JetStream 的心智模型本质上是 Kafka + 部分 AMQP 的语义组合——持久化的消息流，有 stream，有 consumer，有 offset，有 replay。它还是在"存储和消费消息"这个框架里，解决的是持久化和可靠传递的问题。

mq9 完全不是这个东西。

mq9 的心智模型是**邮箱**。不是消息队列，不是流，不是 pub/sub，是邮件系统。我发出去，你空了来取，不需要同时在线。这个语义跟 Kafka、AMQP、JetStream 完全不在一个概念体系里。

它是一个为 Agent 设计的通信应用，不是消息队列。虽然底层实现用了消息队列的核心能力——持久化、路由、TTL——但暴露给用户的语义是邮箱，是通信管道，是 Agent 的协作基础设施。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

这个区分很重要。用消息队列的框架去理解 mq9，就会问错问题——"吞吐怎么样""offset 怎么管理""partition 怎么分配"。这些问题在 mq9 的语义里都不存在。

## 慢就是快

RobustMQ 做了三年。三年里推翻过架构，重构过核心模块，否定过自己以为想清楚的设计。

这不是在为慢找借口，而是基础软件的必经之路。一个设计上的妥协，可能在三年后变成无法绕开的技术债。架构想清楚了，后续每增加一个协议的成本是线性的；架构没想清楚，每增加一个功能都是一次重构。

今天 RobustMQ 的架构，能够支撑 MQTT、Kafka、NATS、AMQP、mq9 共享同一套存储层，每增加一个协议不需要动存储。这个性质是三年打磨的结果，不是一开始就有的。

基础软件没有捷径。Kafka 用了十年才成为数据管道的事实标准，MQTT 用了二十年才在 IoT 领域站稳。

下一代基础软件，核心是多，不是量。这个判断我越来越确定。RobustMQ 和 mq9 是我们在这个方向上的探索，还在路上，慢慢干。
