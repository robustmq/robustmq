# 我们的答案：mq9

上一篇文章《消息队列的下一个十年：忘了 Kafka 吧》里，我提了一个判断：消息队列这个领域被 Kafka 的心智模型固化了十年，所有人都在 append only log 的框架里优化——更快、更便宜、更弹性——但没人退后一步问：通信是不是只有这一种形态？

那篇文章只提问题，不提方案。这篇聊方案。

---

## 从哪里开始

先说结论：我们做了 mq9——一个为 AI Agent 设计的通信层，长在 RobustMQ 上。

这不是一个突然冒出来的想法。RobustMQ 做了三年，最初要解决的是多协议割裂的问题：IoT 用 MQTT，数据管道用 Kafka，企业系统用 RabbitMQ，每套独立运维，数据靠桥接搬运。我们从架构底层选了一条不同的路——统一存储层，协议只是读写接口。一条消息写入一次，MQTT、Kafka、NATS、AMQP 各自按自己的语义读取。

这个设计的初衷是多协议互通。但做着做着，我发现它真正的价值不只是互通，而是一个副产品：**新增一个协议，在架构层面的代价极轻。** 存储层不需要动，协议只是加一层语义解析。1 个协议和 100 个协议，底层是同一套东西。

mq9 就是从这个地基上长出来的第五个协议。

---

## 为什么是邮箱，不是队列

上一篇文章里我说，通信的形态远比 append only log 丰富——邮箱、广播、最新值、请求响应、临时通道。这些形态用 Kafka 的模型去套，每一个都别扭。

mq9 选了其中最基本的一个：**邮箱。**

不是消息队列，不是流，不是 pub/sub。是邮件系统。我发出去，你空了来取，不需要同时在线。

为什么 Agent 需要邮箱？因为 Agent 不是服务。服务是长期在线的，你发请求它就响应。Agent 是临时的，可能只活几秒，随时上下线。今天 Agent A 给 Agent B 发消息，B 不在线，消息直接丢了。

每个构建多 Agent 系统的团队都在用自己的临时方案绕过这个问题——Redis pub/sub、轮询数据库、自研任务队列。能用，但都是绕路。

mq9 直接解决它：

```bash
# 申请邮箱
nats req '$mq9.AI.MAILBOX.CREATE' '{}'

# 发消息，对方不在线也没关系
nats pub '$mq9.AI.INBOX.{agent_id}.NORMAL' '{"payload":"..."}'

# 广播，谁关心谁订阅
nats pub '$mq9.AI.BROADCAST.task.available' '{"task_id":"t-001"}'
```

三行命令，Agent 有了邮箱，有了广播，有了异步通信能力。

## 不是 append only，是 pub/sub

这是 mq9 和 Kafka 系产品的根本分野。

Kafka 的一切——高吞吐、顺序性、可回放——都来自 append only log 这个设计决策。这个模型对数据管道是完美的。但在邮箱语义下，它的每一个假设都是多余的：

消息不需要顺序回放，需要按优先级取。不需要 offset 管理，需要 TTL 自动清理。不需要 partition，需要动态 subject。不需要三副本持久化每一条消息，需要按场景选择存储级别。

mq9 走的是 pub/sub 模型，基于 NATS 协议。这意味着两件事：

第一，NATS 的客户端生态直接就是 mq9 的客户端生态。Go、Python、Rust、Java、JavaScript——所有语言的 NATS SDK 开箱即用。不需要构建任何新的 SDK，零生态成本。

第二，`$mq9.AI.*` 的命名空间设计，让协议本身成为文档。看到 subject 就懂语义，不需要查文档。`$mq9.AI.INBOX.{id}.URGENT` ——mq9 的协议，AI 场景，收件箱，指定 Agent，紧急优先级。一行字符串，五层信息。

## 底层原子，上层组合

上一篇文章里我说，下一代消息基础设施的底层必须是原子的、细粒度的，上层按场景组合封装。mq9 就是这个判断的实践。

RobustMQ 的存储层提供三种原子能力：

**Memory**——纯内存，最轻，不持久化。适合 Agent 之间的协调信号，发完就没了，丢了重发。

**RocksDB**——临时持久化。消息落盘，等对方上线取走，TTL 到期自动清理。邮箱的默认选择。

**File Segment**——长期持久化。审计日志、关键事件，永久保留。

mq9 不是把这三种存储焊死在某个配置里，而是让每条消息按需选择。一条协调信号走内存，一条任务指令走 RocksDB，一条审计记录走 File Segment——同一个系统，同一个 API，不同的代价。

这就是上一篇文章里说的"选择权比绝对保障更重要"。不是所有消息都值得三副本，也不是所有消息都可以丢。让用户自己决定。

## 评价体系不一样

用 Kafka 的评价体系看 mq9，会问错所有问题。

"吞吐怎么样？" ——不是重点。Agent 邮箱的消息量级跟数据管道不是一个概念。

"offset 怎么管理？" ——没有 offset。邮箱语义不需要顺序消费。

"partition 怎么分配？" ——没有 partition。每个 Agent 一个 subject，动态创建，自动清理。

"延迟多少毫秒？" ——够用就行。Agent 不是高频交易系统。

mq9 关心的指标是另一组：

**能同时支撑多少个 Agent 邮箱？** ——统一存储 + 轻量 subject + TTL 自动管理，上限是资源约束，不是架构约束。

**创建和销毁一个邮箱的成本是多少？** ——趋近于零。NATS subject 不需要预创建，不需要管理员操作。

**一个 Agent 消亡后，它的邮箱多久被清理？** ——TTL 自动管理，不需要人工干预。

**能不能一键启动，运维简单？** ——没有 ZooKeeper，没有外部依赖，单个二进制。

这就是"多"的评价体系。衡量的不是你能搬多少数据，而是你能同时服务多少个独立的通信实体。

## 跟 db9 的关系

黄东旭做 db9，解决 Agent 的存储问题——每个 Agent 有自己独立的 Postgres + 文件系统，底层虚拟化共享资源，成本是传统方案的 1/100。

mq9 解决 Agent 的通信问题——每个 Agent 有自己的邮箱，异步通信，离线可达，生命周期自动管理。

```
db9    → Agent 存储层：数据怎么存，怎么查
mq9    → Agent 通信层：消息怎么传，怎么协作
```

两者互补，不竞争。Agent 系统同时需要两者——通过 mq9 传递消息协调行动，通过 db9 存储状态和上下文。

有意思的是，我们从完全不同的方向出发——他从数据库，我从消息队列——但推导出了同一个结论：下一代基础软件的核心是多，不是量。两个独立验证指向同一个判断，这比自说自话有力量得多。

---

## 它是它

mq9 看起来像消息队列，看起来像 pub/sub，看起来像 JetStream，看起来像谁都有点像。但它是它。

拆开看，mq9 用到的每一个能力都不新鲜——pub/sub 有、持久化有、TTL 有、优先级有、轻量创建有。但这个组合，为 Agent 场景专门设计，用邮箱语义统一表达，今天没有人做过。

就像电子邮件不是数据库，虽然邮件系统用数据库存邮件。mq9 不是消息队列，虽然 mq9 用消息队列的能力实现邮箱。

不需要跟任何人比较。用的人自然懂。

---

## 三个可能被问到的问题

做基础软件，方向对了还不够，还要能回答质疑。以下三个问题是我预判会被问到的，提前给出答案。

**问题一：mq9 的第一个真实用户在哪里？**

会有的。慢慢做，慢慢等。

基础软件不是靠第一天就有用户证明价值的，是靠方向正确、持续积累。MQTT 等了二十年，Kafka 等了十年。今天构建多 Agent 系统的团队正在用 Redis、轮询数据库、自研队列绕过通信问题——这些人就是 mq9 的潜在用户，他们的痛点今天就存在。

等 mq9 有了可运行的 demo，等第一个团队用它解决了真实问题，那一天会是真正的转折点。在那之前，把代码做扎实，把文档写清楚，让有这个问题的人能找到答案。

**问题二："多"的量级你想到多少？**

这个问题问错了维度。百亿、十亿，只是数字，数字不重要。重要的是架构有没有这个能力。

RobustMQ 的统一存储 + 轻量 subject + TTL 自动管理，天然支持水平扩展。上限不是架构约束，是资源约束。资源可以加，架构不能改。三年打磨的统一存储层，给了我们在这个维度上肆意挥洒的底气——想支撑多少租户，加资源就好，不需要重新设计。

地基对了，上面能建多高是工程问题，不是架构问题。

**问题三：mq9 跟 NATS JetStream 的本质差异是什么？**

这是最值得说清楚的一个问题。

NATS JetStream 的心智模型本质上还是 Kafka + 部分 AMQP 的语义组合——持久化的消息流，有 stream，有 consumer，有 offset，有 replay。它还是在"存储和消费消息"这个框架里。

mq9 完全不是这个东西。mq9 的心智模型是邮箱。不是消息队列，不是流，是邮件系统。这个语义跟 Kafka、AMQP、JetStream 不在一个概念体系里。

它是一个为 Agent 设计的通信应用，不是消息队列。虽然底层实现用了消息队列的核心能力——持久化、路由、TTL——但暴露给用户的语义是邮箱，是通信管道，是 Agent 的协作基础设施。

用消息队列的框架去理解 mq9，会问错问题——"吞吐怎么样""offset 怎么管理""partition 怎么分配"。这些问题在 mq9 的语义里都不存在。

---

## 一句话

**只要你需要跑很多 Agent，你就需要 mq9。**

一键启动，运维简单，协议简单，各种模式。

上一篇我说，底层越原子，上层越自由。这篇是我们的实践：RobustMQ 三年打磨的统一存储是原子，mq9 是在这个原子能力上组合出来的、为 Agent 场景定制的通信层。

问题提完了，这是我们的答案。还在路上，慢慢干。
