# 从 EMQX 最新版本思考 RobustMQ

EMQX 最新版本开始原生支持消息流和队列语义，同时通过与 GreptimeDB 的战略合作，在 EMQX Cloud 中内置了时序数据库能力。作为一个在消息中间件领域深耕多年的产品，EMQX 的演进方向值得认真借鉴。不是为了比较谁好谁坏，而是借此把 RobustMQ 自己的方向想得更清楚一些。

## 消息系统的边界在哪里

一个 MQTT broker 为什么要支持流语义？这个问题背后有一个更本质的问题：消息系统的边界应该在哪里？

IoT 数据天然是流。设备持续上报传感器数据、状态变化、事件触发，本质上就是一条无尽的时间序列。传统的消息中间件解决的是"数据怎么送达"的问题，但随着业务复杂度提升，用户越来越关心"数据送达之后怎么用"。于是消息系统开始往流处理、存储、分析方向延伸，这是整个行业自然演进的结果。

EMQX 选择在 MQTT 语义上叠加流和队列的能力，走的是在已有协议基础上不断扩展的路径。这种方式对存量用户非常友好，迁移成本低，生态延续性好。RobustMQ 想走的是另一条路——不是在协议层叠加能力，而是从存储层重新想这个问题。

## 统一存储：一份数据，多种形态

RobustMQ 目前支持三种底层存储引擎：内存存储、RocksDB 和 File Segment。这三种存储各自对应不同的场景——内存存储追求极致低延迟，适合像 NATS 那样的轻量消息场景；RocksDB 提供可靠的持久化，适合 RabbitMQ 风格的队列语义；File Segment 面向高吞吐的顺序写入，是 Kafka 那类日志流场景的基础。

三种存储引擎共存，但对上层协议来说，它们是透明的。用户根据自己的场景选择合适的存储策略，而不是因为"要用 Kafka 语义就必须部署 Kafka，要用队列语义就必须部署 RabbitMQ"。

更重要的是，这个存储层未来是可以平滑扩展的。S3 对象存储、Lakehouse 这类面向大规模冷数据的存储形态，可以作为新的存储引擎接入进来，整个协议层不需要改动。随着数据量增长，用户可以把历史数据自动下沉到更便宜的存储介质，而消费端的接口完全不变。

这是 RobustMQ 存储层设计最核心的一个判断：**存储引擎是可替换的，但上层协议对存储的依赖应该是统一的。** 换句话说，不是为每种协议单独维护一套存储，而是所有协议共享同一套存储抽象。

## 多协议：不是功能堆叠，是同一份数据的不同视图

理解了存储层的设计之后，多协议支持的价值就变得很直接：因为底层存储是统一的，所以 MQTT 写入的消息，可以直接被 Kafka 协议消费，反之亦然，不需要任何数据搬运。

这件事听起来简单，但实际上很多系统做不到，因为它们的多协议支持本质上是"为每种协议维护独立的存储，再在协议之间建数据管道"。数据管道意味着延迟、意味着一致性窗口、意味着额外的运维成本。而 RobustMQ 里，MQTT 进、Kafka 出，或者 Kafka 进、MQTT 出，访问的是同一份数据，没有中间环节。

这个模型往后可以继续扩展。当 AMQP、NATS 这些协议接入进来，它们同样访问的是同一份底层存储。从 A 协议写入，B、C、D 协议都可以读出——这不是"我们支持多少种协议"的功能列表，而是一套基于统一存储视图的架构能力。协议越多，这个能力的价值越大，因为不同生态之间的数据流动不再需要任何胶水层。

## 这对用户意味着什么

一个典型的 IoT 数据平台，通常需要维护多套系统协同工作：MQTT broker 负责设备接入，Kafka 负责流处理，时序数据库负责持久化查询，可能还有一套数据湖做长周期分析。每两套系统之间都需要数据管道，每条管道都是运维负担，也是潜在的故障点。

RobustMQ 想做的是把这条链路变短。设备通过 MQTT 写入，流处理系统直接用 Kafka 协议消费，不需要单独部署 Kafka 集群；历史数据的查询可以通过内置的轻量查询接口满足大部分场景，复杂分析则通过 Kafka 协议把数据导出到专业工具——整个链路的接缝更少，需要维护的系统更少。

这里有一个选择的取舍需要说清楚：RobustMQ 不会去做复杂的流处理引擎，也不会去做专业的时序数据库，这些事情有更专业的工具来做，而且做得很好。RobustMQ 的定位是把消息层的事情做好、做透，通过标准协议让专业工具可以无缝接入，而不是把所有事情都收进来自己做。

## 存储引擎的扩展性是长期赌注

回头看这个设计，存储层的可扩展性其实是一个长期的赌注。

短期内，内存 + RocksDB + File Segment 已经可以覆盖大部分场景。但随着 IoT 数据规模持续增长，冷热分层的需求会越来越明确——热数据留在本地高速存储，冷数据自动下沉到 S3 或者 Lakehouse，对用户完全透明。这不是一个新鲜想法，Tiered Storage 在 Kafka 社区讨论了很多年，真正做好的还不多。

RobustMQ 现在打下存储引擎可插拔的基础，是为了将来这个方向做好准备。等到 S3 存储引擎接入进来，用户的数据可以在不同存储介质之间自动流动，而上层协议不感知任何变化。这种演进的成本会比较低，因为架构从一开始就为此做了预留。

## 多协议是架构，不是特性

最后回到一个更根本的问题：为什么要做多协议？

很多人理解的多协议是"我们支持 MQTT，也支持 Kafka，也支持 AMQP"，是一张功能清单。但 RobustMQ 想表达的多协议是另一个意思：**同一份数据，被不同协议的用户各自以自己熟悉的方式使用，互不干扰，也不需要彼此感知。**

MQTT 的用户不需要知道下游有没有 Kafka consumer，Kafka 的用户不需要知道上游是不是 IoT 设备通过 MQTT 写进来的。每个人用自己的工具、自己的生态，数据在底层天然流通。

这是一个关于消息系统应该是什么形态的判断，和存储引擎的可扩展性一样，都是在赌一个方向：**未来最有价值的消息基础设施，不是功能最全的，而是接缝最少的。**

## 本质还是通信管道

说了这么多，有一点需要说清楚：RobustMQ 的定位始终是消息系统，不是数据平台，不是流计算引擎，不是时序数据库。它做的事情，本质上还是把消息从一个地方可靠地送到另一个地方——一条通信管道，只是这条管道可以同时被不同协议的系统使用。

所有的设计，无论是存储层的抽象、多协议的支持，还是未来的 S3 分层存储，都是在为这条管道服务，而不是在管道之外生长出新的系统。这个边界很重要。一旦开始往数据处理、分析、可视化方向延伸，系统就会变得越来越重，最终失去焦点。

RobustMQ 希望的成长方式是自底向上的。先把存储层的抽象做稳，再在上面支持不同的协议，再逐步扩展存储引擎的形态。每一步都建立在上一步的基础上，而不是一开始就规划一个大而全的系统。这种方式慢，但每个决策都有实际的场景支撑，也更容易保持克制。

做产品很容易陷入一个误区：看到竞品有什么功能，就觉得自己也要有；用户提了什么需求，就往里加。功能越堆越多，系统越来越复杂，反而丢掉了最初想清楚的那件事。做减法比做加法难，聚焦比扩张难，但好产品往往就是在这种克制里长出来的。

RobustMQ 想做的事情其实不复杂：**一个统一的消息系统，能适应不同的场景，真正好用。** 不需要用户在不同中间件之间做权衡，不需要维护多套系统之间的数据同步。消息该去哪里，系统自然处理好，用户不用操心。

仅此而已。
