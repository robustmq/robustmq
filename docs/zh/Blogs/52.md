# 从 EMQX 最新版本思考 RobustMQ

EMQX 最新版本开始支持流和消息队列的语义，同时通过与 GreptimeDB 的战略合作，在 EMQX Cloud 中内置了时序数据库能力（EMQX Tables）。

这个演进方向值得认真思考一下背后的逻辑，不是为了对比，而是借此把 RobustMQ 自己的方向想得更清楚。

## IoT 数据的完整链路

一个 MQTT broker 为什么要支持流语义？

因为 IoT 数据天然是流。设备持续上报传感器数据、状态变化、事件触发，这些数据本质上就是一条无尽的时间序列。MQTT 解决的是连接问题——如何让海量设备可靠地把数据送到平台上来。但数据送来之后，怎么处理、怎么消费，是一个完全不同的问题。

IoT 场景的用户面对的完整链路是：

1. **设备连接**：海量设备通过 MQTT 接入，数据可靠送达
2. **实时处理**：数据流需要被实时消费，接入流处理引擎做过滤、聚合、告警
3. **时序存储**：处理后的数据需要持久化，支持按时间范围查询
4. **复杂分析**：长周期数据的分析、挖掘、可视化

这四件事，用户通常要维护四套系统：MQTT broker、Kafka 集群、时序数据库、分析引擎。每个系统之间都需要数据搬运，每一条管道都是运维负担和潜在的故障点。

RobustMQ 的方向是把这条链路变短。不是通过集成更多系统，而是通过统一底层存储，让不同的协议共享同一份数据。

## 流处理：用 Kafka 协议而不是重造轮子

流处理系统（Flink、Spark Streaming 等）需要从消息队列拉数据，Kafka 是事实上的标准接口。这意味着 IoT 平台如果想让设备数据进入流处理链路，就必须解决 MQTT 数据和 Kafka 之间的对接问题。

常见的解法是建一条管道：MQTT → 规则引擎 → Kafka。数据被搬运一次，复制一份，用户需要维护两套系统，还要处理数据一致性和延迟问题。

RobustMQ 的选择是直接支持 Kafka 协议。MQTT 写入的数据和 Kafka 消费的数据是同一份，不同的协议是同一份存储的不同视图。设备不需要知道下游是什么，流处理系统不需要知道上游是什么，两边各用各自熟悉的协议接入。

这个选择的价值不只是少复制一份数据：

- **没有额外延迟**：数据写入即可消费，没有中间管道的传输延迟
- **没有一致性窗口**：不存在 MQTT 端写入但 Kafka 端还没同步的中间状态
- **没有产品绑定**：任何支持 Kafka 协议的客户端和工具都可以直接接入，Kafka Connect、Kafka Streams、以及所有基于 Kafka 构建的数据基础设施，不需要任何适配层
- **Kafka 生态的杠杆**：Kafka 生态经过多年发展，连接器、处理框架、监控工具极为丰富，RobustMQ 通过协议兼容直接继承这个生态

这是一个有意识的架构选择：**用标准协议来承载流的能力，而不是自己造一个流处理引擎。**

## 时序存储：内置能力的边界

时序存储是 IoT 平台绕不开的能力。设备数据需要持久化，需要按时间范围查询，需要支持聚合统计。

但这里有一个值得想清楚的问题：IoT 平台内置的时序查询，需要多强？

IoT 场景 80% 的查询需求是相当简单的：

- 查询某个设备最近 N 条上报记录
- 按时间窗口统计平均值、最大值、最小值
- 按设备 ID 或标签过滤，取一段时间内的数据
- 监控某个设备的状态变化趋势

这些查询不需要复杂的执行引擎。一个针对时序模式优化的轻量索引层完全可以覆盖，而且可以做到更低的延迟、更简单的运维、更小的资源占用。

真正复杂的分析场景是另一回事：多维聚合、跨设备关联分析、长周期数据挖掘、机器学习特征提取。这些场景对查询引擎的要求很高，GreptimeDB、InfluxDB、TDengine 这类专业工具在这方面有深厚的积累，RobustMQ 没有必要在这里竞争。

而 Kafka 协议本身就是天然的数据出口：下游接 GreptimeDB、InfluxDB、进数据湖，都是标准操作，用户可以根据自己的需求选择合适的工具。

这是一个清晰的产品边界：**内置的做到够用，外接的做到无缝。** 把简单查询的低延迟体验留给内置层，把复杂分析的专业能力交给专业工具，两者之间通过标准协议连通。

## 统一存储、多协议视图

把这几件事放在一起，RobustMQ 的产品形态可以描述为：

| 层次 | 协议 | 场景 |
|------|------|------|
| 设备写入 | MQTT | IoT 设备连接，海量并发写入 |
| 流消费 | Kafka | 实时处理，流处理引擎接入 |
| 轻量查询 | SQL / HTTP | 简单时序查询，设备数据检索 |
| 复杂分析 | Kafka 数据出口 | 外接专业时序数据库或数据湖 |

这四层共享同一份底层存储。每一层的用户都用自己熟悉的方式接入，互不感知，数据只存一次。

这种形态的核心价值在于：**整个链路没有多余的接缝。** 数据从设备写入到被流处理系统消费，不需要经过任何搬运；从流处理到时序查询，不需要额外的存储层；从内置查询到外部分析，通过标准协议自然衔接。

对用户来说，这意味着：搭建一套 IoT 数据平台，不再需要在技术选型上做太多权衡，也不需要维护多套系统之间的数据同步。

## 内置轻量分析的方向

在简单时序查询的基础上，RobustMQ 未来可以逐步支持一些内置的轻量流式计算能力：

- **滑动窗口聚合**：对某个设备或设备组，按滑动时间窗口计算统计值
- **简单规则告警**：当设备数据满足某个条件时触发通知，不需要外接流处理引擎
- **设备状态追踪**：维护每个设备的最新状态，支持状态变化的实时查询

这些能力不需要引入重型的流处理引擎，在平台内部就能完成，覆盖更多开箱即用的场景。它的定位和时序查询一样——够用，不是最强，但能让大多数用户在不引入额外系统的情况下解决问题。

## 核心竞争力：消息 MQ 层面的统一视图

把这些放在一起，RobustMQ 的核心竞争力可以用一句话来描述：**多协议 = 消息 MQ 层面的统一视图。**

传统 MQ 的思路是"我支持这个协议"——为每个协议实现一套接入层，协议之间如果要互通，就需要搭桥、转发、同步。这本质上是连接器的堆叠，协议越多，系统越复杂，数据一致性越难保证。

RobustMQ 的思路不同：**同一份数据，对所有协议都是可见的。** 不管是 MQTT 写入、Kafka 消费、还是未来支持的 AMQP 接入，访问的是同一份存储，看到的是同一份数据。协议是视图，不是边界。

这个特性越往后越值钱。接入的协议越多，每个协议背后覆盖的生态越大，这个统一视图的价值就越高。MQTT 背后是 IoT 设备生态，Kafka 背后是流处理和数据工程生态，AMQP 背后是企业消息生态。这些生态原本是割裂的，需要用户在不同系统之间搭建数据管道来打通。而在 RobustMQ 里，它们天然连通，不需要任何胶水层。

这不是靠功能叠加形成的竞争力，而是架构设计层面的护城河。功能可以被复制，结构层面的优势很难被追上。

## 这条路的价值

IoT 数据基础设施的演进，最终都会走向一个方向：减少系统数量，减少数据搬运，减少运维负担，同时保持对不同生态的开放性。

RobustMQ 试图通过统一存储和多协议视图来达到这个目标。这不是一个"功能更强"的方案，而是一个"结构更简单"的方案。功能可以一点点加，结构一旦错了就很难改。

从这个角度看，现在每一行代码、每一个架构决策，都是在为这个方向打基础。
